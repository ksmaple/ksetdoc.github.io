常见算法与场景

核心结论：不同密码学算法有各自的适用场景和权衡点，选型时需综合考虑安全性、性能、合规性和生态系统成熟度。

密码学算法选型需匹配场景的安全需求和性能约束，避免过度设计或不足

对称加密算法选型

场景推荐算法原因
通用数据加密AES-256-GCM安全、支持认证加密、硬件加速
国内合规系统SM4国密要求
低功耗IoT设备AES-128计算量小，安全性足够
文件系统加密AES-XTS针对块设备优化

非对称加密算法选型

场景推荐算法原因
HTTPS/TLS证书RSA-2048 / ECDSA P-256广泛兼容
移动端API签名ECDSA签名短、验证快
国内金融系统SM2国密合规
区块链/数字货币ECDSA / Ed25519签名短小高效

哈希算法选型

场景推荐算法原因
密码存储bcrypt / scrypt / Argon2计算慢可防暴力破解
数据完整性校验SHA-256快速且安全
数字签名SHA-256 + RSA/ECDSA标准组合
国内合规SM3国密要求

典型组合场景

场景组合方式说明
HTTPS通信RSA/ECDHE + AES-GCM + SHA-256密钥交换 + 对称加密 + 完整性校验
JWT签名HS256(RSA) / ES256(ECDSA)Header.Payload 用私钥签名
文件加密AES-256 + RSA随机AES密钥加密文件，RSA加密AES密钥
数字证书私钥签名 + 公钥验签 + 证书链CA用私钥签发，浏览器用CA公钥验证

国密算法体系

算法类型用途替代的国际算法
SM2非对称加密加密、签名、密钥交换RSA、ECDSA
SM3哈希数据摘要、签名SHA-256
SM4对称加密数据加密AES
SM9标识密码基于标识的加密和签名IBE方案

常见问题与误区

问题1：密钥长度过短

根因：使用1024位RSA或128位以下AES以追求性能 解决：RSA至少2048位，AES至少128位，密码学安全不能妥协

带新人

核心结论：带新人的目标是在可控时间内让其独立交付价值，通过明确路径、及时反馈和适当放手，既保障业务产出又完成人才梯队建设。

新人培养通过分阶段目标、具体任务和持续反馈，实现从"需要指导"到"独立交付"的转化

三阶段培养路径

阶段目标导师动作新人动作
融入期熟悉环境和流程1v1每天同步、分配保姆级任务主动提问、记录踩坑
成长期理解业务和代码每周Review进展、讲解设计思路写学习笔记、承担完整需求
独立期自主交付价值放手+关键节点把关独立设计方案、反馈流程问题

任务设计原则

原则说明示例
由简到难先成功再挑战先改文案，再改逻辑，再独立做需求
有明确产出能看到效果修复一个可见的Bug，而非纯阅读代码
允许犯错安全环境试错非核心功能的Bug修复，即使回退也无风险
与业务相关理解业务价值做真实需求而非Demo项目

反馈机制

频率形式内容
每天即时反馈代码Review意见、当天问题解答
每周1v1沟通进展、困难、下阶段计划
每月阶段评估能力成长、待改进点、目标调整

常见问题与应对

问题表现应对
不敢问卡了很久不说主动询问进度，营造"提问安全"氛围
过度依赖每一步都要确认逐步减少提示，让其先给出方案再讨论
进展慢估时总是不准任务拆更小，明确每个子任务的完成标准
融入难团队协作不畅安排Buddy（非导师）日常陪伴

常见问题与误区

问题1：让新人直接看代码"熟悉"，没有具体任务

根因：认为看代码就能理解系统 解决：给具体问题和目标，在解决问题中理解代码，而非被动阅读

第三方系统交互设计

核心结论：与第三方系统交互的核心挑战是网络不可靠和接口差异，通过统一接口层封装异构协议，通过幂等和重试保障可靠性，通过熔断降级保障自身稳定性。

第三方交互通过统一封装、幂等重试、熔断降级三大机制保障可靠性

设计原则

原则说明实现
统一封装屏蔽第三方差异适配器模式，统一出入参
幂等设计重复调用无副作用业务订单号唯一性
超时重试网络抖动容错指数退避，限制重试次数
熔断降级第三方故障自我保护失败率阈值触发熔断
异步解耦降低耦合和延迟MQ中转

分库分表

核心结论：分库分表通过水平拆分（数据分散）和垂直拆分（职责分离）突破单库性能瓶颈，但引入了分布式事务、跨分片查询、数据迁移等复杂度，应在单库优化无法满足需求时作为最后手段引入。

分库分表通过拆分策略将数据分布到多个节点，以牺牲部分查询灵活性换取存储和性能扩展性

拆分策略

策略方式适用场景
垂直分库按业务拆分（用户库/订单库/商品库）业务边界清晰，耦合低
垂直分表大字段拆分（主表+扩展表）单表字段过多，冷热数据分离
水平分库同表数据分散到不同库单库数据量过大
水平分表同表数据分散到不同表单表数据量过大，同一实例内

分片键选择

方案分片键优点缺点
哈希分片user_id % 4数据均匀分布范围查询需扫描全部分片
范围分片create_time支持范围查询热点问题（新数据集中在最新分片）
混合分片时间+哈希兼顾均匀与范围实现复杂

分布式ID

方案优点缺点
数据库自增简单性能瓶颈，不适合分片
Snowflake高性能，趋势递增时钟回拨问题
号段模式批量获取，性能高依赖数据库
UUID无中心无序，占用空间大

ShardingSphere集成

分片后的挑战

问题说明解决方案
跨分片JOIN无法直接JOIN不同分片的数据全局表广播、应用层聚合、宽表冗余
跨分片排序分页ORDER BY + LIMIT需全局排序应用层归并、限制深度分页
分布式事务跨库事务无法保证ACIDSeata AT模式、最终一致性
数据迁移扩容需重新分片双写过渡、增量同步

分页优化

常见问题与误区

问题1：过早引入分库分表

根因：单表几百万数据就开始拆分，增加不必要复杂度 解决：先尝试索引优化、读写分离、归档历史数据，单表千万级以上再考虑拆分
