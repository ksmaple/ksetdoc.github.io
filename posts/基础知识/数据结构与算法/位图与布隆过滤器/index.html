位图与布隆过滤器

核心结论：位图用1bit表示一个状态，将存储空间压缩至1/8，适合大规模去重和状态标记；布隆过滤器在位图基础上增加多哈希函数，以可控的假阳性为代价实现O(1)的成员查询，是缓存穿透防护和大集合判存的经典方案。

位图将状态压缩到1bit，布隆过滤器用多哈希+位图实现概率型判存，二者核心都是空间换时间

位图（Bitmap）：用bit数组表示状态集合

存储10亿个int（约4GB） vs 位图标记10亿个状态（约119MB）

10亿个整数（去重判断）：
  HashSet: 10亿 * 4字节 ≈ 4GB（不考虑HashMap开销）
  位图: 10亿 bit = 10^9 / 8 ≈ 119MB
  
  压缩比: 4GB / 119MB ≈ 34倍
// Java中的位图实现：BitSet
BitSet bitSet = new BitSet(1000000000);

// 标记数字n存在
bitSet.set(n);

// 判断数字n是否存在
boolean exists = bitSet.get(n);

// 位运算本质
// 第n位: arr[n/64] 的第 (n%64) 位
// 设置: arr[index] |= (1L << offset)
// 查询: (arr[index] >> offset) & 1

位图的应用场景：

场景用法示例
大规模去重标记已出现的数字10亿个int找重复
状态标记用户签到、在线状态365天签到，1个用户365bit
位运算统计两个集合的交并差A & B（交集）、A | B（并集）
内存页管理标记页是否使用OS内存管理
// 用户签到系统：1年365天，1亿用户
// 传统方案: 1亿 * 365 * 1字节 ≈ 36.5GB
// 位图方案: 1亿 * 365 bit ≈ 4.5GB

// 判断用户userId在第day天是否签到
BitSet[] userSigns = new BitSet[100000000];  // 或按天组织
boolean signed = userSigns[userId].get(day);

// 统计某用户本月签到天数
userSigns[userId].get(dayStart, dayEnd).cardinality();

布隆过滤器（Bloom Filter）：概率型数据结构

核心思想：用k个哈希函数将元素映射到位图的k个位置

插入"apple"：
  hash1("apple") % m → 位置3，置1
  hash2("apple") % m → 位置7，置1
  hash3("apple") % m → 位置12，置1

查询"apple"：
  检查位置3、7、12是否都为1 → 是 → "可能存在"（可能有假阳性）
  检查位置3、7、12 → 任一不为1 → "一定不存在"（无假阴性）

查询"banana"（未插入）：
  hash1("banana") → 位置7 ✓
  hash2("banana") → 位置12 ✓
  hash3("banana") → 位置3 ✓
  → 误判为存在（假阳性）！因为3、7、12被其他元素置1了

布隆过滤器的参数设计：

n: 预期插入元素数量
p: 可接受的假阳性率
m: 位数组大小
k: 哈希函数数量

最优公式：
  m = -n * ln(p) / (ln(2)^2)  ≈ n * 1.44 * log2(1/p)
  k = m/n * ln(2)  ≈ log2(1/p)

示例：n=100万，p=1%
  m = 100万 * 1.44 * 6.64 ≈ 960万 bit ≈ 1.2MB
  k = 7 个哈希函数

示例：n=100万，p=0.1%
  m = 100万 * 1.44 * 10 ≈ 14.4M bit ≈ 1.8MB
  k = 10 个哈希函数

布隆过滤器的变体：

变体特点适用场景
计数布隆过滤器位图改为计数器，支持删除元素可能删除
布谷鸟过滤器每个元素2个位置，可替换更好的假阳性率、支持删除
可扩展布隆过滤器多个布隆过滤器串联元素数量不确定

实现原理与关键机制

布隆过滤器的假阳性率推导

假设：
- 位数组大小m，哈希函数k个，插入n个元素
- 每个哈希函数将位独立均匀映射

单个位仍为0的概率：
  P(某位未被某个哈希函数命中) = 1 - 1/m
  P(某位在k次哈希后仍为0) = (1 - 1/m)^k
  P(插入n个元素后某位仍为0) = (1 - 1/m)^(k*n)

当m很大时：(1 - 1/m)^(k*n) ≈ e^(-k*n/m)

假阳性率（k个位都为1的概率）：
  p ≈ (1 - e^(-k*n/m))^k

最优k = (m/n) * ln(2) 时：
  p ≈ (0.6185)^(m/n)

Guava的BloomFilter实现

// Guava布隆过滤器使用示例
BloomFilter<String> filter = BloomFilter.create(
    Funnels.stringFunnel(StandardCharsets.UTF_8),
    1000000,  // 预期插入100万元素
    0.01      // 假阳性率1%
);

// 插入
filter.put("user:12345");
filter.put("user:67890");

// 查询
if (filter.mightContain("user:12345")) {
    // 可能存在 → 查数据库确认
    // 不存在 → 直接返回，避免数据库查询
}

// 存储到Redis
byte[] filterData = BloomFilter.writeTo(filter);
jedis.set("bf:users".getBytes(), filterData);

Redis的Bitmap操作

# 设置第100位为1
SETBIT user:sign:2026:01 100 1

# 获取第100位
GETBIT user:sign:2026:01 100

# 统计1的位数（签到天数）
BITCOUNT user:sign:2026:01

# 连续2天签到统计（位与操作）
BITOP AND sign:both user:sign:2026:01:01 user:sign:2026:01:02
BITCOUNT sign:both

# 查找第一个1或0的位置
BITPOS user:sign:2026:01 1

实际应用中的决策与权衡

场景方案空间特点
精确去重（小数据）HashSet较大精确，无误差
精确去重（大数据）位图1/32数字范围有限时
缓存穿透防护布隆过滤器极小假阳性可接受
URL去重爬虫布隆过滤器极小亿级URL判重
数据库是否存在预判布隆过滤器极小减少无效DB查询
垃圾邮件过滤布隆过滤器极小训练集特征判存

常见问题与误区

问题1：布隆过滤器假阳性太高怎么办？

根因：位数组太小或哈希函数太少 解决：增大位数组（4倍空间可将假阳性率降到1/10）、增加哈希函数、使用布谷鸟过滤器

问题2：布隆过滤器不能删除元素？

根因：删除可能影响其他元素的位 解决：使用计数布隆过滤器（Counter Bloom Filter），每个位改为计数器

问题3：位图和布隆过滤器的哈希函数选择

根因：哈希冲突影响分布均匀性 解决：使用多个独立的哈希函数（如MurmurHash的不同seed），或布隆过滤器的标准做法：用两个哈希函数模拟k个（Giulio Boldrini方法）
