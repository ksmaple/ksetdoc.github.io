倒排索引

核心结论：倒排索引将文档-词项的正向关系反转存储为词项-文档列表，使关键词查询从O(n)扫描降为O(1)查找，是搜索引擎和全文检索系统的核心数据结构。

倒排索引将"文档包含哪些词"反转为"词出现在哪些文档"，实现关键词快速定位

核心结构：词典 + 倒排列表

文档集合：
Doc1: "Java is a programming language"
Doc2: "Python is also a programming language"
Doc3: "Java and Python are popular"

正向索引（文档 → 词）：
Doc1 → [Java, is, a, programming, language]
Doc2 → [Python, is, also, a, programming, language]
Doc3 → [Java, and, Python, are, popular]

倒排索引（词 → 文档列表）：
Java         → [Doc1, Doc3]
Python       → [Doc2, Doc3]
programming  → [Doc1, Doc2]
language     → [Doc1, Doc2]
is           → [Doc1, Doc2]
popular      → [Doc3]
...

倒排索引的存储结构：

┌─────────────────────────────────────────────────┐
│                 词典（Term Dictionary）           │
│  （排序数组/B+树/前缀树/FST）                      │
│                                                 │
│  Java ──→ 倒排列表指针                            │
│  Python → 倒排列表指针                            │
│  ...                                            │
└─────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────┐
│              倒排列表（Posting List）              │
│                                                 │
│  Java: [(Doc1, pos:1), (Doc3, pos:1)]           │
│  Python: [(Doc2, pos:1), (Doc3, pos:3)]         │
│                                                 │
│  压缩存储：Frame Of Reference / Roaring Bitmap    │
└─────────────────────────────────────────────────┘

索引构建流程

1. 文档预处理
   - 分词（Tokenization）："Java编程" → ["Java", "编程"]
   - 标准化（Normalization）：大写转小写、去除停用词
   - 词干提取（Stemming）："running" → "run"

2. 生成倒排记录
   遍历每个文档的每个词项，记录 (文档ID, 词频, 位置)

3. 排序与合并
   按词项排序，相同词项的记录合并为倒排列表

4. 索引优化
   - 词典压缩：前缀共享、FST有限状态转换
   - 倒排列表压缩：差值编码、变长编码

查询处理：从关键词到结果列表

单关键词查询 "Java"：
  词典查找 "Java" → 获取倒排列表 [Doc1, Doc3] → 返回结果

多关键词查询 "Java programming"：
  查找 "Java" → [Doc1, Doc3]
  查找 "programming" → [Doc1, Doc2]
  
  AND查询（交集）：[Doc1, Doc3] ∩ [Doc1, Doc2] = [Doc1]
  OR查询（并集）：[Doc1, Doc3] ∪ [Doc1, Doc2] = [Doc1, Doc2, Doc3]
  
  交集算法（跳表指针优化）：
    Doc1 Doc3      Doc1 Doc2
     ↑              ↑
    相等 → 输出Doc1，双指针前进
    
    Doc3           Doc2
     ↑              ↑
    Doc3 > Doc2 → Doc2指针前进
    
    Doc3           结束
    Doc2指针到底 → 结束

实现原理与关键机制

TF-IDF：经典相关性评分

TF (Term Frequency): 词项在文档中的频率
  TF(t,d) = (词t在文档d中出现次数) / (文档d的总词数)

IDF (Inverse Document Frequency): 词项的稀有程度
  IDF(t) = log(总文档数 / 包含词t的文档数)

TF-IDF = TF * IDF

词t对文档d的相关性得分：词出现越多、越稀有，得分越高

示例：
  总文档数：1000
  "Java"出现在100篇文档中 → IDF = log(1000/100) = log(10) ≈ 2.3
  "the"出现在990篇文档中 → IDF = log(1000/990) ≈ 0.004

倒排列表压缩技术

原始文档ID列表：[10, 11, 12, 15, 16, 20, 25]

差值编码（Delta Encoding）：
  [10, 1, 1, 3, 1, 4, 5]  （相邻差值）
  小整数更容易压缩

变长编码（Variable Byte / Gamma编码）：
  小数值用更少bit存储

Frame Of Reference（FOR）：
  分块（如128个一组），每块记录最小值，其余存偏移
  适合密集型文档ID

Roaring Bitmap：
  稀疏时用数组，密集时用位图，自动切换
  现代搜索引擎（Lucene 5+）采用

词典的数据结构演进

结构查询复杂度空间适用场景
排序数组O(log n) 二分小静态索引
B+树O(log n)中动态索引
前缀树（Trie）O(k) k为词长大前缀匹配、自动补全
FST（有限状态转换）O(k)极小Lucene采用，前缀共享
FST 压缩示例：

普通前缀树存储 ["car", "cat", "dog"]：
      root
     /    \
    c      d
   / \      \
  a   a      o
 /     \      \
r       t      g

FST 共享前缀并最小化状态：
  c → a → [r|t]  （共享"ca"前缀）
  d → o → g

实际应用中的决策与权衡

场景索引策略原因
搜索引擎分片倒排索引 + 分布式查询数据量大，单机无法承载
电商商品搜索倒排 + 属性过滤 + 评分排序多维度查询需求
日志检索时间分片 + 列式存储按时间范围查询为主
站内搜索轻量级倒排（如Lucene）中等数据量，快速部署
实时搜索内存索引 + 定期合并磁盘索引平衡实时性与持久性

Elasticsearch的索引架构：

索引分片结构：

Index: "articles"
├── Primary Shard 0 (Lucene索引)
│   ├── Segment 1 (不可变)
│   ├── Segment 2 (不可变)
│   └── Segment 3 (正在写入)
├── Primary Shard 1 (Lucene索引)
│   └── ...
├── Replica Shard 0 (Shard 0的副本)
└── Replica Shard 1 (Shard 1的副本)

写入流程：
  文档 → 内存缓冲区 → 刷新为Segment → 合并Segment
  
Segment不可变特性：
  - 写入新Segment而非修改旧Segment
  - 删除标记为逻辑删除（.del文件）
  - 定期Merge回收空间

常见问题与误区

问题1：倒排索引为什么不适合模糊查询？

根因：倒排索引按精确词项组织，前缀匹配可用Trie，但通配符中间匹配（如 *ava）需要扫描全词典 解决：N-gram分词、布隆过滤器辅助、正则索引

问题2：索引更新为什么慢？

根因：倒排列表是有序结构，插入新文档需要修改多个词项的列表，Segment合并开销大 解决：近实时搜索（1秒刷新）、异步合并、内存索引缓冲

问题3：为什么搜索引擎返回结果有相关性分数？

根因：单纯返回包含关键词的文档数量可能巨大，需要排序 方法：TF-IDF、BM25（考虑文档长度归一化）、向量相似度（语义搜索）
