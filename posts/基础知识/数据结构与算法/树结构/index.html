树结构

核心结论：红黑树通过颜色约束维持近似平衡保证O(log n)操作，B+树通过多叉结构降低磁盘IO成为数据库索引首选，跳表用概率平衡替代严格平衡实现简单高效的并发有序结构。

红黑树、B+树、跳表分别通过颜色约束、多叉节点和概率层级三种不同机制实现高效查找

红黑树：近似平衡的二叉搜索树

红黑树的五条性质：

  1. 节点是红色或黑色
  2. 根节点是黑色
  3. 所有叶子（NIL）是黑色
  4. 红色节点的子节点必须是黑色（不能有两个连续红色）
  5. 从任一节点到其叶子的所有路径包含相同数目的黑色节点
性质5的推论：最长路径不超过最短路径的2倍

最短路径：全黑节点        最长路径：红黑交替
    B                       B
   / \                     / \
  B   B                   R   B
 / \                     / \   \
B   B                   B   R   B
                            /
                           B

黑高=3，最短路径3，最长路径5（不超过2倍）

红黑树的平衡调整：

情况触发条件调整操作
左旋/右旋局部不平衡旋转节点，调整父子关系
变色双红冲突父、叔、祖父节点颜色翻转
LR/RL双旋折线关系先左旋/右旋，再右旋/左旋
插入调整示例（父节点红色，叔节点黑色）：

    G(黑)              P(黑)
   /                  /    \
  P(红)    →        N(红)  G(红)
   \                       /
   N(红)                S(黑)

情况：父红、叔黑、新节点是父的右孩子（折线）
调整：先对P左旋 → 再对G右旋 → 变色

红黑树 vs AVL树：

维度红黑树AVL树
平衡条件近似平衡（最长≤2倍最短）严格平衡（高度差≤1）
查找O(log n)，略慢O(log n)，更快
插入/删除旋转次数少（O(1)调整）旋转次数多（O(log n)调整）
适用场景频繁插入删除（Java TreeMap）查找频繁、修改少

B+树：数据库索引的多路平衡树

B+树结构（m=4阶）：

          [10, 20, 30]                    ← 内部节点：存储键，引导查找
         /    |    |    \
   [5,7,9] [15,18] [25,28] [35,40,45]    ← 叶子节点：存储实际数据/指针
    ↓↓       ↓↓      ↓↓       ↓↓↓
   数据页    数据页   数据页    数据页
    
叶子节点通过链表连接，支持范围扫描：
[5,7,9] → [15,18] → [25,28] → [35,40,45]

B+树的设计优势（针对磁盘IO优化）：

特性作用
多叉结构降低树高，减少磁盘读取次数
节点大小=页大小（4KB/16KB）一次IO读取一个完整节点
数据只存叶子节点内部节点更小，容纳更多键，树更矮
叶子节点链表连接范围查询只需顺序遍历叶子
非叶子节点冗余键即使内部节点损坏，叶子仍能恢复索引
对比：二叉树 vs B+树查找100万条数据

红黑树（高度~20）：
  每个节点一次磁盘IO → 最坏20次IO

B+树（4阶，高度~3）：
  根节点常驻内存 → 2-3次IO即可定位

跳表：概率平衡的有序链表

跳表结构（概率p=0.5，最大层数16）：

Level 3:  head ────────────────────→  12
Level 2:  head ──────────→ 6 ──────→ 12 ──────→ 20
Level 1:  head ──→ 3 ──→ 6 ──→ 9 ──→ 12 ──→ 15 ──→ 20
Level 0:  head → 1 → 3 → 4 → 6 → 7 → 9 → 10 → 12 → 13 → 15 → 17 → 20
                                          ↑
                                       实际数据
                                        
查找15：从顶层开始，向右或向下
  head(L3) → 12(L3) → down → 12(L2) → 20(L2) > 15 → down 
  → 12(L1) → 15(L1) ✓
  
查找时间复杂度：O(log n)（期望，概率保证）

跳表的层级生成：

// Redis中的跳表实现
int zslRandomLevel(void) {
    int level = 1;
    while ((random() & 0xFFFF) < (ZSKIPLIST_P * 0xFFFF))
        level += 1;
    return (level < ZSKIPLIST_MAXLEVEL) ? level : ZSKIPLIST_MAXLEVEL;
}
// p=0.25时：50%概率1层，37.5%概率2层，9.4%概率3层...

跳表 vs 红黑树：

维度跳表红黑树
实现复杂度简单复杂
范围查询极简单（遍历底层链表）中序遍历
并发安全容易（锁分层或CAS）困难（旋转涉及多节点）
内存占用稍多（指针数组）较少
实际应用Redis ZSet、LevelDBJava TreeMap、C++ std::map

实现原理与关键机制

红黑树的旋转操作

左旋（以x为支点）：

    x                    y
   / \                  / \
  a   y      →         x   c
     / \              / \
    b   c            a   b

右旋是左旋的逆操作

旋转的性质：
- 中序遍历顺序不变（保持二叉搜索树性质）
- 改变树的高度分布，使其更平衡

B+树的插入与分裂

插入过程：
1. 从根节点找到合适的叶子节点
2. 若叶子未满（键数 < m-1），直接插入
3. 若叶子已满，分裂为两个节点，中间键提升到父节点
4. 若父节点也满，递归分裂向上

分裂示例（m=4，叶子最多3个键）：

插入15到 [10, 20, 30]：
  叶子分裂：[10, 15] 和 [20, 30]
  中间键15提升到父节点
  
  原父节点：[5, 10, 20, 25]（满）
  父节点也分裂：[5, 10] 和 [20, 25]
  中间键15提升到新根节点
  
  树高增加1层

跳表的索引节点概率

设p=0.5（Redis实际用0.25）

节点层数概率分布：
P(1层) = 1 - p = 0.5
P(2层) = p * (1-p) = 0.25
P(3层) = p² * (1-p) = 0.125
...
P(k层) = p^(k-1) * (1-p)

期望层数 = 1/(1-p) = 2层（p=0.5时）
期望查找步数 ≈ O(log_{1/p} n)

实际应用中的决策与权衡

场景推荐结构原因风险
内存索引、频繁增删红黑树调整代价低实现复杂
数据库磁盘索引B+树降低磁盘IO实现复杂
内存有序集合、简单实现跳表实现简单、并发友好内存稍多
需要排序的MapTreeMap（红黑树）标准库支持比HashMap慢
范围查询频繁B+树/跳表天然支持顺序遍历-
高并发有序结构跳表（ConcurrentSkipListMap）锁粒度小-

常见问题与误区

问题1：为什么数据库不用红黑树而用B+树？

根因：红黑树是二叉树，树高随数据量线性增长，磁盘IO次数多 解决：B+树是多叉树，节点大小匹配磁盘页（4KB/16KB），大幅降低树高

问题2：跳表性能不如红黑树稳定？

根因：跳表是概率平衡，最坏情况O(n) 事实：实际中概率极低，Redis和LevelDB生产验证性能优秀，实现简单带来的工程收益远大于理论风险

问题3：B+树和B树的区别？

根因：B树数据存所有节点，B+树数据只存叶子 差异：B+树内部节点更小容纳更多键、叶子链表支持范围查询、查询性能更稳定（都到叶子）
