Hash表

核心结论：Hash表通过哈希函数将键映射到数组索引，实现O(1)的平均查找效率，冲突解决和扩容机制是工程实现的关键，一致性Hash则解决了分布式环境下的数据分布问题。

Hash表通过哈希函数+数组实现O(1)查找，冲突解决和动态扩容是工程实现的核心机制

核心原理：哈希函数将任意键映射到有限数组空间

键(Key) ──→ 哈希函数 ──→ 哈希值 ──→ 取模 ──→ 数组索引
                                
"apple"    hashCode()    93029231    % 16     → 15
"banana"   hashCode()    63595437    % 16     → 13
"cherry"   hashCode()    12345678    % 16     → 14
// Java HashMap 的扰动函数：高16位与低16位异或，减少低位的规律性
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}

// 数组索引计算：(n-1) & hash 等价于 hash % n，当n为2的幂时
// n=16:  (16-1) & hash = 0b1111 & hash  → 取低4位

冲突解决：链地址法 vs 开放寻址法

方法原理优点缺点
链地址法冲突元素组成链表/树实现简单、删除方便缓存不友好、链表过长退化
开放寻址法冲突时探测下一个空位缓存友好、无指针开销删除复杂、装填因子受限
再哈希法多个哈希函数依次尝试分布均匀计算开销大

链地址法（Java HashMap采用）：

数组索引:  0     1     2     3     ...   13     14     15
          ↓     ↓     ↓     ↓           ↓      ↓      ↓
         null  null  A→B   null        D      E→F→G   H
                    ↓                      ↓
                    C                      null

JDK7：链表头插法（并发扩容可能产生环形链表）
JDK8+：链表尾插法 + 长度≥8转红黑树

开放寻址法（ThreadLocalMap采用）：

线性探测：hash(key) → 被占 → hash+1 → 被占 → hash+2 → 空位插入
二次探测：hash(key) → 被占 → hash+1² → 被占 → hash+2² → 空位插入
双重哈希：hash1(key) → 被占 → hash1 + i*hash2(key) → 空位

扩容机制：容量翻倍 + 重新散列

// Java HashMap 扩容触发条件：size > capacity * loadFactor
// 默认：capacity=16, loadFactor=0.75, 阈值=12

扩容过程：
1. 创建新数组，容量为原来的2倍
2. 遍历旧数组，将每个元素重新计算索引（hash & (newCap-1)）
3. 放入新数组

JDK8优化：
- 容量为2的幂，扩容后元素新索引只可能是 原索引 或 原索引+旧容量
- 只需检查hash新增的那一位是0还是1

旧容量=16，hash=...1101
  hash & (16-1) = hash & 1111 = 1101 = 13
  hash & (32-1) = hash & 11111 = 01101 = 13（新增位是0）
  
旧容量=16，hash=...11101
  hash & (16-1) = 1101 = 13
  hash & (32-1) = 11101 = 29 = 13 + 16（新增位是1）

实现原理与关键机制

HashMap 红黑树转换阈值为什么是8

泊松分布：当负载因子为0.75时，单个桶中元素个数的概率

长度    概率
0       0.6065
1       0.3033
2       0.0758
3       0.0126
4       0.0016
5       0.00016
6       0.000013
7       0.0000009
8       0.00000006

结论：正常情况下链表长度达到8的概率极低（千万分之六），
      达到8说明哈希函数被攻击（精心构造冲突键）或实现有问题
      此时转红黑树（O(log n)）防止退化到O(n)
      
树化阈值8，反树化阈值6（防止在8附近频繁转换）

一致性Hash：分布式环境下的数据分布

传统Hash取模：server = hash(key) % N
问题：N变化时，几乎所有数据需要重新映射

一致性Hash：将服务器和数据都映射到同一个Hash环上

      0°                                360°
      ├─────────────────────────────────┤
      S1(30°)      S2(150°)      S3(280°)
         ↓           ↓             ↓
      顺时针找到最近的服务器
      
数据D1(45°) → 顺时针 → S2(150°)
数据D2(200°) → 顺时针 → S3(280°)
数据D3(300°) → 顺时针 → S1(30°)

S2下线时：
  D1(45°) 原来→S2，现在→S3(280°)（顺时针最近）
  仅D1需要迁移，D2/D3不受影响

虚拟节点：为每个物理服务器创建多个虚拟节点，均匀分布数据

S1 → S1#1(10°), S1#2(90°), S1#3(170°), S1#4(250°), S1#5(330°)
S2 → S2#1(50°), S2#2(130°), S2#3(210°), S2#4(290°), S2#5(350°)

数据均匀分布到多个虚拟节点，物理服务器故障时负载均匀转移

实际应用中的决策与权衡

场景推荐实现原因风险
单线程环境HashMap无锁开销非线程安全
高并发读多写少ConcurrentHashMap读无锁，写细粒度锁size()非精确值
需要有序LinkedHashMap维护插入/访问顺序额外指针开销
需要排序TreeMap红黑树，O(log n)无O(1)查找
分布式缓存一致性Hash + 虚拟节点节点变动时最小迁移数据倾斜需调虚拟节点数
大量数据、内存敏感开放寻址法（如ThreadLocalMap）无指针开销删除困难

常见问题与误区

问题1：HashMap并发扩容导致死循环（JDK7）

根因：头插法 + 多线程并发扩容 → 链表成环 解决：升级到JDK8+（尾插法+红黑树），或使用ConcurrentHashMap

问题2：自定义对象作为HashMap的Key，取出失败

根因：未重写hashCode()和equals()，默认使用Object的引用比较 解决：重写hashCode()和equals()，且equals相等的对象hashCode必须相等

问题3：HashMap初始容量设置不当导致频繁扩容

根因：未预估数据量，默认16多次扩容 解决：new HashMap<>(预期数量 / 0.75f + 1)
