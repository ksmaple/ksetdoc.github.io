HTTP演进

核心结论：HTTP从1.0到3.0的演进主线是"减少连接数→复用连接→消除队头阻塞→降低握手延迟"，每次升级都在解决前一代的瓶颈，但协议复杂度也相应增加。

HTTP从1.0到3.0围绕连接效率、队头阻塞、握手延迟三个核心问题逐步优化

HTTP/1.0：无状态的请求-响应协议

每次请求新建TCP连接：

客户端                  服务端
  |  TCP握手 (SYN-SYN/ACK-ACK)  |
  | <-------------------------> |
  |      GET /page1.html        |
  |  ------------------------>  |
  |      200 OK + 数据           |
  |  <------------------------  |
  |      TCP挥手 (FIN-ACK...)   |
  | <-------------------------> |
  
  [下一个请求重复以上全过程]

核心问题：

  • 每个请求独立TCP连接，三次握手开销大
  • 无法复用连接，浏览器同时只能发少量请求（6-8个域名并发）

HTTP/1.1：持久连接与管道化

特性说明解决的问题
持久连接（Keep-Alive）复用TCP连接发送多个请求减少握手开销
管道化（Pipelining）连续发送多个请求不等待响应提高吞吐量
分块传输（Chunked）服务端分块发送，无需预知Content-Length动态内容传输
缓存控制Cache-Control、ETag、Last-Modified减少重复传输
Host头支持同一IP多域名虚拟主机
HTTP/1.1 持久连接：

客户端                  服务端
  |  TCP握手                |
  | <---------------------> |
  |  GET /page1.html        |
  |  -------------------->  |
  |  200 OK                 |
  |  <--------------------  |
  |  GET /style.css         |
  |  -------------------->  |
  |  200 OK                 |
  |  <--------------------  |
  |  GET /image.jpg         |
  |  -------------------->  |
  |  200 OK                 |
  |  <--------------------  |
  |  Connection: close      |
  |  [可选挥手或保持]         |

HTTP/1.1 的队头阻塞问题：

请求顺序：Req1 → Req2 → Req3

场景：Req2的处理需要1秒，Req3只需要10ms

HTTP/1.1（无管道化）：串行发送，Req3必须等Req2完成
HTTP/1.1（管道化）：可连续发送，但响应必须按请求顺序返回
  → 如果Req2响应慢，Req3即使处理完也必须等（队头阻塞）

解决方案——域名分片：

  • 将资源分散到多个子域名（a.cdn.com, b.cdn.com）
  • 浏览器对每个域名建立6个连接，变相提高并发
  • 副作用：DNS查询增多、TCP连接数增加

HTTP/2：二进制分帧与多路复用

特性说明解决的问题
二进制分帧数据拆分为Frame传输解析效率更高
多路复用单一TCP连接并发多个请求/响应队头阻塞（应用层）
头部压缩（HPACK）动态表压缩重复头部减少冗余传输
服务端推送服务端主动发送资源减少RTT
流优先级客户端指定资源加载优先级优化体验
HTTP/2 多路复用：

单一TCP连接内，请求/响应被拆分为Frame，交错传输：

客户端 ────────────────────────────────────── 服务端
  │  HEADERS[StreamID=1] :method GET :path /page   │
  │ ──────────────────────────────────────────────>│
  │  HEADERS[StreamID=3] :method GET :path /css    │
  │ ──────────────────────────────────────────────>│
  │  DATA[StreamID=3]  CSS内容（先完成）            │
  │ <──────────────────────────────────────────────│
  │  HEADERS[StreamID=1] 200 OK                    │
  │ <──────────────────────────────────────────────│
  │  DATA[StreamID=1]  HTML内容                    │
  │ <──────────────────────────────────────────────│
  │  HEADERS[StreamID=5] :method GET :path /js     │
  │ ──────────────────────────────────────────────>│

Stream相互独立，一个Stream阻塞不影响其他Stream

HTTP/2 的TCP队头阻塞：

HTTP/2解决了应用层队头阻塞，但无法解决传输层队头阻塞：

Stream1 Frame → Stream3 Frame → Stream1 Frame → Stream5 Frame
       ↓
   [TCP传输]
       ↓
若某个TCP包丢失，整个TCP连接阻塞等待重传
→ 所有Stream都受影响（即使其他Stream的数据包已到达）

HTTP/3：基于QUIC的无队头阻塞传输

特性说明优势
基于UDP放弃TCP，使用QUIC协议避免TCP队头阻塞
0-RTT握手首次连接1-RTT，后续0-RTT降低延迟
连接迁移连接ID标识而非IP+端口网络切换（WiFi→4G）保持连接
内置TLSQUIC强制集成TLS 1.3安全且减少握手往返
HTTP/2 vs HTTP/3 队头阻塞对比：

HTTP/2 (TCP):
  Stream1 Packet1 → Stream3 Packet1 → Stream1 Packet2
         ↓
     [TCP单流]
         ↓
  Packet1丢失 → 全部Stream阻塞等待重传

HTTP/3 (QUIC/UDP):
  Stream1 Packet1 → Stream3 Packet1 → Stream1 Packet2
         ↓           ↓                ↓
     [QUIC独立流] [QUIC独立流]  [QUIC独立流]
         ↓           ↓                ↓
  Stream1 Packet丢失 → 仅Stream1阻塞，Stream3不受影响

QUIC握手：

首次连接（1-RTT）：
  客户端: Initial(CHLO + TLS ClientHello)
  服务端: Initial + Handshake(SHLO + TLS ServerHello + 证书)
  客户端: Handshake(认证完成 + 请求数据)

后续连接（0-RTT）：
  客户端: Initial(复用之前的会话票据 + 请求数据)
  [无需等待服务端响应即可发送应用数据]

实现原理与关键机制

HPACK头部压缩：静态表 + 动态表 + 哈夫曼编码

HTTP/1.1 头部冗余：
  GET /page1
  Host: example.com
  User-Agent: Mozilla/5.0 ...
  Accept: text/html
  
  GET /page2
  Host: example.com        ← 重复
  User-Agent: Mozilla/5.0 ...  ← 重复
  Accept: text/html        ← 重复

HPACK压缩后：
  静态表索引：:method:GET → 索引2
  动态表索引：Host:example.com → 动态表索引62（首次发送后缓存）
  哈夫曼编码：长字符串压缩

HTTP/2 流与优先级

Stream是HTTP/2的逻辑通道：
- 每个Stream有唯一的31位ID
- 客户端发起的StreamID为奇数，服务端发起的为偶数
- Stream可设置优先级和依赖关系

优先级示例：
  HTML (Stream1, 权重256) ──┐
                              ├── 依赖关系树
  CSS  (Stream3, 权重128) ──┤
                              ├── JS (Stream5, 权重64) ── 依赖CSS
  图片 (Stream7, 权重32) ────┘

实际应用中的决策与权衡

场景推荐版本原因风险
通用Web服务HTTP/2主流支持，显著性能提升需要TLS（浏览器要求）
移动端/API网关HTTP/3网络切换不断连，0-RTT中间设备可能拦截UDP
内部微服务HTTP/2或gRPC多路复用减少连接数gRPC需额外学习成本
遗留系统兼容HTTP/1.1最大兼容性性能瓶颈
实时音视频WebSocket/QUIC低延迟双向通信HTTP语义不匹配

常见问题与误区

问题1：HTTP/2一定比HTTP/1.1快？

根因：小页面、低延迟网络差异不大；大页面、高延迟网络提升明显 结论：HTTP/2主要优势在减少连接数和头部压缩，单请求延迟优化有限

问题2：HTTP/3的0-RTT有安全风险？

根因：0-RTT数据可能被重放攻击（Replay Attack） 解决：TLS 1.3限制0-RTT只允许幂等操作（GET/HEAD），禁止非幂等操作

问题3：为什么HTTP/2强制TLS？

根因：浏览器厂商共同推动（Chrome/Firefox只支持HTTPS上的HTTP/2），防止中间设备干扰二进制帧 事实：HTTP/2协议本身不强制TLS，但实际部署几乎全是h2（TLS）
