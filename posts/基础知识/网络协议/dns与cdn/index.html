DNS解析与CDN

核心结论：DNS通过分布式层级系统将域名解析为IP地址，CDN利用DNS的智能调度和边缘节点缓存，将内容分发到离用户最近的节点，二者结合是互联网性能优化的基础设施。

DNS通过分布式层级数据库实现域名到IP的映射，CDN在此基础上叠加边缘缓存和智能调度

DNS域名空间：树状层级结构

                    根域名服务器 (.)
                         │
    ┌────────────────────┼────────────────────┐
    │                    │                    │
   com                  org                  cn
    │                    │                    │
 example              wikipedia            baidu
    │                    │                    │
   www                 www                  www
    │
  记录: A 93.184.216.34
        CNAME example.cdn.com
层级说明示例
根域名（Root）最顶层，全球13组（字母A-M）.
顶级域（TLD）通用（gTLD）或国家/地区（ccTLD）.com .org .cn
二级域注册购买的主体域名example.com
子域二级域下的扩展www.example.com api.example.com

DNS解析流程：从递归查询到本地缓存

用户输入 www.example.com

1. 浏览器缓存 → 检查是否最近解析过
2. 操作系统缓存 (/etc/hosts, DNS Client Service)
3. 本地DNS服务器（ISP或公共DNS如8.8.8.8）
   └── 递归查询开始 ──────────────────────────┐
                                              │
4. 本地DNS → 根域名服务器（.）                 │
   ← 返回 com 的NS记录                        │
                                              │
5. 本地DNS → com 顶级域服务器                  │
   ← 返回 example.com 的NS记录                │
                                              │
6. 本地DNS → example.com 权威DNS服务器         │
   ← 返回 www.example.com 的A记录: 93.184.216.34│
                                              │
7. 本地DNS缓存结果，返回给用户 ────────────────┘

总耗时：通常 20-120ms（取决于缓存命中率）

DNS记录类型：

类型用途示例
AIPv4地址www IN A 93.184.216.34
AAAAIPv6地址www IN AAAA 2606:2800:220:1::
CNAME别名指向另一域名www IN CNAME example.github.io
MX邮件服务器@ IN MX 10 mail.example.com
NS权威DNS服务器example.com IN NS ns1.example.com
TXT文本信息（SPF、DKIM验证）@ IN TXT "v=spf1 include:_spf.google.com ~all"
SOA区域授权起始包含主DNS、序列号、刷新时间等

CDN加速原理：边缘节点缓存 + 智能调度

无CDN：
  北京用户 ──────────────────────────→ 美国源站
         200ms+ RTT, 跨洋带宽瓶颈

有CDN：
  北京用户 ──→ 北京CDN节点（边缘缓存）
              命中缓存 → 直接返回（<10ms）
              未命中 → 回源获取 → 缓存并返回
                    ↓
              源站（仅首次访问或缓存过期时请求）

CDN节点层级：
  边缘节点（Edge）：最接近用户，缓存热点内容
    ↑ 未命中
  区域节点（Parent）：汇聚多个边缘节点请求
    ↑ 未命中
  源站（Origin）：原始服务器

CDN的智能调度：

调度方式原理适用场景
DNS调度根据用户IP返回最近的CDN节点IP通用场景，最常用
HTTP DNS客户端通过HTTP接口获取最优IP绕过ISP DNS劫持
Anycast同一IP在不同地理位置广播抗DDoS，自动路由
302跳转首次请求到中心节点，再302到边缘精确调度，额外RTT

实现原理与关键机制

DNS负载均衡：一个域名对应多个IP

www.example.com 的A记录：

;; ANSWER SECTION:
www.example.com.    300    IN    A    192.168.1.10
www.example.com.    300    IN    A    192.168.1.11
www.example.com.    300    IN    A    192.168.1.12

负载均衡策略：
- 轮询（Round Robin）：按顺序返回不同IP
- 权重轮询：根据服务器性能分配权重
- 地理就近：返回离用户最近的数据中心IP
- 健康检查：自动剔除故障IP

CDN缓存策略：TTL与缓存失效

HTTP响应头控制缓存：

Cache-Control: max-age=3600        # 缓存1小时
Cache-Control: no-cache            # 每次回源验证
Cache-Control: no-store            # 完全不缓存
Expires: Wed, 21 Oct 2026 07:28:00 GMT  # 绝对过期时间

CDN专用头：
X-Cache: HIT                       # CDN边缘节点命中
X-Cache: MISS                      # CDN边缘节点未命中
Age: 1200                          # 缓存已存在1200秒

缓存刷新方式：

  • URL刷新：主动使特定URL缓存失效
  • 目录刷新：使某目录下所有内容失效
  • 预热：在高峰前主动将内容推送到边缘节点
  • 版本号：script.v2.js 替代 script.js，天然缓存隔离

DNSSEC：防止DNS劫持和缓存投毒

DNSSEC通过数字签名验证DNS响应真实性：

根域名 (.) ──KSK──→ 签名 .com 的ZSK
    │
    └── .com ──ZSK──→ 签名 example.com 的ZSK
              │
              └── example.com ──ZSK──→ 签名 www.example.com 的A记录

验证链：
  根KSK（预装在系统）→ 验证 .com ZSK → 验证 example.com ZSK → 验证记录签名

实际应用中的决策与权衡

场景配置建议原因
静态资源加速CDN全站加速缓存JS/CSS/图片，减少源站带宽
API接口CDN动态加速或不用API数据变化快，缓存收益低
全球化业务多地域CDN + 智能DNS就近访问，降低延迟
大文件下载CDN分段缓存 + P2P补充减少回源带宽
高安全要求DNSSEC + HTTPS + CDN WAF防劫持、防篡改、防攻击

CDN选型考量：

维度自建CDN商业CDN（阿里云/腾讯云/AWS CloudFront）
节点覆盖可控，需自建全球节点，即开即用
成本前期投入大，长期可控按流量/请求计费，弹性
定制能力完全可控受限于平台能力
运维复杂度高低

常见问题与误区

问题1：DNS解析慢，导致首屏时间增加

根因：递归查询链路长、DNS服务器响应慢 解决：

  • 使用公共DNS（114.114.114.114、223.5.5.5、8.8.8.8）
  • 增大TTL值（减少重复查询）
  • 使用HTTP DNS绕过ISP DNS
  • CDN的DNS调度优化

问题2：CDN缓存导致内容更新不及时

根因：缓存TTL设置过长，未主动刷新 解决：

  • 更新时主动调用CDN刷新API
  • 使用版本号/文件名哈希（style.a3f2b1.css）
  • 缩短关键资源的TTL

问题3：DNS负载均衡不均匀

根因：DNS轮询不考虑服务器实际负载，且客户端可能缓存第一个IP 解决：

  • 结合健康检查自动剔除故障节点
  • 使用权重轮询
  • 应用层负载均衡（如LVS、Nginx upstream）作为补充
