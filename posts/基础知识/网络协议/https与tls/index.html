HTTPS与TLS

核心结论：HTTPS通过TLS协议在HTTP之上提供加密、认证和完整性保护，TLS握手协商对称密钥，证书链验证服务端身份，是当前Web安全的基石。

HTTPS通过TLS协议实现加密传输、身份认证和数据完整性，TLS握手是性能和安全的关键

TLS/SSL演进：从SSL到TLS 1.3

版本年份关键变化现状
SSL 2.01995首个公开发布版本已废弃（严重安全漏洞）
SSL 3.01996广泛应用已废弃（POODLE攻击）
TLS 1.01999标准化，与SSL 3.0几乎相同已废弃
TLS 1.12006引入显式IV防CBC攻击已废弃
TLS 1.22008支持AEAD、SHA-256、扩展当前主流
TLS 1.32018简化握手、删除弱算法、0-RTT推荐部署

TLS 1.2 握手：2-RTT建立安全连接

客户端                                               服务端
  |        ClientHello (支持的加密套件、随机数)        |
  |  ──────────────────────────────────────────────>  |
  |                                                   |
  |  ServerHello (选定加密套件、随机数)                 |
  |  Certificate (服务端证书链)                         |
  |  [ServerKeyExchange] (DHE/ECDHE参数)              |
  |  ServerHelloDone                                   |
  |  <──────────────────────────────────────────────  |
  |                                                   |
  |  ClientKeyExchange (预主密钥，用公钥加密)            |
  |  [ChangeCipherSpec]                                |
  |  Finished (验证握手完整性)                          |
  |  ──────────────────────────────────────────────>  |
  |                                                   |
  |  [ChangeCipherSpec]                                |
  |  Finished                                          |
  |  <──────────────────────────────────────────────  |
  |                                                   |
  ↓                                                   ↓
应用数据传输（对称加密）

总耗时：2-RTT（ClientHello → ServerHello → Finished）

TLS 1.3 握手：1-RTT（首次）/ 0-RTT（重连）

首次连接（1-RTT）：

客户端                                               服务端
  |  ClientHello (密钥共享 + 支持的组)                 |
  |  ──────────────────────────────────────────────>  |
  |                                                   |
  |  ServerHello (选定密钥共享)                         |
  |  {EncryptedExtensions} (加密传输)                  |
  |  {Certificate}                                     |
  |  {CertificateVerify}                               |
  |  {Finished}                                        |
  |  <──────────────────────────────────────────────  |
  |                                                   |
  |  {Finished}                                        |
  |  ──────────────────────────────────────────────>  |
  |                                                   |
  ↓                                                   ↓
应用数据传输

重连（0-RTT）：
  客户端发送早期数据（Early Data），使用上一次会话的PSK
  → 无需等待服务端响应即可发送应用数据

TLS 1.3 的简化：

  • 删除RSA密钥交换（仅支持前向安全的DHE/ECDHE）
  • 删除大量弱算法（MD5、SHA-1、RC4、CBC模式等）
  • 握手消息加密（除ClientHello外）
  • 支持0-RTT（会话恢复）

证书链验证：信任根到服务端的完整链路

证书链验证过程：

服务端证书 (example.com)
    ↑ 由 Intermediate CA 签名
Intermediate CA (Let's Encrypt R3)
    ↑ 由 Root CA 签名
Root CA (ISRG Root X1)
    ↑ 已预装在操作系统/浏览器信任库
  可信

验证步骤：
1. 检查证书有效期（Not Before / Not After）
2. 检查域名匹配（Subject CN / SAN）
3. 验证签名链（用上级公钥验证下级签名）
4. 检查CRL或OCSP（证书是否被吊销）
5. 根证书是否在本地信任库

实现原理与关键机制

对称加密 vs 非对称加密的分工

阶段算法类型作用原因
握手阶段非对称加密（RSA/ECDHE）安全交换对称密钥解决密钥分发问题
数据传输对称加密（AES/GCM/ChaCha20）高效加密应用数据速度快（1000x+）
密钥交换方式对比：

RSA密钥交换：
  客户端生成预主密钥 → 服务端公钥加密 → 服务端私钥解密
  问题：无"前向安全性"（服务端私钥泄露可解密历史流量）

ECDHE密钥交换：
  客户端和服务端各自生成临时密钥对，交换公钥
  双方通过ECDH计算共享密钥（预主密钥）
  即使私钥泄露，也无法计算历史会话的共享密钥（前向安全）

HTTPS的完整请求流程

浏览器访问 https://example.com

1. DNS解析 → 获取IP地址
2. TCP三次握手 → 建立连接
3. TLS握手 → 协商加密参数（1-RTT for TLS 1.3）
4. 应用层数据传输（HTTP请求/响应，对称加密）
5. 连接保持或关闭

总耗时（TLS 1.3）：
  DNS(可能缓存) + TCP握手(1-RTT) + TLS握手(1-RTT) + 首个HTTP请求(1-RTT)
  = 3-RTT + DNS

HSTS：强制HTTPS访问

Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

作用：
1. 浏览器记住该域名必须通过HTTPS访问
2.  max-age内自动将HTTP重定向改为HTTPS（无需服务器跳转）
3. includeSubDomains: 对子域名同样生效
4. preload: 加入浏览器内置列表，首次访问即生效

实际应用中的决策与权衡

场景配置建议原因
生产环境TLS 1.2 + 1.3，禁用1.0/1.1兼容性+安全性平衡
高安全要求仅TLS 1.3删除所有弱算法
性能优先TLS 1.3 + 会话票据复用0-RTT重连
证书选择ECDSA证书（而非RSA）更小体积，更快握手
兼容性要求RSA + ECDSA双证书兼容老旧客户端

常见TLS配置检查：

# 检查服务器支持的TLS版本和加密套件
openssl s_client -connect example.com:443 -tls1_3

# 检查证书信息
openssl x509 -in cert.pem -text -noout

# 扫描SSL配置弱点
nmap --script ssl-enum-ciphers -p 443 example.com

常见问题与误区

问题1：HTTPS握手慢，影响首屏

根因：TLS握手增加1-2 RTT 解决：

  • 升级到TLS 1.3（1-RTT）
  • 开启会话复用（Session ID / Session Ticket）
  • 使用OCSP Stapling（避免客户端单独查询证书状态）
  • HTTP/2 Server Push（已废弃，建议用103 Early Hints）

问题2：证书过期导致服务不可用

根因：Let’s Encrypt证书90天有效期，商业证书通常1年 解决：自动化证书管理（certbot、acme.sh、各云厂商托管证书）

问题3：自签名证书浏览器报警

根因：自签名证书不在浏览器信任链中 解决：内网使用私有CA，客户端安装根证书；公网使用权威CA签发的证书
