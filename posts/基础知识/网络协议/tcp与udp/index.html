TCP vs UDP

核心结论：TCP通过三次握手、重传机制和流量控制实现可靠传输，适合数据完整性要求高的场景；UDP无连接、低延迟，适合实时性要求高于可靠性的场景。

TCP通过连接管理和拥塞控制实现可靠传输，UDP以无连接和最小开销换取低延迟

TCP核心机制：可靠传输的三重保障

机制解决的问题实现方式
连接管理端到端通信确认三次握手建立、四次挥手关闭
确认重传丢包恢复ACK确认 + 超时重传 + 快速重传
流量控制接收方处理不过来滑动窗口（rwnd）
拥塞控制网络拥塞慢启动、拥塞避免、快重传、快恢复

三次握手（建立连接）：

客户端                    服务端
   |     SYN, seq=x      |
   | ------------------> |
   |   SYN, ACK, seq=y, ack=x+1  |
   | <------------------ |
   |     ACK, seq=x+1, ack=y+1   |
   | ------------------> |
   |                     |
   ↓                     ↓
ESTABLISHED           ESTABLISHED

为什么是三次？

  • 两次不够：客户端无法确认服务端是否收到自己的ACK（防止历史重复连接）
  • 四次冗余：SYN和ACK可以合并为第二次握手

四次挥手（关闭连接）：

客户端                    服务端
   |     FIN, seq=u      |
   | ------------------> |
   |     ACK, ack=u+1    |
   | <------------------ |
   |     [数据传输中]     |
   |     FIN, seq=w      |
   | <------------------ |
   |     ACK, ack=w+1    |
   | ------------------> |
   |                     |
   ↓                     ↓
TIME_WAIT(2MSL)       CLOSED
CLOSED

TIME_WAIT状态（2MSL）的作用：

  1. 确保最后一个ACK能被服务端收到（若丢失，服务端重发FIN，客户端可重发ACK）
  2. 让网络中残留的数据包自然消失，避免下一个连接收到旧数据

滑动窗口：流量控制的动态平衡

发送端缓冲区：

|已确认|  已发送未确认  |    可发送    | 不可发送 |
|<-----|---------------|-------------|---------|
       ↑               ↑             ↑
     SND.UNA        SND.NXT      SND.UNA + SND.WND

接收端通告窗口（rwnd）根据缓冲区剩余空间动态调整：
- 应用读取快 → rwnd大 → 发送端发得快
- 应用读取慢 → rwnd小 → 发送端发得慢（甚至为0）

拥塞控制：慢启动与拥塞避免

拥塞窗口（cwnd）变化：

 cwnd
  ↑
  │      ╱ 拥塞避免（线性增长）
  │     ╱
  │    ╱        ╱ 新的慢启动（阈值减半）
  │   ╱        ╱
  │  ╱   × 拥塞点（丢包）
  │ ╱
  │╱ 慢启动（指数增长）
  └──────────────────────→ 时间
阶段触发条件cwnd变化特点
慢启动连接建立或超时重传每ACK +1（指数增长）快速探测带宽
拥塞避免cwnd ≥ ssthresh每RTT +1（线性增长）保守增长
快重传收到3个重复ACK立即重传，不等待超时快速恢复
快恢复快重传后ssthresh=cwnd/2, cwnd=ssthresh+3避免回到慢启动

UDP：无连接的最小传输协议

UDP头部结构（仅8字节）

 0      7 8     15 16    23 24    31
+--------+--------+--------+--------+
|     源端口      |     目的端口    |
+--------+--------+--------+--------+
|     长度        |     校验和      |
+--------+--------+--------+--------+
|              数据（可选）           |

UDP的特点：

  • 无连接：发送前不需握手，直接发数据
  • 不可靠：不保证到达、不保证顺序、不重传
  • 无拥塞控制：发送速率由应用决定
  • 头部开销小：8字节 vs TCP的20字节

UDP的应用场景与加强方案

场景为什么选UDP可靠性补充方案
DNS查询单次请求响应，连接开销不划算超时重传（应用层）
音视频通话实时性优先，丢包可容忍FEC前向纠错、NACK选择性重传
在线游戏低延迟优先，旧数据无意义状态同步、客户端预测
QUIC/HTTP3避免TCP队头阻塞基于UDP的应用层可靠传输

实现原理与关键机制

TCP粘包与拆包：面向字节流的特性

TCP发送：发送方写入10字节 + 20字节
TCP传输：[30字节数据流]
TCP接收：接收方可能一次性读到30字节，也可能分两次读（10+20）

问题：接收方无法区分两次写入的边界
解决：
1. 固定长度消息
2. 特殊分隔符（如\n）
3. 消息头+长度字段（最常用）
   | 长度(4B) | 消息体(NB) |

TCP队头阻塞：可靠传输的代价

数据包发送顺序：  P1 → P2 → P3 → P4

场景1：P2丢失
  接收端：P1 ✓  P2 ✗  P3 ✓  P4 ✓
  结果：P3、P4已到达但无法交付应用（TCP必须按序），等待P2重传

场景2：HTTP/1.1 管道化
  请求1 → 请求2 → 请求3（同一连接串行）
  若请求1响应慢，请求2、3阻塞

HTTP/3解决方案：基于QUIC（UDP），每个流独立传输，单个流丢包不影响其他流

实际应用中的决策与权衡

场景推荐协议原因风险
文件传输、网页浏览TCP数据完整性必须保证高延迟网络性能差
实时音视频UDP + 应用层FEC100ms延迟比1%丢包更影响体验高丢包率下画质/音质下降
DNS查询UDP（默认）单次请求响应，开销小大响应需转TCP
金融交易TCP + 应用层心跳数据不能丢失网络抖动导致重连
物联网传感器UDP/MQTT设备资源受限需应用层保障可靠性
内网微服务通信TCP/自定义协议可靠 + 可控TCP连接数过多

常见问题与误区

问题1：TCP一定比UDP慢吗？

根因：TCP握手和拥塞控制引入延迟，但连接建立后传输效率相当 结论：长连接场景下差异不大，短连接高频场景UDP优势明显

问题2：UDP加应用层重传等于TCP？

根因：TCP的拥塞控制是全局最优，单个UDP连接的自实现重传缺乏全局视角 结论：应用层可靠UDP（如QUIC）是可行方案，但复杂度远高于直接使用TCP

问题3：TCP的Nagle算法导致延迟

根因：小数据包合并发送，等待ACK才发下一个 解决：TCP_NODELAY 禁用Nagle（实时应用），或 TCP_CORK 批量发送
