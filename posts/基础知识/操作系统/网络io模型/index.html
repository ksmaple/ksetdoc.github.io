网络IO模型

核心结论：网络IO模型从BIO的阻塞线程到NIO的多路复用再到AIO的异步回调，本质是在IO等待、数据拷贝和线程资源之间做权衡，epoll通过事件驱动和边缘触发机制支撑了Linux高并发服务的核心能力。

Linux网络IO从阻塞到非阻塞、从同步到异步演进，epoll事件驱动是当前高并发的核心机制

五种IO模型：从阻塞到异步的演进

以网络read为例，涉及两个阶段：
  阶段1：等待数据就绪（数据从网卡到内核缓冲区）
  阶段2：数据拷贝（从内核缓冲区到用户空间）
模型阶段1（等待数据）阶段2（数据拷贝）特点
阻塞IO（BIO）阻塞等待阻塞拷贝最简单，一个连接一个线程
非阻塞IO（NIO）轮询（立即返回EAGAIN）阻塞拷贝CPU空转浪费
IO多路复用select/poll/epoll阻塞等待多个fd阻塞拷贝单线程管理多个连接
信号驱动IO注册回调，数据就绪时信号通知阻塞拷贝Linux少见
异步IO（AIO）不阻塞不阻塞（内核拷贝完通知）最复杂，真正异步
阻塞IO（BIO）：
  应用            内核
    |  read()    |
    | ---------> |  等待数据就绪 ──────→ 数据到达
    |            |  拷贝数据到用户空间
    |  返回数据   |
    | <--------- |
    |  [全程阻塞] |
    
IO多路复用：
  应用            内核
    | epoll_wait()|  等待多个fd任一就绪
    | ---------> |  ─────────────────→ fd3就绪
    |  fd3可读    |
    | <--------- |
    | read(fd3)  |  拷贝数据（阻塞）
    | ---------> |
    |  返回数据   |
    | <--------- |
    
异步IO（AIO）：
  应用            内核
    | aio_read() |  提交请求
    | ---------> |  内核等待数据+拷贝
    |  立即返回   |
    | <--------- |
    | [继续执行]  |  ─────────────────→ 拷贝完成
    |            |  发送信号/回调通知
    |  处理数据   |

IO多路复用：select、poll、epoll的演进

机制实现方式最大fd数时间复杂度数据拷贝
select位图数组1024（FD_SETSIZE）O(n)扫描每次调用拷贝fd集合到内核
poll链表数组无限制（内存限制）O(n)扫描每次调用拷贝fd数组到内核
epoll红黑树 + 就绪链表无限制O(1)活跃fd首次epoll_ctl拷贝，后续不拷贝

epoll的核心机制：

epoll_create()  → 创建epoll实例（内核中的红黑树根）
epoll_ctl()     → 注册/修改/删除监听的fd和事件（红黑树操作）
epoll_wait()    → 等待事件，返回就绪的fd列表（就绪链表）

内核中的数据结构：
  红黑树：存储所有监听的fd（快速增删查）
  就绪链表：存储已就绪的fd（epoll_wait直接返回）
  
  fd就绪时：
    网卡中断 → 内核处理 → 回调epoll回调函数
    → 将fd加入就绪链表 → epoll_wait唤醒

LT（水平触发）vs ET（边缘触发）：

模式触发条件特点
LT（默认）fd可读/可写时持续触发安全，不遗漏事件
ETfd状态变化时仅触发一次高效，需一次性读完数据
// Java NIO 多路复用示例
Selector selector = Selector.open();
ServerSocketChannel server = ServerSocketChannel.open();
server.configureBlocking(false);
server.bind(new InetSocketAddress(8080));
server.register(selector, SelectionKey.OP_ACCEPT);

while (true) {
    selector.select();  // 阻塞等待就绪事件
    Iterator<SelectionKey> it = selector.selectedKeys().iterator();
    while (it.hasNext()) {
        SelectionKey key = it.next();
        it.remove();
        
        if (key.isAcceptable()) {
            SocketChannel client = server.accept();
            client.configureBlocking(false);
            client.register(selector, SelectionKey.OP_READ);
        }
        if (key.isReadable()) {
            SocketChannel client = (SocketChannel) key.channel();
            ByteBuffer buffer = ByteBuffer.allocate(1024);
            client.read(buffer);
            // 处理数据
        }
    }
}

实现原理与关键机制

从BIO到NIO的线程模型变化

BIO（阻塞IO）：

[连接1] → Thread-1 → read(阻塞) → 处理 → write(阻塞)
[连接2] → Thread-2 → read(阻塞) → 处理 → write(阻塞)
[连接3] → Thread-3 → read(阻塞) → 处理 → write(阻塞)

10000连接 = 10000线程 → 内存耗尽、上下文切换开销巨大

NIO（多路复用）：

[连接1,2,3...N] → Selector线程 → epoll_wait(阻塞)
                        ↓
                   事件分发 → 工作线程池处理
                        ↓
                   连接1可读 → Worker-1处理
                   连接3可写 → Worker-2处理

1个Selector线程管理大量连接，少量工作线程处理业务

// Buffer的状态转换
ByteBuffer buffer = ByteBuffer.allocate(1024);  // 创建，position=0, limit=1024

channel.read(buffer);  // 写入数据，position增加
buffer.flip();         // 切换读模式：limit=position, position=0

while (buffer.hasRemaining()) {
    byte b = buffer.get();  // 读取数据，position增加
}

buffer.clear();  // 清空，position=0, limit=capacity，准备再次写入
// 或 buffer.compact()：保留未读数据，position=未读数据末尾

零拷贝：减少数据在用户态和内核态之间的拷贝

传统文件发送（4次拷贝）：
  磁盘 → 内核Page Cache → 用户缓冲区 → Socket缓冲区 → 网卡

sendfile零拷贝（2次拷贝）：
  磁盘 → 内核Page Cache ──────────────→ 网卡
                ↓
            DMA直接传输（无需CPU参与）
            
Java中的零拷贝：
  FileChannel.transferTo(position, count, SocketChannel)
  
Kafka、Nginx大量使用sendfile提升文件传输性能

实际应用中的决策与权衡

场景推荐模型原因风险
连接数少、逻辑简单BIO编程简单并发能力有限
高并发网络服务NIO + epoll单线程管理大量连接编程复杂，需处理半包粘包
文件传输服务NIO + sendfile零拷贝减少CPU和内存不适用于需要修改数据的场景
超高并发（百万级）AIO/io_uring真正的异步，无阻塞Linux AIO不成熟，io_uring较新
实时性要求高ET模式 + 忙轮询最低延迟CPU占用高

常见问题与误区

问题1：NIO一定比BIO快？

根因：NIO优势在高并发连接复用，单连接场景下BIO更直接 结论：连接数<1000时差异不大，连接数>10000时NIO优势明显

问题2：epoll_wait返回后必须遍历所有fd？

根因：epoll_wait只返回就绪的fd列表（通过就绪链表），无需遍历全部 对比：select/poll需要遍历全部fd检查状态

问题3：Java NIO的Selector空轮询导致CPU 100%

根因：JDK在Linux的epoll实现存在空轮询bug（JDK-6427854） 解决：Netty等框架通过重建Selector绕过此问题
