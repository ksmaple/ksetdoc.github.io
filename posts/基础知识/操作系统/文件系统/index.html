文件系统与目录结构

核心结论：文件系统通过inode管理文件元数据，通过目录项建立文件名到inode的映射，Linux的目录结构遵循FHS标准，理解inode、文件描述符和VFS抽象层是排查磁盘和文件相关问题的关键。

Linux文件系统通过inode管理元数据，VFS提供统一抽象，目录结构遵循FHS标准

inode：文件的元数据结构

文件名只是目录项的别名，真正的文件身份是inode号

文件名 ──→ 目录项（dentry）──→ inode ──→ 数据块
                          
inode内容（元数据）：
  - 文件类型（普通文件/目录/链接/设备）
  - 权限（rwxrwxrwx）
  - 所有者UID、所属组GID
  - 文件大小
  - 时间戳（atime/mtime/ctime）
  - 数据块指针（直接/间接/双重间接/三重间接）
  - 链接计数
# 查看文件inode号
ls -i filename
# 输出: 131074 filename

# 查看inode详细信息
stat filename
# File: filename
# Size: 1024       Blocks: 8          IO Block: 4096   regular file
# Device: 801h/2049d Inode: 131074     Links: 1
# Access: (0644/-rw-r--r--)  Uid: ( 1000/   user)   Gid: ( 1000/   user)
# Access: 2026-04-30 10:00:00.000000000 +0800
# Modify: 2026-04-30 09:00:00.000000000 +0800
# Change: 2026-04-30 09:00:00.000000000 +0800

硬链接 vs 软链接：

特性硬链接软链接（符号链接）
本质同一inode的多个目录项独立的inode，存储目标路径
跨文件系统否是
链接目标删除仍可访问（引用计数）失效（ dangling link）
链接目录通常不允许允许
inode号相同不同
# 硬链接
ln original.txt hardlink.txt
# original.txt 和 hardlink.txt 共享同一个inode

# 软链接
ln -s original.txt softlink.txt
# softlink.txt 是一个新文件，内容是 "original.txt"

Linux目录结构（FHS标准）

/                          ← 根目录
├── bin/                   ← 基本命令（ls, cp, mv）
├── sbin/                  ← 系统管理命令（fdisk, ifconfig）
├── boot/                  ← 引导加载器文件
├── dev/                   ← 设备文件
│   ├── null               ← 空设备（丢弃写入）
│   ├── zero               ← 零源（读取全0）
│   ├── random             ← 真随机数
│   ├── urandom            ← 伪随机数（不阻塞）
│   └── sda1               ← 磁盘分区
├── etc/                   ← 系统配置文件
│   ├── passwd             ← 用户账户信息
│   ├── hosts              ← 静态DNS映射
│   └── fstab              ← 文件系统挂载表
├── home/                  ← 用户主目录
│   └── user/
├── lib/                   ← 共享库
├── media/                 ← 可移动媒体挂载点
├── mnt/                   ← 临时挂载点
├── opt/                   ← 可选应用软件
├── proc/                  ← 虚拟文件系统（进程信息）
│   ├── 1/                 ← PID为1的进程信息
│   ├── cpuinfo            ← CPU信息
│   ├── meminfo            ← 内存信息
│   └── sys/               ← 内核参数
├── root/                  ← root用户主目录
├── run/                   ← 运行时变量数据
├── srv/                   ← 服务数据
├── sys/                   ← sysfs虚拟文件系统
├── tmp/                   ← 临时文件（重启清除）
├── usr/                   ← 用户程序
│   ├── bin/               ← 非基本命令
│   ├── local/             ← 本地安装软件
│   └── share/             ← 共享数据
└── var/                   ← 可变数据
    ├── log/               ← 日志文件
    ├── spool/             ← 队列数据
    └── tmp/               ← 持久临时文件

文件描述符：进程访问文件的抽象句柄

进程打开文件时，内核分配一个文件描述符（非负整数）：

进程文件描述符表：
  fd 0 → 标准输入（stdin）
  fd 1 → 标准输出（stdout）
  fd 2 → 标准错误（stderr）
  fd 3 → /var/log/app.log（打开的文件）
  fd 4 → socket:192.168.1.10:8080（网络连接）
  
每个fd指向内核中的文件表项，包含：
  - 文件状态标志（读写模式、追加、非阻塞等）
  - 当前文件偏移量
  - 指向inode的指针

多个fd可指向同一个文件表项（dup），多个文件表项可指向同一个inode（不同进程打开同一文件）
# 查看进程打开的文件描述符
ls -l /proc/<pid>/fd/

# 查看系统文件描述符限制
ulimit -n           # 单个进程限制
cat /proc/sys/fs/file-max   # 系统全局限制

# 查看某进程使用的fd数量
ls /proc/<pid>/fd/ | wc -l

实现原理与关键机制

ext4文件系统：延迟分配、 extents、日志

特性说明优势
Extents用起始块+长度表示连续空间替代传统块指针，减少碎片
延迟分配写入时不立即分配块提高连续分配概率
日志（Journal）记录元数据变更崩溃后快速恢复
多块分配器预分配多个块减少文件碎片
传统ext2/ext3块指针：
  inode → 12个直接块指针
        → 1个间接块指针（指向块指针数组）
        → 1个双重间接块指针
        → 1个三重间接块指针
        
  最大文件: (12 + 256 + 256² + 256³) * 4KB ≈ 4TB

ext4 extents：
  inode → extent树节点
        → [起始块:100, 长度:50]  表示块100~149连续
        → [起始块:500, 长度:30]  表示块500~529连续
        
  大文件存储更高效，碎片更少

VFS虚拟文件系统：统一不同文件系统的接口

用户空间
    ↓ 系统调用（open/read/write）
VFS层（虚拟文件系统）
    ├── 通用文件模型（superblock, inode, dentry, file）
    ↓
具体文件系统驱动
    ├── ext4
    ├── xfs
    ├── btrfs
    ├── proc（虚拟）
    └── nfs（网络）
    ↓
块设备层 → 磁盘驱动 → 物理存储

文件I/O方式：缓冲IO vs 直接IO vs 内存映射

方式机制优点缺点
缓冲IO（Buffered）数据先拷贝到Page Cache减少磁盘IO、预读缓存数据拷贝开销
直接IO（Direct）绕过Page Cache，直接读写磁盘减少内存拷贝、应用自管理缓存要求对齐、无预读
内存映射（mmap）文件映射到进程地址空间零拷贝、共享内存、随机访问文件大小变化处理复杂
// Java NIO 内存映射文件
RandomAccessFile raf = new RandomAccessFile("largefile.dat", "r");
FileChannel channel = raf.getChannel();
MappedByteBuffer buffer = channel.map(MapMode.READ_ONLY, 0, channel.size());

// buffer直接映射到文件，无需read/write系统调用
while (buffer.hasRemaining()) {
    byte b = buffer.get();
}

实际应用中的决策与权衡

场景推荐方案原因
小文件频繁读写缓冲IOPage Cache减少磁盘访问
数据库文件直接IO + 应用自管理缓存避免双缓存问题
大文件随机访问mmap零拷贝，内核自动管理页面
日志追加写入缓冲IO + 定期sync合并小写入
临时文件tmpfs（/dev/shm或/tmp）内存存储，无需磁盘IO

常见问题与误区

问题1：删除文件后磁盘空间未释放

根因：文件被进程占用（文件描述符未关闭），inode引用计数不为0 排查：lsof +L1 查看已删除但仍被占用的文件 解决：重启占用进程或清空文件（> /proc/<pid>/fd/<fd_num>）

问题2：Too many open files

根因：进程打开的文件描述符超过ulimit限制，或连接未关闭导致泄漏 排查：lsof -p <pid> 查看进程打开的文件 解决：增大ulimit -n，排查并关闭泄漏的连接/文件

问题3：df和du显示的磁盘使用量不一致

根因：du统计目录大小，df统计文件系统使用；已删除但仍被占用的文件df计数但du不统计 解决：使用lsof +L1查找并处理被删除但未释放的文件
