JVM原理

核心结论：JVM通过类加载机制将字节码转换为运行时结构，在堆/栈/方法区组成的内存模型中执行，GC自动管理堆内存生命周期，理解这三者的交互是JVM调优的基础。

JVM由类加载子系统、运行时数据区、执行引擎三部分组成，共同完成字节码到机器码的执行

类加载机制：双亲委派模型保证核心类库安全

类加载过程分七个阶段，其中加载、验证、准备、解析、初始化五个阶段是核心：

加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载

双亲委派模型：

自定义类加载器
    ↑ 委派
应用类加载器 (AppClassLoader)    - classpath
    ↑ 委派
扩展类加载器 (ExtClassLoader)    - ext目录
    ↑ 委派
启动类加载器 (Bootstrap)         - <JAVA_HOME>/lib
// 打破双亲委派：自定义类加载器直接加载
public class CustomClassLoader extends ClassLoader {
    @Override
    protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
        // 先自己尝试加载，找不到再委派给父类
        Class<?> clazz = findLoadedClass(name);
        if (clazz == null) {
            clazz = findClass(name);  // 自定义加载逻辑
            if (clazz == null) {
                clazz = super.loadClass(name, resolve);  // 委派父类
            }
        }
        return clazz;
    }
}

运行时数据区：堆存对象、栈存帧、方法区存元数据

区域线程共享存储内容溢出表现
堆（Heap）是对象实例、数组OutOfMemoryError: Java heap space
虚拟机栈否栈帧（局部变量、操作数栈）StackOverflowError
本地方法栈否Native方法执行同虚拟机栈
程序计数器否当前执行字节码地址不会溢出
方法区（Metaspace）是类元数据、常量池OutOfMemoryError: Metaspace
直接内存是NIO的BufferOutOfMemoryError: Direct buffer memory
// 栈帧结构示意
public void methodA() {
    int x = 10;              // 局部变量表
    methodB();               // 操作数栈压入返回地址
}

public void methodB() {
    Object obj = new Object();  // obj引用在栈，对象在堆
}

GC垃圾回收：分代收集基于"弱分代假设"

弱分代假设：大部分对象朝生夕死，熬过越多次GC的对象越可能长期存活。

收集器算法适用场景特点
Serial复制/标记整理单核小内存STW单线程
Parallel复制/标记整理多核后台计算吞吐优先
CMS标记清除低延迟应用并发回收，内存碎片
G1标记复制（Region）大堆均衡可预测停顿时间
ZGC/Shenandoah着色指针/读屏障极低延迟亚毫秒级停顿
// G1 GC 关键参数
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200          // 目标最大停顿200ms
-XX:G1HeapRegionSize=16m          // Region大小
-XX:InitiatingHeapOccupancyPercent=45  // 堆占用45%触发并发标记

G1回收过程：

  1. 初始标记（STW，极短）
  2. 并发标记（与用户线程并行）
  3. 最终标记（STW）
  4. 筛选回收（STW，复制存活对象到空Region）

实现原理与关键机制

对象内存布局：对象头 + 实例数据 + 对齐填充

| 对象头 (Header) | 实例数据 (Instance Data) | 对齐填充 (Padding) |
|    12 bytes     |        按字段计算         |   8字节对齐补齐    |

对象头组成（64位JVM，压缩指针）：

  • Mark Word（8 bytes）：哈希码、GC分代年龄、锁状态标志
  • Class Pointer（4 bytes）：指向方法区类元数据
  • 数组长度（4 bytes，仅数组对象）

对象的创建与访问

Object obj = new Object();

创建过程：

  1. 检查类是否已加载（未加载则触发类加载）
  2. 分配内存（指针碰撞或空闲列表，取决于GC是否带整理）
  3. 零值初始化（保证字段无需赋值即可使用）
  4. 设置对象头（哈希码、GC年龄等）
  5. 执行<init>构造方法

对象访问方式：

  • 句柄访问：栈引用 → 句柄池（对象实例指针 + 类型数据指针）
  • 直接指针（HotSpot采用）：栈引用 → 对象头中的类型指针

volatile 的内存语义实现

volatile int flag = 0;

JVM通过内存屏障实现 volatile：

  • 写操作前插入 StoreStore屏障
  • 写操作后插入 StoreLoad屏障
  • 读操作后插入 LoadLoad屏障 + LoadStore屏障
CPU层面：lock addl $0x0, (%rsp)  // 空操作，触发缓存行刷新

实际应用中的决策与权衡

场景推荐GC/配置原因风险
小内存（<4G）、单核Serial简单高效停顿时间长
后台批处理、吞吐优先Parallel最大化吞吐量停顿不可控
大堆（>8G）、均衡需求G1可预测停顿、自动Region管理小对象过多时效率下降
金融交易、极低延迟ZGC（JDK15+）亚毫秒停顿、可扩展至TB级堆总吞吐量略低于G1
频繁动态生成类增大Metaspace避免类元数据溢出物理内存占用增加

常见问题与误区

问题1：线上OOM但堆内存设置很大

根因：非堆内存溢出（Metaspace、直接内存、线程过多导致栈空间耗尽） 排查：jmap -heap查看堆实际使用，jcmd <pid> VM.native_memory summary查看Native内存

问题2：Full GC频繁但老年代对象并不多

根因：System.gc()显式调用、元空间不足、晋升阈值设置不当 解决：添加-XX:+DisableExplicitGC，调大Metaspace，调整-XX:MaxTenuringThreshold

问题3：JNI调用导致崩溃

根因：本地代码访问已释放的Java对象，或违反JNI规范 解决：使用NewGlobalRef保持对象存活，检查GetStringUTFChars后必须ReleaseStringUTFChars
