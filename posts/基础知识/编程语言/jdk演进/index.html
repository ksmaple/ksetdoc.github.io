JDK版本演进

核心结论：JDK从8到21的演进主线是"语法糖简化开发 + JVM性能提升 + 并发模型现代化"，其中8的Stream/Lambda、11的ZGC、17的密封类和21的虚拟线程是影响架构决策的关键版本。

JDK 8（2014）：函数式编程与Stream革命

特性说明影响
Lambda表达式函数式接口的简洁语法替代大量匿名内部类
Stream API声明式集合处理内部迭代、惰性求值、并行支持
方法引用Class::method语法进一步简化Lambda
接口默认方法default关键字接口可扩展不破坏实现类
新日期时间APIjava.time包线程安全、设计清晰
PermGen移除方法区移至Metaspace（本地内存）减少OOM，但内存不自动回收

// After

// 密封类：明确控制哪些类可以继承
public abstract sealed class Shape
    permits Circle, Rectangle, Square { }

public final class Circle extends Shape { }
public final class Rectangle extends Shape { }
public non-sealed class Square extends Shape { }

// switch 模式匹配（JDK17预览，21正式）
String desc = switch (shape) {
    case Circle c    -> "圆，半径" + c.radius();
    case Rectangle r -> "矩形" + r.width() + "x" + r.height();
    case Square s    -> "正方形" + s.side();
};

特性说明影响

// 虚拟线程：创建成本极低，无需池化
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    IntStream.range(0, 1_000_000).forEach(i ->
        executor.submit(() -> {
            Thread.sleep(Duration.ofSeconds(1));
            return i;
        })
    );
}  // 100万个并发任务，1个OS线程可处理数千虚拟线程

实现原理与关键机制

虚拟线程 vs 平台线程

维度平台线程（OS线程）虚拟线程
创建成本~1MB栈空间~几百字节
调度器OS内核JVM用户态调度器（ForkJoinPool）
阻塞代价OS线程阻塞虚拟线程卸载（unmount），OS线程复用
适用场景CPU密集型、持有锁IO密集型、大量并发
平台线程：Java Thread → JVM Thread → OS Thread（1:1:1）
虚拟线程：   Java VThread → Carrier Thread（M:N映射）
                      ↓
              阻塞时VThread挂载到堆栈，Carrier释放

虚拟线程的陷阱：

  • synchronized 块或 ReentrantLock 持有期间阻塞会钉住（pin）Carrier线程
  • 解决方案：使用 java.util.concurrent.locks.Lock（虚拟线程感知）替代 synchronized

分代ZGC：从Region到分代的优化

JDK21之前ZGC不分代，每次GC都扫描全堆。分代ZGC利用弱分代假设：

  • 年轻对象集中在特定Region
  • 频繁回收年轻Region，减少全堆扫描

实际应用中的决策与权衡

场景推荐版本原因风险
存量系统维护JDK8生态成熟，第三方兼容最好2024年Oracle停止公共更新
新服务开发JDK17/21LTS长期支持，现代语法部分老旧框架需升级
高并发IO密集型JDK21 + 虚拟线程百万级并发，代码简洁synchronized需重构为Lock
大内存低延迟JDK21 + 分代ZGC亚毫秒停顿总吞吐略低于G1

常见问题与误区

问题1：JDK8升级后反射报错（Illegal reflective access）

根因：JDK9+模块化系统强封装内部API 解决：添加JVM参数 --add-opens 或升级依赖库到兼容版本

问题2：虚拟线程性能不如预期

根因：代码中存在大量 synchronized 或 ThreadLocal 滥用 解决：将 synchronized 改为 ReentrantLock，减少 ThreadLocal 使用

问题3：JDK11后String底层从char[]改为byte[]

根因：节省内存（Latin-1字符仅占1字节） 影响：String.getBytes() 行为不变，但 StringCoding 内部实现变化，某些反射操作可能失效
