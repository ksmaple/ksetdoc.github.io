并发与锁问题汇总

本文档汇总了项目历史上处理的并发与锁相关技术问题，包括分布式锁、线程安全、阻塞修复、异步处理等。

1. Redisson MarshallingCodec ThreadLocal 缓存导致的内存泄漏

提交记录: kella-server 399e3d18 (2026-04-22) 涉及文件:

  • kella-user/src/main/java/me/ddkj/user/biz/impl/SubscribeBizImpl.java
  • kella-user/src/main/java/me/ddkj/user/biz/impl/SuperCardBizImpl.java
  • kella-user/src/main/java/me/ddkj/user/biz/impl/UserLVipBizImpl.java
  • kella-user/src/main/java/me/ddkj/user/service/label/dao/impl/UserLabelDaoImpl.java
  • kella-user/src/main/java/me/ddkj/user/service/operator/impl/UserOperatorDaoImpl.java

问题背景

Redisson 默认使用 MarshallingCodec 进行序列化，该底层依赖 JBoss Marshalling 的 RiverUnmarshaller。在高并发场景下，RiverUnmarshaller 内部使用 ThreadLocal 缓存实例，因线程池长期存活导致缓存无法释放，最终引发内存泄漏（OOM）。

解决方案

  1. 替换序列化方式：将所有 Redisson 缓存操作从默认 MarshallingCodec 改为 StringCodec（字符串）或 IntegerCodec（整型），彻底消除 JBoss Marshalling 的 ThreadLocal 缓存问题。
  2. Key 版本隔离：所有缓存 key 增加 _v2 后缀，与旧数据隔离，防止新老数据混用。
  3. 双读兼容 + 数据迁移：对于 SubscribeBizImpl 的过期队列，采用新 key 优先读取，新 key 不存在时回退读取旧 key，并将旧数据迁移到新 key 的方案，保证线上服务平滑升级。

关键代码

// 使用 IntegerCodec 替代默认 MarshallingCodec
private static final IntegerCodec INTEGER_CODEC = new IntegerCodec();

public Map<Integer, Integer> map(List<Integer> uids) {
    String cacheKey = "bo_user_subscribe_lv_map_v2";  // key 加 _v2 后缀
    final RMap<Integer, Integer> map = redisClientSession.get().getMap(cacheKey, INTEGER_CODEC);
    return map.getAll(new HashSet<>(uids));
}
// 双读兼容 + 数据迁移
public List<Integer> rangeExpireLv() {
    int now = (int) (System.currentTimeMillis() / 1000L);
    String newKey = "bo_user_subscribe_lv_expire_v2";
    final RScoredSortedSet<Integer> newSet = redisClientSession.get().getScoredSortedSet(newKey, INTEGER_CODEC);

    if (newSet.isExists()) {
        Collection<Integer> result = newSet.valueRange(0, true, now, true);
        return new ArrayList<>(result);
    }

    // 新 key 不存在，回退读旧 key
    String oldKey = "bo_user_subscribe_lv_expire";
    final RScoredSortedSet<Integer> oldSet = redisClientSession.get().getScoredSortedSet(oldKey);
    Collection<Integer> result = oldSet.valueRange(0, true, now, true);

    // 迁移旧数据到新 key
    for (Integer uid : result) {
        Double score = oldSet.getScore(uid);
        if (score != null) {
            newSet.add(score, uid);
        }
    }
    return new ArrayList<>(result);
}

2. 幸运蛋库存扣减并发问题（Set去重 + 死循环修复）

提交记录: kella-server 40f4bcb6 (2026-03-16) 涉及文件:

  • kella-wealth/src/main/java/me/ddkj/wealth/service/luckyegg/LuckyEggResultEntity.java

问题背景

幸运蛋抽奖活动中，当库存不足时需要从概率池中隐藏已耗尽库存的奖励项。原实现使用 List 存储无库存项，在并发或多轮抽奖时会导致同一 ID 重复累积，且 while(true) 死循环条件判断存在缺陷（newHiddeList.size() >= curProbability.size()），可能导致无限循环或提前退出。

解决方案

  1. List 改为 Set：使用 HashSet 替代 ArrayList 存储无库存项，天然去重。
  2. 修复死循环条件：将 hiddenSet.size() >= curProbability.size() 改为 hiddenSet.size() >= validCount（validCount 为实际有效的概率项数量）。
  3. 状态标记退出：引入 stockOk 标记，当库存全部耗尽时跳过本次抽奖，而非继续死循环。

关键代码

// 存储已经没有库存的id（用Set去重，防止跨迭代重复累积）
Map<Integer, Set<String>> mapHiddenSet = new HashMap<>();

for (int i = 0; i < count; i++) {
    int poolType = isBigProb(probType) ? ... : ...;
    Set<String> hiddenSet = mapHiddenSet.computeIfAbsent(poolType, k -> new HashSet<>());
    
    if (!hiddenSet.isEmpty()) {
        randomBox.setHiddenList(new ArrayList<>(hiddenSet));
    }
    
    boolean stockOk = true;
    while (true) {
        if (stockService.decr(curConfig.getId(), curConfig, allConfigs)) {
            break;
        }
        hiddenSet.add(String.valueOf(id));
        long validCount = curProbability.values().stream().filter(v -> v != null && v > 0).count();
        if (hiddenSet.size() >= validCount) {
            stockOk = false;
            break;
        }
        id = Integer.valueOf(randomBox.random());
        curConfig = configEntityMap.get(id);
    }
    
    if (!stockOk) {
        continue;  // 库存全部耗尽，跳过本次
    }
}

3. 聚合服务循环调用异常穿透问题

提交记录: kella-server f3f2fb89 (2025-10-20) 涉及文件:

  • kella-ugc/src/main/java/me/ddkj/ugc/rpc/AggServiceImpl.java

问题背景

聚合服务（AggService）在处理用户上线事件时，需要循环调用多个活动的 touch 方法和多个内部事件处理器的 handleUserOnlineEvent 方法。原实现中，如果某一个活动或处理器抛出异常，会导致后续的 activity/handler 都无法执行，造成部分业务逻辑遗漏。

解决方案

在循环内部对每个调用增加 try-catch 包裹，捕获异常并记录 warn 日志，确保单个异常不影响后续调用。

关键代码

for (CampaignAggDO rank : ranks) {
    try {
        aggFactory.select(rank.getActivityId()).touch(rank, uid);
    } catch (Exception e) {
        log.warn(e.getMessage());
    }
}

for (ActivityAggInnerEventHandle handle : handleList) {
    try {
        handle.handleUserOnlineEvent(uid, language);
    } catch (Exception e) {
        log.warn(e.getMessage());
    }
}

4. 单例双检锁模式优化（DCL）

提交记录: kella-spring-boot-starter a1ae1e82 (2020-04-14) 涉及文件:

  • src/main/java/verify/ddkj/config/RedisClientRoom.java
  • src/main/java/verify/ddkj/config/RedisClientSession.java
  • src/main/java/verify/ddkj/config/RedisClientUgc.java

问题背景

Redis 客户端初始化为单例模式，原实现使用 synchronized 方法但缺少双重检查锁（DCL）机制。在高并发场景下，多个线程可能同时进入同步块，导致 JedisCluster 被多次初始化，浪费资源且可能引发连接异常。

解决方案

  1. 增加 volatile 关键字：确保 JedisCluster 实例的可见性和禁止指令重排序。
  2. 双重检查锁：在同步块内部再次检查实例是否为 null，避免重复初始化。

关键代码

private static volatile JedisCluster jedisClusterRoom;

@Override
protected JedisCluster getJedis() {
    if (jedisClusterRoom == null) {
        jedisClusterRoom = jedisClusterRoom();
    }
    return jedisClusterRoom;
}

private synchronized JedisCluster jedisClusterRoom() {
    if (jedisClusterRoom != null) {
        return jedisClusterRoom;  // 二次检查，防止重复初始化
    }
    try {
        JedisPoolConfig config = new JedisPoolConfig();
        // ... 初始化配置
        jedisClusterRoom = new JedisCluster(...);
    } catch (Exception e) {
        log.error("init redis error! ", e);
    }
    return jedisClusterRoom;
}

5. Dubbo 注册中心缓存文件配置

提交记录: kella-cms 9b0db864 (2025-02-12) 涉及文件:

  • cms-starter/src/main/resources/application-starter-dev.yml
  • cms-starter/src/main/resources/application-starter-prod.yml
  • cms-starter/src/main/resources/application-starter-test.yml

问题背景

Dubbo 默认将注册中心缓存文件写入系统临时目录（如 /tmp），在容器或服务器重启后临时目录被清理，导致 Dubbo 消费者无法快速恢复服务引用，可能出现服务调用失败或超时。

解决方案

显式配置 dubbo.registry.file 指向持久化目录（/data/applogs/dubbo.cache/${spring.dubbo.application.name}.cache），确保缓存文件在重启后仍然可用。

关键配置

spring:
  dubbo:
    registry:
      address: zookeeper://...
      file: /data/applogs/dubbo.cache/${spring.dubbo.application.name}.cache

6. 充值订单状态变更分布式锁

提交记录: kella-server 29f08ce8 (2026-02-25) 涉及文件:

  • kella-wealth/src/main/java/me/ddkj/wealth/service/order/AbstractOrderFlow.java

问题背景

充值订单回调处理中，异常失败的订单可以被成功订单覆盖并重复发放金币。根本原因是并发场景下订单状态变更没有加锁，多个回调请求同时到达时可能产生竞态条件。

解决方案

  1. 增加参数校验：订单号不能为空。
  2. 增加空值检查：数据库查询不到订单时直接返回。
  3. 精确状态判断：只有订单状态不是 FINISH_SUCCESS、PAY_SEND、REFUND 时才允许修改，防止已完成的订单被覆盖。
  4. 分布式锁：使用 Redis 分布式锁（RLock）保证同一订单的回调串行处理。

关键代码

public void callbackSuccessOrder(String orderNo) {
    MessageEnums.ERR_INVALID_PARAM.assertIsTrue(StringUtils.isNotEmpty(orderNo));
    String lockKey = StringUtils.joinWith(":", BO_ORDER_STATE_CHANGE, orderNo);
    RLock lock = redisClientUgc.get().getLock(lockKey);
    boolean isLock = false;
    try {
        isLock = lock.tryLock(3, TimeUnit.SECONDS);
        if (!isLock) {
            MessageEnums.ERR_PAY_TO_FAST.assertIsTrue(false);
        }
        BeanOrderEntity beanOrderEntity = beanOrderMapper.getBeanOrderByOrderNoV1(orderNo);
        if (beanOrderEntity == null) {
            log.warn("[orderNo={}] callback order not found", orderNo);
            return;
        }
        // 精确状态判断：只有未完成状态才允许修改
        if (!beanOrderEntity.getState().equals(PayEnums.PayState.FINISH_SUCCESS.getState()) &&
            !beanOrderEntity.getState().equals(PayEnums.PayState.PAY_SEND.getState()) &&
            !beanOrderEntity.getState().equals(PayEnums.PayState.REFUND.getState())) {
            beanOrderEntity.orderPayed();
        }
    } catch (Exception e) {
        log.error("[orderNo={}] callbackSuccessOrder error", orderNo, e);
    } finally {
        if (isLock) {
            lock.unlock();
        }
    }
}

7. 活动初始化异步化

提交记录: kella-cms 5b326eaa / f397991b (2024-12-13) 涉及文件:

  • cms-starter 相关启动类

问题背景

CMS 启动时需要初始化大量活动配置数据，同步初始化导致应用启动时间过长，影响服务可用性。

解决方案

将启动时的数据初始化任务改为异步执行，减少启动阻塞时间。

8. 房间奖励锁问题

提交记录: kella-cms 9b0db864 (2025-02-12) 涉及文件:

  • cms-starter/src/main/resources/application-starter-*.yml

问题背景

CMS 房间奖励发放时，Dubbo 注册中心缓存丢失导致服务引用异常，影响奖励发放的一致性。

解决方案

配置 Dubbo 注册中心缓存文件到持久化目录，确保服务引用在重启后快速恢复。
