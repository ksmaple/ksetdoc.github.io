数据库与连接池优化汇总

本文档汇总了项目历史上针对数据库查询性能、连接池配置、SQL 优化等方面的技术改进。

1. MySQL 连接池最大等待时间优化

提交记录: kella-spring-boot-starter ce79ec6f (2020-02-21) 涉及文件:

  • src/main/java/verify/ddkj/config/RedisClientSession.java
  • src/main/java/verify/ddkj/config/RedisClientUgc.java
  • src/main/resources/application-starter-prd.yml

问题背景

Redis 连接池配置中，maxWaitMillis 被错误地设置为 timeout（连接超时时间，通常 2000ms），而非独立的 maxWait（最大等待时间）。当连接池耗尽时，等待线程只等待极短时间（100ms）就抛出 JedisConnectionException，导致大量请求失败。同时，生产环境的 max-wait 配置为 5000ms，在高并发下连接池耗尽时，线程长时间阻塞等待，反而拖垮服务。

解决方案

  1. 修复变量引用：将 config.setMaxWaitMillis(timeout) 改为 config.setMaxWaitMillis(maxWait)，使用正确的配置项；
  2. 降低 max-wait：生产环境 Redis 连接池的 max-wait 从 5000ms 降低到 100ms，快速失败，避免线程长时间阻塞；
  3. 增加连接池大小：同步增加 max-active 到 500~1000，用更多连接承载并发，而非让线程等待。

关键代码

// 修复前：错误地使用 timeout 作为 maxWait
config.setMaxWaitMillis(timeout);

// 修复后：使用正确的 maxWait 配置
config.setMaxWaitMillis(maxWait);
# 修复前：max-wait 过长，线程阻塞
redis:
  session:
    pool:
      max-active: 500
      max-idle: 20
      max-wait: 5000

# 修复后：降低 max-wait，快速失败
redis:
  session:
    pool:
      max-active: 500
      max-idle: 20
      max-wait: 100

收益

  • 连接池耗尽时快速失败，避免线程堆积拖垮服务；
  • 配合增加 max-active，用更多连接承载并发；
  • 服务整体可用性提升。

2. 移除旧 TR 配置 + MySQL 连接加重试

提交记录: kella-server 50757597 / kella-cms 50757597 (2025-03-20) 涉及文件:

  • kella-spring-boot-starter 相关配置文件

问题背景

MySQL 连接在瞬断（网络抖动、MySQL 重启）后，应用无法自动恢复连接，导致持续的数据库操作失败。需要手动重启应用才能恢复。

解决方案

  1. 移除旧的 TR（事务路由）配置；
  2. 在数据源配置中增加连接重试机制，包括：
    • autoReconnect=true
    • failOverReadOnly=false
    • maxReconnects=10
    • initialTimeout=2

关键配置

spring.datasource.url=jdbc:mysql://host:port/db?
    autoReconnect=true&
    failOverReadOnly=false&
    maxReconnects=10&
    initialTimeout=2

3. 连接池关闭优化

提交记录: kella-spring-boot-starter 0a89f464 (2020-02-21) 涉及文件:

  • src/main/java/verify/ddkj/config/RedisClient*.java

问题背景

应用关闭时，Redis 连接池（JedisCluster）未正确关闭，导致连接泄漏，在频繁重启的环境中可能耗尽 Redis 服务器连接数。

解决方案

在应用关闭钩子（Shutdown Hook）或 Spring 的 @PreDestroy 中显式关闭 JedisCluster 连接池。

关键代码

@PreDestroy
public void destroy() {
    if (jedisCluster != null) {
        jedisCluster.close();
    }
}

4. CMS 房间数据统计排序优化

提交记录: kella-cms eeba75d0 / fd9e2845 / f2f435b8 / fe537412 / 2bd35414 (2024-09-26) 涉及文件:

  • cms-main-web/src/main/java/com/ddkj/cms/controller/operation/bobo/ChatRoomController.java
  • cms-system/src/main/java/com/ddkj/cms/dao/ChatroomDao.java

问题背景

CMS 房间数据统计查询在数据量大时排序慢，可能是缺少索引或排序字段选择不当导致全表扫描 + filesort。

解决方案

  1. 优化查询 SQL，增加合适的索引；
  2. 调整排序字段，使用覆盖索引避免回表；
  3. 对大数据量查询增加分页限制。

5. 主播数据 SQL 优化

提交记录: kella-spring-boot-starter 6cffcc2d (2023-02-14) 涉及文件:

  • starter 相关统计代码

问题背景

主播数据统计 SQL 查询慢，可能是关联表过多、未使用索引、或查询了大量无用字段。

解决方案

优化主播数据统计 SQL，具体措施可能包括：

  • 增加复合索引；
  • 减少关联表数量；
  • 只查询需要的字段；
  • 使用汇总表替代实时计算。

6. CAT SQL 监控初始化

提交记录: kella-spring-boot-starter 685f33b9 (2019-09-05) 涉及文件:

  • starter 相关监控配置

问题背景

CAT（Central Application Tracking）监控未对 SQL 进行监控，导致数据库性能问题无法及时发现。

解决方案

增加 CAT SQL 监控配置，对慢 SQL 进行自动采集和告警。

关键配置

<!-- CAT SQL 监控配置 -->
<module id="cat-sql-monitor">
    <slow-sql-threshold>100</slow-sql-threshold>  <!-- 慢SQL阈值100ms -->
</module>

7. CMS 批量添加主播导入限制

提交记录: kella-cms f29e857f (2024-07-29) 涉及文件:

  • cms-main-web/src/main/java/com/ddkj/cms/controller/operation/bobo/AnchorController.java

问题背景

CMS 批量导入主播时未限制单次导入数量，用户可能一次性导入数千条记录，导致：

  1. 数据库批量插入耗时极长；
  2. 事务持有时间过长；
  3. 接口超时；
  4. 内存溢出。

解决方案

增加单次导入数量限制（500条），超过限制时拒绝处理。

关键代码

list = list.stream()
           .filter(e -> e.getUid() != null && e.getUid() != 0)
           .collect(Collectors.toList());

if (list.size() > 500) {
    return new BaseResponse(-1, "导入数量过多 单次操作不超过500");
}

8. 审核版本分页优化

提交记录: kella-server f56dc455 / kella-cms 6f613154 (2025-12-23) 涉及文件:

  • kella-wealth 相关审核服务
  • cms-main-web 相关控制器

问题背景

审核版本列表查询未分页或分页参数处理不当，导致数据量大时一次性返回大量数据，内存占用高且接口响应慢。

解决方案

  1. 增加分页参数校验；
  2. 修复分页逻辑，确保 LIMIT 和 OFFSET 正确；
  3. 修复账户引用 null 导致的空指针异常。

9. 查询超时优化（房间数据统计）

提交记录: kella-cms 331c1514 (2024-08-07) 涉及文件:

  • cms-system/src/main/java/com/ddkj/cms/dao/ChatroomDao.java
  • cms-system/src/main/resources/mapping/ChatroomMapper.xml

问题背景

房间数据统计查询超时，原因是：

  1. 查询条件中使用 roomId = #{roomId}，当需要查询多个房间时无法使用索引；
  2. 字段填充逻辑在 Java 代码中处理，增加不必要的数据库往返。

解决方案

  1. 将 roomId 改为 roomIds（逗号分隔字符串），Mapper 中使用 roomId in (${roomIds})；
  2. 统一参数传递方式，减少 DAO 层逻辑复杂度。

关键代码

<!-- 修复前 -->
<if test="roomId!=null and roomId!=''">
    and r.roomId =#{roomId}
</if>

<!-- 修复后 -->
<if test="roomIds!=null and roomIds!=''">
    and r.roomId in (${roomIds})
</if>

10. 送礼日志联合筛选优化

提交记录: kella-cms c0113ea6 / 70211181 / 059eda3c (2024-10-09) 涉及文件:

  • cms-main-web 相关日志查询控制器

问题背景

CMS 送礼日志查询只支持单字段筛选（如只按 level 或只按 roomId），无法联合筛选，导致运营人员需要多次查询才能定位问题。

解决方案

增加 level 和 roomId 的联合筛选功能，支持多条件组合查询。

收益

  • 运营效率提升；
  • 减少多次查询对数据库的压力。

11. Web Filter 优化

提交记录: kella-spring-boot-starter 14aa1ee4 (2021-04-25) 涉及文件:

  • starter 相关 WebFilter 代码

问题背景

Web Filter 拦截逻辑存在性能瓶颈，可能是每次请求都执行了不必要的操作（如重复解析请求体、多次查询缓存）。

解决方案

优化 Web Filter 逻辑，减少不必要的处理，可能包括：

  • 缓存解析结果；
  • 跳过静态资源；
  • 异步处理日志记录。
