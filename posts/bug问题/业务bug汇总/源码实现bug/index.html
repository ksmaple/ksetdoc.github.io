源码实现 Bug 汇总

本文档汇总了项目历史上因代码逻辑缺陷、边界条件未处理、状态判断错误等导致的 Bug 修复记录。

1. 周星活动空指针异常（NPE）

提交记录: kella-server c7aa065d (2025-11-05) 涉及文件:

  • kella-ugc/src/main/java/me/ddkj/ugc/biz/activity/agg/ActivityStar2025Biz.java

问题背景

周星活动预排名逻辑中，调用 rankService.queryTwoRank() 查询排名数据时，未对返回结果进行空值检查。当活动尚未产生排名数据时，返回的 RankVO 为 null，后续调用 rankVO.getRanks() 直接导致 NullPointerException。

解决方案

在调用 rankService.queryTwoRank() 后增加空值检查，若返回 null 则记录 warn 日志并直接返回 null。

关键代码

RankVO rankVO = rankService.queryTwoRank(actBaseReq, uid, DefaultCycleTypeEnum.TOTAL.type(), millSeconds, gid.toString(), 1, 1);

// 添加空值检查，避免NullPointerException
if (rankVO == null) {
    log.warn("ActivityStar2025Biz preRank: rankVO is null for gid={}, uid={}, campaignId={}", 
             gid, uid, actBaseReq.getCampaignId());
    return null;
}

2. 常驻活动星光榜送礼价值统计 Bug（多人送礼只统计一次）

提交记录: kella-server 1a785251 (2026-01-07) 涉及文件:

  • kella-ugc/src/main/java/me/ddkj/ugc/biz/activity/task/special/star2025/Star2025GiftSendEventTaskRankBiz.java

问题背景

星光榜活动的送礼任务排名统计中，当多人同时给同一主播送礼时，送礼价值只被统计了一次。这是因为循环内的累加逻辑存在变量作用域或赋值顺序问题，导致同一批礼物中只有第一条被正确累加。

解决方案

修复循环内的累加逻辑，确保每条送礼记录都被正确统计到总送礼价值中。

关键代码

// 修复前：循环内变量未正确累加
for (GiftSendEvent event : events) {
    totalValue = event.getValue();  // 错误：覆盖而非累加
}

// 修复后：正确累加每条送礼记录的价值
totalValue += event.getValue();  // 使用 += 累加

User userInfo = userAdapter.getUserInfo(guildInfoDTO.getGuildId());

}

4. 聊天室用户封禁检查空指针异常

提交记录: kella-server dad44526 (2025-12-31) 涉及文件:

  • kella-chatroom/src/main/java/me/ddkj/chatroom/service/ChatroomLocalService.java
  • kella-relation/src/main/java/me/ddkj/relation/rpc/UserTaskServiceImpl.java
  • kella-system/src/main/java/me/ddkj/system/rpc/IMPushServiceImpl.java

问题背景

disableEnterByCms 方法中，直接调用 chatroomUserBanService.isUserBanned() 后链式获取 .getData()，存在多处 NPE 风险：

  1. req 参数本身可能为 null；
  2. req.getUid() 或 req.getRoomId() 可能为 null；
  3. isUserBanned() 返回的 ObjectRsp 可能为 null；
  4. ObjectRsp.getData() 可能为 null。

此外，还顺手修复了 System.out.println 打印日志的问题（应使用 log.info）。

解决方案

  1. 增加参数判空；
  2. 拆分链式调用，逐步判空；
  3. 使用 Boolean.TRUE.equals() 安全比较布尔值。

关键代码

private void disableEnterByCms(ChatroomEnterReq req) {
    // 参数校验，避免空指针异常
    if (req == null || req.getUid() == null || req.getRoomId() == null) {
        return;
    }
    
    ObjectRsp<Boolean> banRsp = chatroomUserBanService.isUserBanned(
            req.getUid().longValue(), 
            req.getRoomId().longValue()
    );
    
    // 检查返回结果，避免空指针异常
    if (banRsp != null && Boolean.TRUE.equals(banRsp.getData())) {
        throw new SpecialRuntimeException(MessageEnums.ERR_CHATROOM_USER_BAN);
    }
}

5. 订阅任务过期处理空指针异常

提交记录: kella-server a0d88887 (2025-06-04) 涉及文件:

  • kella-user/src/main/java/me/ddkj/user/biz/impl/SubscribeBizImpl.java

问题背景

订阅等级过期定时任务中，从 Redis 获取的过期 UID 列表可能包含 null 值（数据异常或并发写入问题）。后续直接调用 lvDao.getByUid(uid) 时传入 null 导致异常。另外，lvExpire() 方法中也未检查 lvDO 是否为 null。

解决方案

  1. 循环内增加 uid == null 跳过判断；
  2. lvExpire() 方法增加 lvDO == null 提前返回。

关键代码

for (Integer uid : uids) {
    if (uid == null) {
        continue;  // 跳过 null uid
    }
    final UserSubscribeLvDO lvDO = lvDao.getByUid(uid);
    // ...
}

private void lvExpire(Integer uid) {
    final UserSubscribeLvDO lvDO = lvDao.getByUid(uid);
    if (lvDO == null) return;  // 增加空值保护
    lvDO.setExpireUnix(0);
    lvDO.setLv(0);
    lvDO.setLvUuid("");
}

6. 充值订单成功回调覆盖失败订单 Bug

提交记录: kella-server 29f08ce8 (2026-02-25) 涉及文件:

  • kella-wealth/src/main/java/me/ddkj/wealth/service/order/AbstractOrderFlow.java

问题背景

充值订单的支付成功回调处理中，没有对订单当前状态做精确判断。当一笔订单先收到失败回调、后收到成功回调时，失败订单被成功状态覆盖，导致金币重复发放。

解决方案

  1. 增加订单号空值校验；
  2. 查询不到订单时直接返回；
  3. 只有订单状态不是 FINISH_SUCCESS、PAY_SEND、REFUND 时才允许修改为已支付。

关键代码

public void callbackSuccessOrder(String orderNo) {
    MessageEnums.ERR_INVALID_PARAM.assertIsTrue(StringUtils.isNotEmpty(orderNo));
    // ... 获取分布式锁 ...
    
    BeanOrderEntity beanOrderEntity = beanOrderMapper.getBeanOrderByOrderNoV1(orderNo);
    if (beanOrderEntity == null) {
        log.warn("[orderNo={}] callback order not found", orderNo);
        return;
    }
    // 精确状态判断：已完成的订单不允许再次修改
    if (!beanOrderEntity.getState().equals(PayEnums.PayState.FINISH_SUCCESS.getState()) &&
        !beanOrderEntity.getState().equals(PayEnums.PayState.PAY_SEND.getState()) &&
        !beanOrderEntity.getState().equals(PayEnums.PayState.REFUND.getState())) {
        beanOrderEntity.orderPayed();
    }
}

7. CMS 批量添加主播空 UID Bug

  • cms-main-web/src/main/java/com/ddkj/cms/controller/operation/bobo/AnchorController.java

问题背景

CMS 批量导入主播时，前端传入的数据可能包含 uid 为 null 或 0 的记录。原代码中过滤逻辑被注释掉了，导致空 UID 记录进入后续处理流程，引发数据异常。

list = list.stream()
           .filter(e -> e.getUid() != null && e.getUid() != 0)
           .collect(Collectors.toList());

if (list.size() > 500) {
    return new BaseResponse(-1, "导入数量过多 单次操作不超过500");
}

8. 礼物定制档位差值计算 Bug

提交记录: kella-server 80f5aefc (2026-02-06) 涉及文件:

  • kella-wealth/src/main/java/me/ddkj/wealth/rpc/GiftRankServiceImpl.java

问题背景

礼物定制排行榜中，计算用户到下一档位的差值时，原逻辑只判断了当前所在档位的 rechargeRequirement > recharge，未考虑用户可能跨越多个档位的情况。当用户充值金额超过当前档位但未达到更高档位时，差值计算错误。

解决方案

抽取独立的 calculateNextTierDifference() 方法，遍历所有档位列表（降序排列），找到第一个 rechargeRequirement > recharge 的档位并计算差值。如果已达到最高档位，差值设为 0。

关键代码

private void calculateNextTierDifference(Integer uid, Long recharge, GiftCustomRankVO giftCustomRankVO) {
    List<GiftCustomGiftTier> tiers = getGiftCustomGiftTier();
    // 找到下一个更高档位
    for (GiftCustomGiftTier tier : tiers) {
        if (tier.getRechargeRequirement() > recharge) {
            giftCustomRankVO.setDifMoney(
                new BigDecimal(tier.getRechargeRequirement() - recharge)
            );
            return;
        }
    }
    // 已达最高档位，差值为0
}

9. 查询条件逻辑错误（arena 统计）

提交记录: kella-server 67bcdbeb (2025-06-19) 涉及文件:

  • kella-chatroom/src/main/java/me/ddkj/chatroom/dao/impl/ChatroomArenaAuthDaoImpl.java

问题背景

房间竞技场记录统计的 SQL 条件构建存在逻辑错误。原代码中 room_id 和 room_mode 的查询条件被包裹在同一个 if (roomId != null) 块内，导致当 roomId 为 null 时，room_mode 条件也被跳过，即使 roomFunMode 不为 null。

解决方案

将三个查询条件独立判断，各自根据对应的参数是否为 null 来决定是否添加到 QueryWrapper。

关键代码

// 修复前：room_mode 条件被 room_id 条件包裹
if (requestBO.getRoomId() != null) {
    authDos.eq("room_id", requestBO.getRoomId());
    authDos.eq(requestBO.getRoomFunMode() != null, "room_mode", requestBO.getRoomFunMode());
}

// 修复后：三个条件独立判断
authDos.eq(requestBO.getRoomId() != null, "room_id", requestBO.getRoomId());
authDos.eq(requestBO.getUid() != null, "uid", requestBO.getUid());
authDos.eq(requestBO.getRoomFunMode() != null, "room_mode", requestBO.getRoomFunMode());

10. 活动复制 Hash 计算 Bug（时间字段导致匹配失败）

提交记录: kella-server 47c8b214 (2025-10-16) 涉及文件:

  • kella-ugc/src/main/java/me/ddkj/ugc/rpc/NewActivityCmsServiceImpl.java

问题背景

活动复制和活动推送到生产环境时，使用奖励记录的字段生成 MD5 hash 来匹配新旧活动的奖励 ID。原实现将 createTime 和 updateTime 也纳入 hash 计算，但测试环境和生产环境的时间字段必然不同，导致 hash 不匹配，奖励映射关系建立失败。

解决方案

从 hash 计算中移除 createTime 和 updateTime 字段，只使用业务相关字段（goodsType、name、desc、extJson、status）生成 hash。

关键代码

// 修复前：包含时间字段，环境间必然不同
String hashStr = awardDO.getGoodsType() + "|" + awardDO.getName() + "|" 
               + awardDO.getDesc() + "|" + awardDO.getExtJson() + "|" 
               + awardDO.getStatus() + "|" + awardDO.getCreateTime() + "|" + awardDO.getUpdateTime();

// 修复后：移除时间字段
String hashStr = awardDO.getGoodsType() + "|" + awardDO.getName() + "|" 
               + awardDO.getDesc() + "|" + awardDO.getExtJson() + "|" 
               + awardDO.getStatus();

12. 房间数据统计查询 Bug（单房间ID vs 多房间ID列表）

提交记录: kella-cms 331c1514 (2024-08-07) 涉及文件:

  • cms-api/src/main/java/com/ddkj/api/operation/ChatroomServer.java
  • cms-system/src/main/java/com/ddkj/cms/dao/ChatroomDao.java

问题背景

关键代码

<!-- 修复前 -->
<if test="roomId!=null and roomId!=''">and r.roomId =#{roomId} </if>

<!-- 修复后 -->
<if test="roomIds!=null and roomIds!=''">and r.roomId in (${roomIds}) </if>

14. StablePay 充值回调状态区分 Bug

提交记录: kella-server 9630c4b0 (2026-03-19) 涉及文件:

  • kella-wealth/src/main/java/me/ddkj/wealth/service/order/AbstractOrderFlow.java
  • kella-wealth/src/main/java/me/ddkj/wealth/service/order/stablepay/StablePayPayService.java

问题背景

StablePay 支付渠道的回调中，所有失败类型（支付过期、用户取消、系统错误）都被统一标记为 ERROR 状态，无法区分具体失败原因，影响后续订单统计和用户提示。

解决方案

  1. 在 OrderFlow 接口中增加带状态参数的 callbackSuccessFail 重载方法；
  2. 在 StablePayPayService 中根据回调类型 type 区分失败状态：payment.expired → ERROR_TIMEOUT，payment.cancelled → ERROR_FRONT_CANCEL，其他 → ERROR。

关键代码

// 根据错误类型设置不同的状态
PayEnums.PayState errorState;
if ("payment.expired".equals(type)) {
    errorState = PayEnums.PayState.ERROR_TIMEOUT;
} else if ("payment.cancelled".equals(type)) {
    errorState = PayEnums.PayState.ERROR_FRONT_CANCEL;
} else {
    errorState = PayEnums.PayState.ERROR;
}
orderFlow.callbackSuccessFail(beanOrder.getOrderNo(), type, "StablePay payment " + type, errorState);
