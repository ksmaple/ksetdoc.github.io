性能与大数据量优化汇总

本文档汇总了项目历史上针对性能瓶颈、大数据量处理、算法优化等方面的技术改进。

1. Redis 批量获取优化（Jedis → Redisson）

提交记录: kella-spring-boot-starter 2c9ba19d (2022-07-22) 涉及文件:

  • src/main/java/verify/ddkj/config/redis/BaseCache.java
  • src/main/java/verify/ddkj/config/redis/BaseRedisClient.java
  • src/main/java/verify/ddkj/config/redis/RedisClient.java

问题背景

原 Redis 批量获取实现（mget）基于 JedisCluster，需要手动计算 key 的槽位（slot）并按槽位分组，代码复杂且性能不佳。更关键的是，JedisCluster 的 mget 在跨槽位时性能下降明显，且需要自己处理连接异常和重试。

解决方案

将批量获取逻辑从 JedisCluster 迁移到 Redisson，利用 Redisson 的 RBucket.get() 批量操作 API：

  1. 简化接口：删除手动实现的 mmget 和 mget 方法；
  2. 使用 Redisson Buckets：通过 redissonClient.getBuckets(codec).get(keys) 一次性批量获取，Redisson 内部自动处理槽位路由和管道优化；
  3. 支持类型化反序列化：使用 TypedJsonJacksonCodec 自动将 JSON 字符串反序列化为目标类型。

关键代码

@Override
public <T> List<T> getObjList(String preKey, Class<T> clazz, String... keys) {
    Set<String> keyList = Arrays.stream(keys).map(k -> preKey + k).collect(Collectors.toSet());
    // Redisson 自动处理批量获取和槽位路由
    Map<String, T> map = get().getBuckets(new TypedJsonJacksonCodec(clazz))
                               .get(keyList.toArray(new String[0]));
    List<T> results = new ArrayList<>();
    for (int i = 0; i < keys.length; i++) {
        results.add(map.get(preKey + keys[i]));
    }
    return results;
}

收益

  • 代码复杂度大幅降低；
  • Redisson 内部使用管道（pipeline）批量发送命令，减少网络 RTT；
  • 自动处理集群槽位映射和异常重试。

2. CMS 房间数据统计汇总逻辑优化

提交记录: kella-cms c974ebd0 (2024-09-26) 涉及文件:

  • cms-main-web/src/main/java/com/ddkj/cms/controller/operation/bobo/ChatRoomController.java

问题背景

CMS 房间数据统计中，DAU（日活跃用户）的汇总逻辑存在错误。原代码在循环累加时对每个房间的 DAU 除以 2，导致汇总结果偏小。正确的做法应该是先累加所有房间的 DAU，再除以房间总数求平均值。

解决方案

调整计算顺序：先完整累加所有房间的 roomDau，循环结束后再除以房间数量取平均值。

关键代码

// 修复前：循环内就除以2
chatRoomST.setRoomDau(chatRoomST.getRoomDau() + roomST.getRoomDau() / 2);

// 修复后：先完整累加，循环结束后再平均
chatRoomST.setRoomDau(chatRoomST.getRoomDau() + roomST.getRoomDau());
// ...
chatRoomST.setRoomDau(chatRoomST.getRoomDau() / roomSTList.size());

3. 1v1 机器人 Elasticsearch 查询优化（unmappedType）

提交记录: kella-server d9a69f10 (2025-07-09) 涉及文件:

  • kella-relation/src/main/java/me/ddkj/relation/dao/impl/RobotDaoImpl.java

问题背景

1v1 机器人查询使用 Elasticsearch 按 updateTime 字段排序。当索引中某些分片不存在 updateTime 字段映射时，排序操作会抛出 unmapped field 异常，导致查询失败。

解决方案

在排序构造中增加 .unmappedType("long")，指定当字段未映射时按 long 类型处理，避免异常。

关键代码

SearchQuery searchQuery = new NativeSearchQueryBuilder()
    .withQuery(query)
    .withPageable(PageRequest.of(req.getPage() - 1, req.getPageSize()))
    .withSort(SortBuilders.fieldSort("updateTime")
            .order(SortOrder.DESC)
            .unmappedType("long"))  // 指定unmappedType，避免未映射异常
    .build();

4. 白名单查询优化（移除冗余接口）

提交记录: kella-server 4802a0ae (2025-11-28) 涉及文件:

  • kella-api/src/main/java/me/ddkj/api/wealth/OpenSystemService.java
  • kella-wealth/src/main/java/me/ddkj/wealth/biz/impl/OpenSystemBizImpl.java
  • kella-wealth/src/main/java/me/ddkj/wealth/rpc/OpenSystemServiceImpl.java

问题背景

游戏白名单查询接口 listWhiteUser 涉及多次 RPC 调用（查询白名单 → 查询用户信息 → 组装 DTO），在数据量大时性能较差。且该接口实际上已不被前端使用，属于冗余接口。

解决方案

直接移除 listWhiteUser 接口及其实现，减少不必要的 RPC 调用和数据处理逻辑。

收益

  • 减少一次数据库查询 + 一次用户信息服务 RPC 调用；
  • 降低服务复杂度；
  • 避免大数据量下的性能问题。

5. 充值订单查询优化（IPv4-only + 超时重试）

提交记录: kella-server 9630c4b0 (2026-03-19) 涉及文件:

  • kella-utils/src/main/java/me/ddkj/util/OkHttp.java
  • kella-wealth/src/main/java/me/ddkj/wealth/service/order/stablepay/StablePayHelper.java

问题背景

StablePay 支付渠道在查询支付会话状态时，经常出现 UnknownHostException 和 SocketTimeoutException，原因是 DNS 解析返回了 IPv6 地址，而服务器 IPv6 网络不通。此外，原实现使用独立的 OkHttpClient 且缺少重试机制，一旦失败即判定订单失败，可能误判"支付中"订单为失败。

解决方案

  1. IPv4-only DNS：自定义 DNS 解析器，过滤掉 IPv6 地址，只使用 IPv4；
  2. 统一客户端：将 StablePay 的独立客户端合并到全局 OkHttp 单例中，使用统一的连接池和调度器；
  3. 超时重试：查询支付会话增加重试机制（最多 2 次，间隔 150ms）；
  4. 异常降级：查询异常时返回 isPayWait=true，避免误判失败。

关键代码

// IPv4-only DNS
.dns(hostname -> {
    List<InetAddress> all = Dns.SYSTEM.lookup(hostname);
    List<InetAddress> ipv4 = new ArrayList<>();
    for (InetAddress address : all) {
        if (address instanceof Inet4Address) {
            ipv4.add(address);
        }
    }
    return ipv4;
})

// 查询会话带重试
private static final int QUERY_SESSION_MAX_RETRY = 2;
private static final long QUERY_SESSION_RETRY_DELAY_MS = 150L;

public StablePaySessionRsp querySession(String sessionId) {
    for (int attempt = 1; attempt <= QUERY_SESSION_MAX_RETRY; attempt++) {
        try {
            String execute = OkHttp.build().executeIpv4Only(request);
            if (execute != null) {
                return JSONObject.parseObject(execute, StablePaySessionRsp.class);
            }
        } catch (SocketTimeoutException | UnknownHostException e) {
            log.warn("stablepay querySession timeout/unknown host [attempt={}]", attempt);
        }
    }
    return null;
}

7. CMS 赠送金币推送改为异步

提交记录: kella-cms 363b4988 / fb8c6095 / 34ae01e8 / 9a756501 (2024-08-14) 涉及文件:

  • cms-main-web 相关控制器

问题背景

CMS 后台赠送金币时，同步调用推送服务，如果推送服务响应慢或超时，会导致前端接口长时间阻塞，甚至超时失败。

解决方案

将金币推送逻辑改为异步执行，赠送金币操作立即返回成功，推送任务放到后台线程池处理。

收益

  • 前端接口响应时间从秒级降到毫秒级；
  • 推送失败不影响金币赠送主流程。
