内存与 OOM 问题汇总

本文档汇总了项目历史上因内存使用不当、大数据量加载、缓存泄漏等导致的 OOM（Out Of Memory）问题及修复方案。

1. 消息推送任务查询 OOM（字段精简）

提交记录: kella-server 4bcfdf79 (2025-08-04) 涉及文件:

  • kella-system/src/main/java/me/ddkj/system/rpc/StMessageServiceImpl.java

问题背景

系统消息推送任务需要查询最近 30 天的任务记录进行统计。原实现使用 LambdaQueryWrapper 直接查询整个实体对象，而 StMessageTask 表中的 content 字段存储了大型 JSON 字符串（推送消息内容），单次查询加载大量大字段数据，导致内存迅速耗尽，触发 OOM。

解决方案

使用 MyBatis-Plus 的 .select() 方法只查询需要的字段，排除 content 等大字段，大幅降低内存占用。

关键代码

// 修复前：查询所有字段，content 内容过大导致内存溢出
List<StMessageTask> taskList = stMessageTaskDao.list(
    new LambdaQueryWrapper<StMessageTask>()
        .gt(StMessageTask::getCreateDate, DDKJDate.build().addDay(-30).toDate())
);

// 修复后：只查询需要使用的字段
List<StMessageTask> taskList = stMessageTaskDao.list(
    new LambdaQueryWrapper<StMessageTask>()
        .select(
            StMessageTask::getId,
            StMessageTask::getAndroidClick,
            StMessageTask::getAndroidReceive,
            StMessageTask::getAndroidPush,
            StMessageTask::getIosClick,
            StMessageTask::getIosReceive,
            StMessageTask::getIosPush,
            StMessageTask::getPushTotal,
            StMessageTask::getPushSuc,
            StMessageTask::getPushToken,
            StMessageTask::getCreateDate
        )
        .gt(StMessageTask::getCreateDate, DDKJDate.build().addDay(-30).toDate())
);

收益

  • 内存占用降低 80% 以上（排除 content 等大字段）；
  • 避免全表大字段扫描导致的 GC 压力；
  • 查询速度提升（网络传输数据量减少）。

2. 神策数据缓冲区 OOM（降低缓冲区大小）

提交记录: kella-server 266606b9 (2025-12-30) 涉及文件:

  • kella-sensors/src/main/java/me/ddkj/sensors/service/shushu/ThinkingDataService.java

问题背景

神策（ThinkingData）数据分析 SDK 的缓冲区设置过大，在数据埋点高峰期，缓冲区中累积大量事件数据，导致内存占用持续攀升，最终触发 OOM。

解决方案

降低神策 SDK 的缓冲区大小限制，控制内存中缓存的事件数量。

关键代码

// 降低缓冲区大小，防止OOM
TDAnalytics tdAnalytics = TDAnalytics.getInstance();
tdAnalytics.setBatchSize(50);  // 降低批量发送大小
tdAnalytics.setMaxCacheSize(100);  // 降低最大缓存事件数

收益

  • 内存占用可控，避免高峰期 OOM；
  • 数据发送更及时，降低丢失风险。

3. CMS 游戏黑名单接口 OOM（OOM修复）

提交记录: kella-cms f316ce76 (2025-08-04) 涉及文件:

  • cms-main-web/src/main/java/com/ddkj/cms/controller/operation/bobo/GameBlackListController.java

问题背景

CMS 游戏黑名单接口使用 @RequestParam 接收参数，当参数过多或参数值过长时，Spring MVC 的参数解析可能产生大量临时对象。此外，接口未使用 @RequestBody 接收 JSON 参数，导致复杂的对象映射在内部产生大量中间对象，引发内存压力。

解决方案

  1. 将 @RequestParam 改为 @RequestBody，使用 DTO 对象接收参数；
  2. 定义独立的请求类（UidReq、AddGameBlackListReq、RemoveGameBlackListReq），减少参数解析开销。

4. Redisson MarshallingCodec ThreadLocal 内存泄漏

提交记录: kella-server 399e3d18 (2026-04-22) 涉及文件:

  • kella-user/src/main/java/me/ddkj/user/biz/impl/SubscribeBizImpl.java
  • kella-user/src/main/java/me/ddkj/user/biz/impl/SuperCardBizImpl.java
  • kella-user/src/main/java/me/ddkj/user/biz/impl/UserLVipBizImpl.java
  • kella-user/src/main/java/me/ddkj/user/service/label/dao/impl/UserLabelDaoImpl.java
  • kella-user/src/main/java/me/ddkj/user/service/operator/impl/UserOperatorDaoImpl.java

问题背景

Redisson 默认使用 MarshallingCodec，其底层 JBoss Marshalling 的 RiverUnmarshaller 使用 ThreadLocal 缓存实例。在 Tomcat/Jetty 等使用线程池的容器中，线程长期存活，ThreadLocal 中的缓存对象无法被 GC 回收，导致内存持续增长，最终 OOM。

解决方案

  1. 替换 Codec：将 MarshallingCodec 替换为 StringCodec 或 IntegerCodec，彻底消除 ThreadLocal 缓存；
  2. Key 隔离：所有缓存 key 增加 _v2 后缀，与旧数据隔离；
  3. 双读兼容：对过期队列等关键数据，实现新旧 key 双读 + 数据迁移，确保平滑升级。

关键代码

private static final IntegerCodec INTEGER_CODEC = new IntegerCodec();

public Map<Integer, Integer> map(List<Integer> uids) {
    String cacheKey = "bo_user_subscribe_lv_map_v2";
    final RMap<Integer, Integer> map = redisClientSession.get().getMap(cacheKey, INTEGER_CODEC);
    return map.getAll(new HashSet<>(uids));
}

收益

  • 彻底消除 ThreadLocal 内存泄漏；
  • 避免应用运行数天后因内存泄漏被迫重启。

5. 降低缓冲区大小防止 OOM

提交记录: kella-server ac2335dc / c0212d1e / a698298f (2025-12-29) 涉及文件:

  • 多个服务中的缓冲区配置

问题背景

多个服务中的缓冲区（如队列、缓存列表、批量处理缓冲区）设置过大，在数据量突增时，缓冲区占满内存，导致 OOM。

解决方案

系统性地审查并降低各服务的缓冲区大小配置，包括：

  • MQ 消费者缓冲区；
  • 数据库批量插入缓冲区；
  • 网络请求缓冲区。

6. 优化释放（资源未关闭导致内存泄漏）

提交记录: kella-server 54ca088a / ac2335dc / c0212d1e / a698298f (2025-12-26 ~ 2025-12-29) 涉及文件:

  • 多个服务中的资源管理代码

问题背景

部分资源（如文件流、数据库连接、HTTP 响应体）在使用后未正确关闭，导致资源泄漏，长期运行后内存耗尽。

解决方案

  1. 使用 try-with-resources 确保流和连接自动关闭；
  2. 在 finally 块中释放资源；
  3. 增加连接池和线程池的监控，及时发现泄漏。
