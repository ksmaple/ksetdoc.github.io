Redisson编解码器创建异常

核心结论：Redisson默认使用Kryo编解码器时，若类加载器环境复杂（如OSGi、热部署）或类无默认构造函数，会导致编解码器初始化失败，显式指定FstCodec或JsonJacksonCodec并确保类可被正确加载可规避此问题。

问题现象

线上服务启动时偶发报错：

org.redisson.client.RedisException: 
Codec for class xxx not found
    at org.redisson.codec.Kryo5Codec$1.encode(Kryo5Codec.java:xx)
Caused by: com.esotericsoftware.kryo.KryoException: 
Unable to find class: com.example.UserDTO

影响范围

  • 服务启动阶段约30%概率失败
  • 影响缓存读写功能，但Redis服务本身正常
  • 重启可能恢复，但不可接受

排查过程

第一步：确认编解码器配置

// 原有配置（未显式指定，使用默认值）
Config config = new Config();
config.useClusterServers().addNodeAddress("redis://...");
// 未设置 codec，默认使用 Kryo5Codec

第二步：分析类加载环境

项目使用 Spring Boot + 热部署 devtools
├── 主类加载器：AppClassLoader
├── devtools 重启类加载器：RestartClassLoader
└── Kryo 内部缓存的 Class 引用来自旧类加载器

问题：热部署后类加载器变化，Kryo缓存的Class引用失效

第三步：复现验证

// 本地复现步骤
1. 启动应用，写入缓存（Kryo注册类）
2. 修改代码触发 devtools 热重启
3. 再次读写缓存 → 抛出 KryoException

根因分析

因素说明
Kryo默认行为缓存Class引用以提升序列化性能
热部署机制devtools使用RestartClassLoader重新加载类
引用失效Kryo缓存的Class来自旧类加载器，热部署后无法匹配新类加载器的类
默认无参构造部分DTO类缺少默认构造函数，Kryo无法实例化

解决方案

方案一：更换编解码器（推荐）

// 显式使用 JsonJacksonCodec，避免类加载器依赖
Config config = new Config();
config.setCodec(new JsonJacksonCodec());  // JSON序列化，不缓存Class引用
config.useClusterServers().addNodeAddress("redis://...");
编解码器优点缺点
Kryo5Codec序列化体积小、速度快类加载器敏感、需无参构造
FstCodec性能接近Kryo同样存在类加载器问题
JsonJacksonCodec兼容性好、可读性强体积大、速度慢
TypedJsonJacksonCodec带类型信息兼容性和体积的平衡

方案二：禁用热部署（开发环境）

# application.yml
spring:
  devtools:
    restart:
      enabled: false  # 开发环境若无需热部署可关闭

方案三：确保类可被正确加载

// 为DTO添加无参构造函数
public class UserDTO implements Serializable {
    private static final long serialVersionUID = 1L;
    
    // 必须：无参构造
    public UserDTO() {}
    
    // 业务构造
    public UserDTO(Long id, String name) {
        this.id = id;
        this.name = name;
    }
}

复盘与预防

措施执行方式
编码规范所有缓存对象必须实现Serializable、保留无参构造
配置检查Redisson配置必须显式指定Codec，禁止依赖默认值
环境统一开发和生产使用相同的Codec配置
升级评估升级Redisson版本时评估Codec兼容性变化
