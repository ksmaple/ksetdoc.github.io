网络延迟与稳定性排查

核心结论：网络延迟问题通常由DNS解析、路由跳变、TCP重传、带宽竞争或应用层超时配置不当引起，通过分层排查（物理层→网络层→传输层→应用层）可以快速定位根因。

问题现象

线上服务出现间歇性超时：

Caused by: java.net.SocketTimeoutException: Read timed out
    at java.net.SocketInputStream.socketRead0(Native Method)

HTTP调用P99从50ms上升到800ms
部分请求正常，部分请求超时

影响范围

  • 服务间RPC调用超时率上升
  • 用户体验下降，接口响应变慢
  • 触发熔断，影响正常流量

排查过程

第一步：确认问题层级

OSI分层排查：

应用层（L7）→ HTTP/RPC超时配置是否合理？
  ↓
传输层（L4）→ TCP连接是否正常？是否有重传？
  ↓
网络层（L3）→ 路由是否跳变？是否有丢包？
  ↓
链路层（L2）→ 网卡/交换机是否有错误？
  ↓
物理层（L1）→ 带宽是否打满？光纤/网线是否正常？

第二步：网络层排查命令

# 1. 测试连通性和延迟
ping -c 100 target-ip
# 关注：丢包率、延迟波动

# 2. 追踪路由路径
traceroute target-ip
# 关注：哪一跳延迟突增

# 3. 更详细的网络质量检测
mtr --report --report-cycles 100 target-ip
# 关注：Loss%和Last列

# 4. TCP连接状态
ss -tan | awk '{print $1}' | sort | uniq -c
# 关注：TIME_WAIT、CLOSE_WAIT数量

第三步：传输层排查

# 查看TCP重传率
cat /proc/net/snmp | grep Tcp:
# RetransSegs / OutSegs = 重传率，>1%需关注

# 抓包分析（生产慎用）
tcpdump -i eth0 host target-ip -w capture.pcap
# Wireshark分析：Filter tcp.analysis.retransmission

# 连接跟踪
conntrack -L | grep target-ip | wc -l
# 连接数是否超过阈值

第四步：应用层排查

// 检查连接池配置
HikariConfig config = new HikariConfig();
config.setConnectionTimeout(5000);   // 获取连接等待时间
config.setMaxLifetime(1800000);      // 连接最大生命周期
config.setMaximumPoolSize(50);       // 最大连接数

// HTTP客户端超时配置
RestTemplate restTemplate = new RestTemplate();
SimpleClientHttpRequestFactory factory = new SimpleClientHttpRequestFactory();
factory.setConnectTimeout(3000);     // 建立连接超时
factory.setReadTimeout(5000);        // 读取响应超时
restTemplate.setRequestFactory(factory);

根因分析

根因现象验证方式
DNS解析慢首次请求慢，后续正常dig/nslookup看解析时间
路由跳变traceroute路径不稳定多次traceroute对比
TCP重传网络丢包或拥塞/proc/net/snmp重传率
带宽打满延迟伴随丢包iftop/nload看流量
连接数耗尽大量TIME_WAIT/CLOSE_WAITss -tan统计
超时配置不当超时时间小于正常RTT对比ping延迟和超时配置

解决方案

网络层优化

# 1. DNS优化：使用本地缓存或IP直连
# 2. 路由优化：调整BGP策略或使用专线
# 3. 内核参数调优

# 减少TIME_WAIT
echo 'net.ipv4.tcp_tw_reuse = 1' >> /etc/sysctl.conf
# 增大连接跟踪表
echo 'net.netfilter.nf_conntrack_max = 655350' >> /etc/sysctl.conf
sysctl -p

应用层优化

// 1. 合理设置超时（基于P99 + 缓冲）
// 若正常P99=100ms，设置readTimeout=300ms

// 2. 连接池预热
@PostConstruct
public void warmUp() {
    // 应用启动时预热连接池
    dataSource.getConnection().close();
}

// 3. 熔断降级
@CircuitBreaker(name = "remoteService", fallbackMethod = "fallback")
public Response callRemote() {
    return remoteClient.call();
}

架构层优化

手段说明
就近部署服务部署在同一可用区/机房
连接池复用减少TCP握手开销
异步化非关键调用改异步，降低延迟敏感度
本地缓存减少跨网络调用次数

复盘与预防

措施执行方式
网络基线定期记录各链路RTT和丢包率，建立正常基线
告警阈值延迟超过基线200%时告警
超时设计超时设置基于实际测量值，非拍脑袋
混沌工程定期模拟网络延迟和丢包，验证系统韧性
