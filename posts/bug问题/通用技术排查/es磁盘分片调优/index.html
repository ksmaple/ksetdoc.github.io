ES磁盘不足与分片内存调优

核心结论：Elasticsearch的稳定性高度依赖磁盘空间和JVM内存的合理规划，磁盘不足会触发只读保护导致写入中断，分片过多会耗尽堆内存引发OOM，通过索引生命周期管理、分片数量规划和水位线监控可预防此类问题。

问题现象

线上ES集群出现以下异常：

# 现象1：写入被拒绝
cluster_block_exception: index [logs-2024-01] blocked by: [FORBIDDEN/12/index read-only / allow delete (api)];

# 现象2：节点频繁GC
[gc][23451] overhead, spent [2.1s] collecting in the last [2.3s]

# 现象3：节点掉线
Master node discovery: node-left reason: disconnected

影响范围

  • 日志写入失败，数据丢失风险
  • 查询延迟飙升，影响检索功能
  • 节点频繁掉线，集群稳定性下降

排查过程

第一步：检查集群健康状态

# 集群整体状态
GET _cluster/health
# status: red/yellow, unassigned_shards > 0

# 节点磁盘和内存
GET _nodes/stats/fs,os,jvm
# fs.total.free_in_bytes: 磁盘剩余
# jvm.mem.heap_used_percent: 堆内存使用百分比

第二步：分析磁盘使用

# 查看各索引占用空间
GET _cat/indices?v&s=store.size:desc
# 发现旧日志索引占用了80%空间

# 查看磁盘水位线
GET _cluster/settings?include_defaults=true&filter_path=**.disk.watermark
# low: 85%, high: 90%, flood_stage: 95%

第三步：分析内存使用

# 分片级别内存占用
GET _cat/shards?v&s=store:desc
# 发现单个节点承载300+分片

# 段内存占用（Lucene底层）
GET _nodes/stats/indices/segments?filter_path=nodes.*.indices.segments
# memory_in_bytes 过大

根因分析

磁盘不足根因

因素说明
缺乏清理机制日志索引只增不减，无限累积
副本设置过多重要索引5副本，空间放大
无生命周期管理冷热数据未分离
水位线触发磁盘使用>95%自动进入只读

内存不足根因

因素说明
分片过多每分片都有堆内存开销，推荐单节点<600分片
字段爆炸动态映射导致字段数失控
聚合查询大聚合占用大量内存
堆内存设置不当超过32GB无法使用CompressedOops

解决方案

磁盘空间紧急恢复

# 1. 释放只读限制（临时）
PUT */_settings
{
  "index.blocks.read_only_allow_delete": null
}

# 2. 删除过期索引（确认无用后）
DELETE /logs-2023-*

# 3. 收缩副本（临时释放空间）
PUT /important-index/_settings
{
  "number_of_replicas": 0
}

索引生命周期管理（ILM）

PUT _ilm/policy/logs_policy
{
  "policy": {
    "phases": {
      "hot": {
        "actions": {
          "rollover": {
            "max_size": "50GB",
            "max_age": "1d"
          }
        }
      },
      "warm": {
        "min_age": "7d",
        "actions": {
          "shrink": { "number_of_shards": 1 },
          "forcemerge": { "max_num_segments": 1 }
        }
      },
      "cold": {
        "min_age": "30d",
        "actions": {
          "freeze": {}
        }
      },
      "delete": {
        "min_age": "90d",
        "actions": {
          "delete": {}
        }
      }
    }
  }
}

分片与内存优化

# 1. 合理设置分片数
# 经验：单分片20-50GB，避免过小分片
PUT /new-index
{
  "settings": {
    "number_of_shards": 3,
    "number_of_replicas": 1
  }
}

# 2. 限制字段数
PUT _cluster/settings
{
  "persistent": {
    "mapping.total_fields.limit": 1000
  }
}

# 3. JVM配置
# jvm.options
-Xms16g
-Xmx16g  # 不超过32GB，留一半内存给Lucene文件缓存

监控告警配置

# Prometheus规则示例
groups:
  - name: elasticsearch
    rules:
      - alert: EsDiskHigh
        expr: elasticsearch_filesystem_data_free_percent < 15
        for: 5m
        annotations:
          summary: "ES节点磁盘不足"
      
      - alert: EsHeapHigh
        expr: elasticsearch_jvm_memory_used_bytes / elasticsearch_jvm_memory_max_bytes > 0.85
        for: 5m
        annotations:
          summary: "ES堆内存使用率过高"
      
      - alert: EsShardTooMany
        expr: elasticsearch_indices_shards_total > 600
        annotations:
          summary: "ES节点分片数过多"

复盘与预防

措施执行方式
容量规划按日增量保留天数（1+副本数）估算磁盘
强制ILM所有日志类索引必须绑定生命周期策略
分片审核新建索引Review分片数，避免默认5分片
定期ForceMerge只读索引定期合并段，减少内存占用
冷热分离热节点SSD、温节点SATA、冷节点可搜索快照
