Claude Code与Agents

核心结论：Claude Code通过Skills定义可复用的任务模板、Rules约束行为边界、MCP扩展外部工具能力，将AI从问答助手升级为可执行复杂工程任务的Agent。

Claude Code通过结构化指令和工具扩展，将大模型能力封装为可重复执行的工程工作流

Skills：可复用任务模板

Skill = 触发词 + 上下文 + 执行逻辑

用户输入：/tech-doc learn Redis
Claude加载：.claude/skills/tech-doc.md 中定义的 learn 模板
执行结果：按模板格式生成学习笔记文档
Skill类型用途示例
文档生成按模板创建技术文档/tech-doc
代码审查按检查清单Review代码/review security,performance
问题诊断按排查流程定位问题/debug memory-leak

Rules：行为约束与上下文

层级位置作用范围
全局规则~/.claude/CLAUDE.md所有项目
项目规则项目根目录/CLAUDE.md当前项目
会话规则对话中直接说明当前会话
# CLAUDE.md 示例

## 编码规范
- 优先使用Java 17新特性
- 所有API接口必须标注 @Operation 注解
- 数据库操作使用MyBatis-Plus

## 常用命令
- 构建：mvn clean package
- 测试：mvn test -Dtest=ClassName#methodName

MCP：外部工具扩展

Claude Code ←→ MCP协议 ←→ 外部工具
                   ↓
            ├── 文件系统（读写代码）
            ├── 数据库（查询表结构）
            ├── API服务（调用内部接口）
            └── 版本控制（git操作）
能力说明场景
文件操作读写、搜索、批量修改代码重构
命令执行运行测试、构建、部署CI/CD集成
浏览器抓取网页、验证UI前端开发
数据库查询schema、执行SQL后端开发

Agent协作模式

模式说明场景
单Agent一个Claude实例处理完整任务常规编码任务
多Agent多个实例分工协作复杂系统设计和实现
人机协同AI生成草稿，人工审核确认关键业务代码

常见问题与误区

问题1：AI生成代码未经验证直接使用

根因：过度信任AI输出，未进行测试和审查 解决：始终运行测试验证，关键逻辑人工Review，将AI输出视为草稿而非终稿

问题2：上下文过长导致理解偏差

根因：单次会话包含过多不相关信息 解决：及时开启新会话，使用 /clear 清理上下文，按任务拆分对话
