Spring AI 架构原理与底层实现

核心结论：Spring AI 通过 Model + Prompt + OutputParser 三层抽象统一了不同大模型供应商的 API 差异，底层基于 RestClient 和 ReactiveChatClient 实现同步/流式双模调用，其设计本质是将 AI 能力转换为 Spring 生态中可装配的标准 Bean。

Spring AI 通过分层架构解耦了模型协议、调用方式与业务逻辑

整体架构：五层模型设计

│  应用层：业务代码调用 ChatClient / EmbeddingClient            │
├─────────────────────────────────────────────────────────────┤

├─────────────────────────────────────────────────────────────┤
│  客户端层：ChatClient / StreamingChatClient 接口抽象          │
├─────────────────────────────────────────────────────────────┤
│  模型层：OpenAiChatModel / AnthropicChatModel / Ollama...    │
├─────────────────────────────────────────────────────────────┤

API 层设计：Prompt 与 Message 的协议无关抽象

Message 体系：统一表达对话上下文

// 核心接口：所有消息类型的基类
public interface Message {
    String getContent();           // 消息内容
    MessageType getMessageType();  // 枚举：USER/ASSISTANT/SYSTEM/TOOL
    Map<String, Object> getMetadata();  // 扩展元数据
}

// 具体实现
new SystemMessage("你是一个专业的Java技术顾问");   // 设定角色
new UserMessage("解释一下Spring的AOP原理");        // 用户输入
new AssistantMessage("Spring AOP 基于动态代理..."); // AI回复
new ToolResponseMessage("tool_call_id", "{\"temp\":25}"); // 工具返回

设计要点：MessageType 枚举确保了无论底层模型使用什么字段名（OpenAI 的 role/content，Claude 的 type/text），上层业务代码始终使用统一类型。

Prompt：携带参数与指令的请求封装

public class Prompt {
    private final List<Message> messages;           // 对话历史
    private final ChatOptions modelOptions;         // 模型参数（温度、最大token等）

    // 构造方式
    Prompt prompt = new Prompt(
        List.of(new SystemMessage("翻译助手"), new UserMessage("Hello")),
        ChatOptions.builder().temperature(0.3).maxTokens(500).build()
    );
}
参数类型作用典型值
temperatureDouble创造性 vs 确定性，越高越随机0.0~1.0，代码生成建议 0.1
maxTokensInteger生成文本的最大长度限制根据场景 256~4096
topPDouble核采样，控制候选词集合0.1~1.0，与 temperature 二选一
frequencyPenaltyDouble降低重复词概率-2.0~2.0
presencePenaltyDouble鼓励引入新话题-2.0~2.0

客户端层：ChatClient 的建造者模式与执行链

ChatClient 接口设计

public interface ChatClient {
    // 同步调用：阻塞等待完整响应
    ChatClientCall call();

    // 流式调用：返回 Flux 逐字推送
    ChatClientStream stream();
}

// 实际使用：流畅的建造者 API
ChatResponse response = chatClient.prompt()
    .system("你是一个SQL优化专家")           // 设置系统提示
    .user("优化这条SQL: SELECT * FROM ...") // 用户输入
    .functions("queryDatabase")             // 注册可调用函数
    .advisors(new LoggingAdvisor())         // AOP拦截器
    .call()                                  // 执行同步调用
    .chatResponse();

执行链内部流程

prompt() 创建建造者
    ├── system() / user() / messages()  组装 Prompt
    ├── functions()                      注册 Function Callback
    ├── advisors()                       注册前置/后置拦截器
    ├── options()                        覆盖模型参数
    │
    ├── call() ──→ DefaultChatClient 执行
    │                  │
    │                  ├── 1. Advisor.before()   前置处理（如自动补全上下文）
    │                  ├── 2. 构建最终 Prompt
    │                  ├── 3. ChatModel.call()   调用底层模型
    │                  ├── 4. 处理 Tool Calls（如果有）
    │                  │      ├── 解析 Function 调用参数
    │                  │      ├── 执行本地 Java 方法
    │                  │      └── 将结果回填，再次调用模型
    │                  ├── 5. Advisor.after()    后置处理（如缓存结果）
    │                  └── 6. 封装 ChatResponse
    │
    └── stream() ──→ 同上，但第3步使用 stream()，返回 Flux<ChatResponse>

Advisor 机制：AOP 式扩展点

// 内置 Advisor：实现常见横切关注点
public class MessageChatMemoryAdvisor implements Advisor {
    @Override
    public ChatResponse adviseCall(ChatClientRequest request, CallChain chain) {
        // 1. 从存储中读取历史消息
        List<Message> history = chatMemory.get(request.conversationId());

        // 2. 将历史注入到当前 Prompt 前面
        ChatClientRequest augmented = ChatClientRequest.from(request)
            .messages(mergeHistory(history, request.messages()))
            .build();

        // 3. 执行调用
        ChatResponse response = chain.nextCall(augmented);

        // 4. 保存本轮对话到记忆
        chatMemory.add(request.conversationId(), response.getResult().getOutput());
        return response;
    }
}

// 常用内置 Advisor

│ Advisor                  │ 作用                               │
├──────────────────────────┼────────────────────────────────────┤
│ MessageChatMemoryAdvisor │ 对话记忆管理（基于 conversationId） │
│ PromptChatMemoryAdvisor  │ Prompt 级别的记忆注入               │
│ QuestionAnswerAdvisor    │ RAG 检索增强（基于 VectorStore）    │

模型层：不同协议组件的底层实现差异

OpenAI 协议组件：行业事实标准

OpenAI 的 REST API 已成为业界最广泛支持的协议格式（Ollama、vLLM、OneAPI 等均兼容）。

public class OpenAiChatModel implements ChatModel {
    private final OpenAiApi openAiApi;        // Retrofit/Feign 风格的 API 接口
    private final OpenAiChatOptions defaultOptions;

    @Override
    public ChatResponse call(Prompt prompt) {
        // 1. 将 Spring AI 的 Prompt 转换为 OpenAI 的请求格式
        ChatCompletionRequest request = createRequest(prompt);

        // 2. 调用 REST API
        ResponseEntity<ChatCompletion> response =
            restClient.post()
                .uri("/v1/chat/completions")
                .body(request)
                .retrieve()
                .toEntity(ChatCompletion.class);

        // 3. 将 OpenAI 响应转换为 Spring AI 的 ChatResponse
        return convertResponse(response.getBody());
    }
}

Anthropic (Claude) 协议组件

Claude API 使用不同的字段命名和结构，Spring AI 在模型层完成转换。

public class AnthropicChatModel implements ChatModel {
    @Override
    public ChatResponse call(Prompt prompt) {
        // Claude 使用 "system" 作为顶层字段，而非 message 中的 role
        AnthropicRequest request = AnthropicRequest.builder()
            .model(defaultOptions.getModel())
            .system(extractSystemMessage(prompt))    // Claude 特殊处理
            .messages(convertToAnthropicMessages(prompt))  // 不包含 system
            .maxTokens(defaultOptions.getMaxTokens())
            .temperature(defaultOptions.getTemperature())
            .build();

        ResponseEntity<AnthropicResponse> response =
            restClient.post()
                .uri("/v1/messages")
                .header("x-api-key", apiKey)
                .header("anthropic-version", "2023-06-01")
                .body(request)
                .retrieve()
                .toEntity(AnthropicResponse.class);

        return convertResponse(response.getBody());
    }
}

Ollama 协议组件：本地模型部署

Ollama 提供与 OpenAI 部分兼容的 API，同时保留原生格式。

public class OllamaChatModel implements ChatModel {
    private final OllamaApi ollamaApi;

    @Override
    public ChatResponse call(Prompt prompt) {
        // Ollama 原生请求格式
        OllamaChatRequest request = OllamaChatRequest.builder()
            .model("llama3.2")                    // 本地模型名称
            .messages(convertMessages(prompt))     // 标准消息格式
            .stream(false)
            .options(Map.of(                        // Ollama 特有参数
                "temperature", 0.7,
                "num_ctx", 4096                     // 上下文窗口大小
            ))
            .build();

        // 直接访问本地服务，无需 API Key
        OllamaChatResponse response = restClient
            .post()
            .uri(baseUrl + "/api/chat")
            .body(request)
            .retrieve()
            .body(OllamaChatResponse.class);

        return convertResponse(response);
    }
}

底层调用原理：从 Prompt 到 HTTP 请求的全链路

同步调用完整时序

业务代码
    │ chatClient.prompt().user("...").call()
    ▼
DefaultChatClient
    │ 1. 组装 Prompt（system + user + options）
    │ 2. 执行 Advisor.before() 链
    ▼
具体 ChatModel (如 OpenAiChatModel)
    │ 3. Prompt → 供应商 Request DTO 转换
    │    - Message 按供应商格式序列化
    │    - ChatOptions 映射为供应商参数名
    ▼
RestClient (Spring 6)
    │ 4. 构建 HTTP 请求
    │    - 添加认证头 (Authorization: Bearer sk-xxx)
    │    - 设置 Content-Type: application/json
    ▼
HTTP Client (JDK HttpClient / Apache / OkHttp)
    │ 5. 建立 TCP 连接（可能复用 HTTP/2 连接池）
    │ 6. TLS 握手（HTTPS）
    │ 7. 发送 JSON 请求体
    ▼
模型供应商服务器
    │ 8. 接收请求 → 路由到推理集群
    │ 9. GPU 推理计算 → 逐 token 生成
    │ 10. 组装完整响应 → 返回 JSON
    ▼
逐层返回
    │ 11. HTTP Client 接收响应
    │ 12. RestClient 反序列化为 Response DTO
    │ 13. ChatModel 转换为 ChatResponse
    │ 14. Advisor.after() 处理
    │ 15. 返回给业务代码

流式响应的 SSE 机制

// 流式调用入口
Flux<String> stream = chatClient.prompt()
    .user("讲一个长篇故事")
    .stream()
    .content();  // 提取文本流

// 控制器层配合 SSE 输出
@GetMapping(value = "/chat/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<ServerSentEvent<String>> chatStream(@RequestParam String question) {
    return chatClient.prompt()
        .user(question)
        .stream()
        .content()
        .map(content -> ServerSentEvent.builder(content).build());
}

// Spring AI 内部：将 SSE 行解析为 ChatResponse 流
public Flux<ChatResponse> stream(Prompt prompt) {
    return webClient.post()
        .uri("/v1/chat/completions")
        .bodyValue(createRequest(prompt, true))  // stream=true
        .retrieve()
        .bodyToFlux(String.class)                 // 逐行接收 SSE
        .map(this::parseServerSentEvent)          // 解析 "data: {...}"
        .filter(Objects::nonNull)
        .map(this::convertChunkToChatResponse);   // 增量 → ChatResponse
}

Function Calling 的循环调用机制

// 定义可被 AI 调用的 Java 函数
public record StockPriceRequest(String symbol) {}

@Bean
public Function<StockPriceRequest, String> stockPriceFunction() {
    return request -> {
        // 调用真实股票 API
        return stockService.getRealtimePrice(request.symbol());
    };
}

// 调用时注册
chatClient.prompt()
    .user("查询腾讯和阿里巴巴的股价")
    .functions("stockPriceFunction")  // Spring AI 自动提取 Bean
    .call();

Function Calling 完整流程：

用户: "查询腾讯和阿里巴巴的股价"
    │
    ▼
第一次调用 LLM（带 tools 定义）
    │ 模型决定需要调用工具
    │ 返回：finish_reason="tool_calls"
    │       tool_calls=[{name:"stockPriceFunction", args:{"symbol":"0700.HK"}}]
    ▼
Spring AI 解析 Tool Calls
    │ 1. 根据 name 匹配到 stockPriceFunction Bean
    │ 2. 使用 Jackson 将 args JSON 反序列化为 StockPriceRequest
    │ 3. 执行 Java 方法 stockPriceFunction.apply(request)
    │ 4. 获取返回结果 "412.8 HKD"
    ▼
第二次调用 LLM（带工具结果）
    │ messages = [
    │   user("查询腾讯和阿里巴巴的股价"),
    │   assistant(tool_calls: [...]),           // AI 说要调用
    │   tool("412.8 HKD", tool_call_id: "...")   // 工具返回结果
    │ ]
    │
    ▼
模型生成最终回答
    "腾讯(0700.HK)当前股价为 412.8 港元。"

PostgreSQL vs MySQL：RAG 向量存储场景下的选型对比

核心结论：PostgreSQL 凭借 pgvector 扩展成为 AI 时代向量存储的事实标准，而 MySQL 在原生向量能力上几乎空白；但从通用业务数据角度，MySQL 在国内生态、运维成熟度上仍占优势，RAG 系统通常采用 MySQL 存业务 + PostgreSQL 存向量 的异构架构。

核心能力全维度对比

维度PostgreSQLMySQL
向量存储原生支持（pgvector 扩展，vector/halfvec 类型）不支持，需借助外部服务（如通过 MySQL 存 ID，ES/PG 存向量）
向量索引IVFFlat、HNSW、StreamingDiskANN无
相似度检索内置 <=>（余弦）、<->（欧氏）、<#>（内积）运算符无
JSON 支持JSONB 二进制存储，支持 GIN 索引、路径查询、部分更新JSON 文本存储，JSONB 8.0+ 但性能不如 PG
数组类型原生多维数组，向量化场景天然适配无原生数组，JSON 列模拟效率低
扩展生态丰富：pgvector、PostGIS、TimescaleDB、Citus较少，插件体系不如 PG 开放
GIS 地理信息PostGIS 业界最强Spatial 扩展功能较弱
全文检索内置全文检索 + 权重排序，可接 pg_searchFULLTEXT 索引，中文分词需额外配置
并发控制MVCC 实现更彻底，读写冲突少MVCC + 间隙锁，高并发写易死锁
SQL 标准兼容更接近 ANSI SQL，窗口函数、CTE 成熟8.0+ 大幅追赶，部分语法仍不兼容

为什么 MySQL 不支持原生向量

MySQL 架构限制：

1. 插件体系封闭
   - MySQL 的存储引擎接口（Handler API）对外暴露有限
   - 无法像 PostgreSQL 扩展那样注册新数据类型、新索引方法、新运算符

2. 索引类型固化
   - B-Tree、Hash、Fulltext、Spatial —— 均为 MySQL 内置
   - 不支持 IVFFlat、HNSW 这类近似最近邻（ANN）索引算法

3. 查询执行器限制
   - 无法注册自定义运算符（如 `<=>` 余弦相似度）
   - 排序操作只能基于标量值，无法基于向量距离函数

4. 行存储格式
   - InnoDB 行格式对变长大字段（如 4KB 向量）存储效率低
   - 页内存储限制导致大向量频繁产生页外存储（off-page）

@Service
public class HybridRagService {
    @Autowired
    private JdbcTemplate mysqlTemplate;      // MySQL：业务数据
    @Autowired
    private JdbcTemplate pgTemplate;         // PostgreSQL：向量数据

    public List<Document> search(String query, Long userId) {
        // 1. 从 MySQL 获取用户有权限访问的文档 ID 列表
        List<Long> allowedDocIds = mysqlTemplate.queryForList(
            "SELECT doc_id FROM user_doc_permission WHERE user_id = ?",
            Long.class, userId
        );

        // 2. 将查询向量化（调用 Embedding API）
        float[] queryVector = embeddingClient.embed(query).getOutput();

        // 3. 在 PostgreSQL 中执行向量检索 + ID 过滤
        String pgSql = """
            SELECT doc_id, content, embedding <=> ?::vector AS distance
            FROM document_vectors
            WHERE doc_id = ANY(?)
            ORDER BY embedding <=> ?::vector
            LIMIT 10
            """;

        return pgTemplate.query(pgSql, new DocumentMapper(),
            pgVector(queryVector),          // 查询向量
            pgArray(allowedDocIds),          // 权限过滤
            pgVector(queryVector)            // ORDER BY 用
        );
    }
}

向量相似度算法与存储查询技术方案

核心结论：RAG 的检索质量取决于相似度算法选择、向量索引结构和混合查询能力三者的协同；pgvector 的 HNSW 索引在精度与速度间取得最佳平衡，而生产环境需配合维度量化、分区剪枝和结构化过滤实现百万级毫秒响应。

向量相似度算法：三种核心度量方式

// 两个向量 a = [a1, a2, ..., an], b = [b1, b2, ..., bn]

// 1. 余弦相似度（Cosine Similarity）— 最常用，关注方向而非绝对大小
//    结果范围 [-1, 1]，RAG 中通常取 [0, 1]
similarity = (a · b) / (||a|| * ||b||)

//    特点：对向量长度不敏感，适合文本语义匹配
//    场景：问答检索、文档推荐
//    pgvector: embedding <=> query_vec  （运算符已自动归一化时）
//             或 1 - (embedding <=> query_vec) 计算相似度

// 2. 欧氏距离（L2 Distance）— 关注空间中的绝对距离
//    结果范围 [0, +∞)，越小越相似
distance = sqrt(Σ(ai - bi)²)

//    特点：对绝对数值敏感，适合有明确空间位置的嵌入
//    场景：图像特征检索、数值型 Embedding
//    pgvector: embedding <-> query_vec

// 3. 内积（Inner Product）— 直接点乘
//    结果范围 (-∞, +∞)，越大越相似
//    当向量已归一化时，内积 = 余弦相似度
nsimilarity = Σ(ai * bi)

//    特点：计算最快，无需开方和除法
//    场景：大规模实时检索、已归一化的 Embedding 模型
//    pgvector: embedding <#> query_vec
算法公式关注维度计算成本适用模型
余弦相似度cos(θ) = A·B / (|A||B|)方向一致性中（需归一化）OpenAI text-embedding-3、BGE
欧氏距离|A - B|₂空间绝对距离中CLIP 图像向量、数值 Embedding
内积A·B投影大小最低已归一化向量（如 Google text-multilingual）
汉明距离不同 bit 数二进制差异极低二值化向量（Local Sensitive Hash）

选型决策：

  • 文本语义检索 → 余弦相似度（绝大多数 Embedding 模型输出已归一化，此时内积等价于余弦）
  • 需要结合向量强度和方向 → 欧氏距离
  • 十亿级实时检索 → 内积 + 量化压缩

pgvector 索引原理：精确检索与近似检索

精确检索（无索引）          近似检索（ANN 索引）
      │                          │
      ▼                          ▼
逐行计算距离            预建索引结构，剪枝搜索空间
全表 O(n)               O(log n) ~ O(sqrt(n))
100% 精度               >95% 精度（可调）
适合 < 10 万条          适合 > 10 万条

IVFFlat：倒排文件索引

-- 创建 IVFFlat 索引
CREATE INDEX ON document_vectors
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 100);  -- 聚类中心数，通常 sqrt(总行数)
IVFFlat 原理：

1. 训练阶段（CREATE INDEX 时）
   - 对所有向量做 K-Means 聚类，分成 `lists` 个簇
   - 每个簇有一个中心点（centroid）
   - 构建倒排表：centroid → [属于该簇的向量 ID 列表]

        centroid 1 ──→ [vec_3, vec_7, vec_12, ...]
        centroid 2 ──→ [vec_1, vec_5, vec_9, ...]
        centroid 3 ──→ [vec_2, vec_4, vec_8, ...]
             ...

2. 查询阶段
   - 计算查询向量与所有 centroid 的距离
   - 选择最近的 `probes` 个簇（默认 1）
   - 只在这些簇内做精确距离计算
   - 返回 Top-K
参数作用推荐值
lists聚类中心数rows / 1000，通常 100~1000
probes查询时扫描的簇数lists / 10，用 SET ivfflat.probes = 10 调整
-- 查询前设置扫描簇数（精度 vs 速度权衡）
SET ivfflat.probes = 10;  -- 默认 1，增大提高召回率

SELECT * FROM document_vectors
ORDER BY embedding <=> '[0.1, 0.2, ...]'::vector
LIMIT 10;

IVFFlat 特点：

  • 构建速度快，内存占用低
  • 查询速度中等（依赖 probes 设置）
  • 适合静态或低频更新的数据集（插入新向量可能需要重新聚类）

HNSW：层次导航小世界图

-- 创建 HNSW 索引（推荐用于大多数 RAG 场景）
CREATE INDEX ON document_vectors
USING hnsw (embedding vector_cosine_ops)
WITH (
    m = 16,           -- 每个节点最大连接数
    ef_construction = 64   -- 构建时的搜索宽度
);
HNSW 原理：

多层图结构（类似跳表）：

Layer 2 (稀疏层):    o────────o
                           │
Layer 1 (中层):     o──────o──────o
                     │      │      │
Layer 0 (密集层):  o──o──o──o──o──o──o──o
                   所有向量作为节点，短边保证连通性

1. 构建阶段
   - 逐层插入向量
   - 每层中，新节点与 `m` 个最近邻居建立双向边
   - `ef_construction` 控制建图时的搜索质量

2. 查询阶段
   - 从最高层随机入口点开始
   - 贪心策略：每层找到局部最近点
   - 逐层下降，直到第 0 层精确搜索
   - 参数 `ef_search` 控制搜索宽度（默认 40）
参数作用推荐值
m节点最大邻居数8~32，向量维度高取大值
ef_construction建图搜索宽度64~200，越大索引质量越高、构建越慢
ef_search查询搜索宽度SET hnsw.ef_search = 100;，越大精度越高

HNSW 特点：

  • 查询速度极快（百万级 < 5ms）
  • 支持增量插入（无需重建索引）
  • 内存占用较高（需存储多层图）
  • RAG 场景首选索引

混合查询：向量相似度 + 结构化过滤

生产环境 RAG 必须支持权限过滤 + 元数据筛选 + 时间范围等结构化条件。

-- 场景：用户只能搜索自己部门的文档，且文档状态为 "published"

-- ❌ 错误做法：先向量检索，再应用层过滤
-- 问题：可能检索 100 条才发现 5 条有权限的

-- ✅ 正确做法：将结构化条件推入索引层

-- 1. pgvector 支持在 ORDER BY 前过滤
SELECT doc_id, title, content,
       embedding <=> :query_vec AS distance
FROM document_vectors
WHERE department_id = :dept_id      -- 结构化过滤
  AND status = 'published'
  AND created_at > '2024-01-01'
ORDER BY embedding <=> :query_vec
LIMIT 10;

-- 2. 创建复合索引加速结构化过滤
CREATE INDEX idx_doc_filter ON document_vectors(department_id, status, created_at);
// Spring AI 中的混合查询
SearchRequest request = SearchRequest.query(question)
    .withTopK(10)
    .withFilterExpression(
        // 类似 Spring Data 的查询表达式
        "department_id == 'tech' && status == 'published'"
    );

List<Document> results = vectorStore.similaritySearch(request);
混合查询执行计划：

结构化条件过滤          向量相似度排序
      │                       │
      ▼                       ▼
B-Tree/Hash 索引         HNSW 索引
（命中 1000 条）         （全局扫描）
      │                       │
      └──→ 取交集 ←──────────┘
              │
              ▼
        对交集做精确距离排序
              │
              ▼
           返回 Top-K

优化要点：

  • 高选择性条件（如 user_id = ? 只命中少量行）优先用 B-Tree 过滤
  • 低选择性条件（如 status = 'published' 命中 90%）不要单独索引，交给 HNSW 处理
  • 复合索引列顺序：等值查询列在前，范围查询列在后

RAG 完整实现示例（生产级）

@Service
public class ProductionRagService {
    @Autowired
    private ChatClient chatClient;
    @Autowired
    private VectorStore vectorStore;
    @Autowired
    private EmbeddingClient embeddingClient;

    // 带权限过滤、重排序、引用溯源的生产级 RAG
    public RagResult answer(String question, Long userId) {
        // 1. 权限过滤（只查用户可见的文档）
        List<String> allowedCategories = getUserCategories(userId);

        // 2. 多路召回：向量检索 + 关键词检索
        List<Document> vectorResults = vectorStore.similaritySearch(
            SearchRequest.query(question)
                .withTopK(20)
                .withFilterExpression(
                    "category in (" + String.join(",", allowedCategories) + ")"
                )
        );

        // 3. 重排序（Rerank）— 用轻量模型精排
        List<Document> reranked = reranker.rerank(question, vectorResults, 5);

        // 4. 构建带引用标记的上下文
        String context = IntStream.range(0, reranked.size())
            .mapToObj(i -> String.format("[%d] %s", i + 1, reranked.get(i).getContent()))
            .collect(Collectors.joining("\n\n"));

        // 5. 调用 LLM，要求标注引用来源
        String answer = chatClient.prompt()
            .system("""
                你是一个专业助手。基于参考资料回答问题。
                引用来源时使用 [1]、[2] 等标记。
                如果资料不足，明确说明。
                """)
            .user(String.format("问题：%s\n\n参考资料：\n%s", question, context))
            .call()
            .content();

        return new RagResult(answer, reranked);
    }
}

实现原理与关键机制

Spring Boot 自动配置机制

@Configuration(proxyBeanMethods = false)
@ConditionalOnClass(ChatModel.class)
@EnableConfigurationProperties(OpenAiChatProperties.class)
public class OpenAiAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean  // 用户未自定义时才创建
    public OpenAiApi openAiApi(OpenAiConnectionProperties connProps) {
        // 创建 RestClient，配置 baseUrl、API Key 拦截器
        RestClient restClient = RestClient.builder()
            .baseUrl(connProps.getBaseUrl())
            .requestInterceptor((req, body, exec) -> {
                req.getHeaders().setBearerAuth(connProps.getApiKey());
                return exec.execute(req, body);
            })
            .build();
        return new OpenAiApi(restClient);
    }

    @Bean
    @ConditionalOnMissingBean
    public OpenAiChatModel openAiChatModel(OpenAiApi api, OpenAiChatOptions options) {
        return new OpenAiChatModel(api, options);
    }

    @Bean
    @ConditionalOnMissingBean(ChatClient.class)  // 默认 ChatClient
    public ChatClient chatClient(ChatModel chatModel) {
        return ChatClient.create(chatModel);
    }
}

常见问题与误区

问题1：流式响应中 Function Calling 失效

根因：部分模型在流式模式下不返回 tool_calls，而是将函数调用意图编码在内容中 解决：关闭流式模式处理 Function Calling，或升级到支持流式工具调用的模型版本

问题2：Token 消耗远超预期

根因：Prompt 中的系统消息和历史对话占用大量 token，加上 RAG 注入的长文档 解决：使用 ChatClient 的 Token 计数 advisor 监控，合理设置 maxTokens，对历史消息做截断或摘要

问题3：多模型切换时代码需要大量修改

根因：不同模型支持的参数不同（如 Claude 没有 topP 的某些取值） 解决：使用 ChatOptions 的通用属性，避免使用供应商特有参数；通过配置化切换模型实现

问题4：本地 Ollama 模型与云端模型行为不一致

根因：本地模型参数量小、量化精度低，指令遵循能力弱 解决：为本地模型设计更详细的 system prompt，降低任务复杂度，或选择更大的本地模型（如 70B）
