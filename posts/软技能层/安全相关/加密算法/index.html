常见算法与场景

核心结论：不同密码学算法有各自的适用场景和权衡点，选型时需综合考虑安全性、性能、合规性和生态系统成熟度。

密码学算法选型需匹配场景的安全需求和性能约束，避免过度设计或不足

对称加密算法选型

场景推荐算法原因
通用数据加密AES-256-GCM安全、支持认证加密、硬件加速
国内合规系统SM4国密要求
低功耗IoT设备AES-128计算量小，安全性足够
文件系统加密AES-XTS针对块设备优化
// AES-GCM 加密示例
Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
SecretKeySpec keySpec = new SecretKeySpec(key, "AES");
cipher.init(Cipher.ENCRYPT_MODE, keySpec, new GCMParameterSpec(128, iv));
byte[] encrypted = cipher.doFinal(plainText);

非对称加密算法选型

场景推荐算法原因
HTTPS/TLS证书RSA-2048 / ECDSA P-256广泛兼容
移动端API签名ECDSA签名短、验证快
国内金融系统SM2国密合规
区块链/数字货币ECDSA / Ed25519签名短小高效

哈希算法选型

场景推荐算法原因
密码存储bcrypt / scrypt / Argon2计算慢可防暴力破解
数据完整性校验SHA-256快速且安全
数字签名SHA-256 + RSA/ECDSA标准组合
国内合规SM3国密要求

典型组合场景

场景组合方式说明
HTTPS通信RSA/ECDHE + AES-GCM + SHA-256密钥交换 + 对称加密 + 完整性校验
JWT签名HS256(RSA) / ES256(ECDSA)Header.Payload 用私钥签名
文件加密AES-256 + RSA随机AES密钥加密文件，RSA加密AES密钥
数字证书私钥签名 + 公钥验签 + 证书链CA用私钥签发，浏览器用CA公钥验证

国密算法体系

算法类型用途替代的国际算法
SM2非对称加密加密、签名、密钥交换RSA、ECDSA
SM3哈希数据摘要、签名SHA-256
SM4对称加密数据加密AES
SM9标识密码基于标识的加密和签名IBE方案

常见问题与误区

问题1：密钥长度过短

根因：使用1024位RSA或128位以下AES以追求性能 解决：RSA至少2048位，AES至少128位，密码学安全不能妥协

问题2：ECB模式被误认为安全

根因：AES-ECB相同明文产生相同密文，会泄露模式 解决：使用GCM、CBC（需IV）等模式，禁用ECB
