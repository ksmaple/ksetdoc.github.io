设计原则（SOLID+1）

核心结论：SOLID原则是面向对象设计的基石，通过单一职责、开闭、里氏替换、接口隔离、依赖倒置五个原则降低耦合提升内聚，加上组合复用原则形成完整的设计指导体系。

SOLID+1原则从职责划分、扩展方式、继承约束、接口粒度、依赖方向、复用手段六个维度指导面向对象设计

单一职责原则（SRP）

❌ 一个类处理多个职责
class UserService {
    void register() {}      // 用户注册
    void sendEmail() {}     // 发送邮件
    void generateReport() {} // 生成报表
}

✅ 职责分离
class UserService { void register() {} }
class EmailService { void sendEmail() {} }
class ReportService { void generateReport() {} }
要点说明
定义一个类只负责一项职责
判断标准类变更的原因是否只有一个
违反症状类代码行数过多、修改影响多个功能

开闭原则（OCP）

❌ 新增折扣类型需要修改原有代码
if (type.equals("VIP")) { price * 0.8; }
else if (type.equals("SVIP")) { price * 0.6; }

✅ 通过扩展实现
interface DiscountStrategy { double calculate(double price); }
class VIPDiscount implements DiscountStrategy { ... }
class SVIPDiscount implements DiscountStrategy { ... }
要点说明
定义对扩展开放，对修改关闭
实现方式抽象 + 多态
判断标准新增功能是否只需新增代码

里氏替换原则（LSP）

❌ 子类改变了父类的行为约定
class Rectangle { void setWidth(int w); void setHeight(int h); }
class Square extends Rectangle {  // 正方形不是长方形的正确子类
    void setWidth(int w) { width = height = w; }  // 破坏了setHeight的语义
}

✅ 正例：子类不改变父类契约
class Bird { void fly() {} }
class Sparrow extends Bird {}      // 麻雀会飞，符合LSP
// 企鹅不会飞，不应继承Bird，应继承更抽象的Animal
要点说明
定义子类必须能够替换父类而不影响程序正确性
判断标准继承是否合理（is-a关系）
违反症状子类重写方法抛出异常或改变语义

接口隔离原则（ISP）

❌ 胖接口强制实现不必要的方法
interface Worker { void work(); void eat(); }
class Robot implements Worker {  // 机器人不需要eat
    void eat() { throw new UnsupportedOperationException(); }
}

✅ 接口拆分
interface Workable { void work(); }
interface Eatable { void eat(); }
class Robot implements Workable {}
class Human implements Workable, Eatable {}
要点说明
定义客户端不应依赖它不使用的接口
判断标准接口的实现类是否都需要所有方法

依赖倒置原则（DIP）

❌ 高层依赖低层具体实现
class OrderService {
    private MySqlRepository repository = new MySqlRepository();  // 直接依赖
}

✅ 依赖抽象
class OrderService {
    private OrderRepository repository;  // 依赖接口
    OrderService(OrderRepository repo) { this.repository = repo; }  // 注入
}
要点说明
定义高层模块不依赖低层模块，两者都依赖抽象
实现方式依赖注入（构造器/Setter/接口）

组合复用原则（CRP）

❌ 通过继承复用（白箱复用，耦合高）
class Vehicle { void move() {} }
class Car extends Vehicle {}  // 继承是is-a关系

✅ 通过组合复用（黑箱复用，耦合低）
class Engine { void start() {} }
class Car {
    private Engine engine;  // 组合是has-a关系
    void move() { engine.start(); }
}
复用方式关系耦合度适用场景
继承is-a高真正的泛化关系
组合has-a低大多数场景首选

常见问题与误区

问题1：过度设计，所有场景都套用原则

根因：为使用设计模式而使用，增加不必要的抽象 解决：先满足需求，在确实出现痛点时再重构，YAGNI原则

问题2：将SOLID视为教条而非指导

根因：僵化理解原则，忽视实际业务复杂度 解决：原则是方向而非绝对规则，允许合理权衡
