云服务实践

核心结论：云服务的核心价值是将基础设施运维转化为可配置的服务，在实践中需平衡便利性与成本，掌握云厂商提供的托管能力可以显著减少重复建设，但避免被厂商锁定需要保留抽象层。

云服务实践通过合理选型、成本控制和抽象封装，在享受托管便利的同时保持架构灵活性

网关插件实践

场景：API网关需要统一鉴权、限流、日志

云厂商方案：
├── 阿里云API网关
│   ├── 插件：JWT鉴权、流量控制、参数校验
│   └── 后端：直接路由到VPC内的ECS/容器
│
└── AWS API Gateway + Lambda Authorizer
    ├── Lambda函数做自定义鉴权逻辑
    └── 集成WAF做防护

自研 vs 托管权衡：
- 托管网关：配置即使用，运维成本低
- 自研网关（如Spring Cloud Gateway）：灵活度高，但需要自己维护高可用

成本控制经验

策略说明效果
包年包月+按量混合基线用包年，峰值用按量节省30-50%
资源标签管理按项目/环境打标签清晰成本归属
闲置资源清理定期巡检释放不用的ECS、磁盘避免隐形浪费
存储分层热数据SSD、冷数据归档存储成本降70%

多云策略

策略说明风险
单一云深度使用一家云厂商用透厂商锁定风险
多云容灾核心系统双云部署成本高、复杂度大
云无关抽象上层代码不依赖云SDK需要自研适配层
降低锁定风险的做法：
- 对象存储用标准S3协议
- 数据库用开源产品（MySQL/Redis）
- 容器化部署便于迁移
- 配置管理层封装云厂商差异

常见问题与误区

问题1：云服务默认高可用，忽视跨区域容灾

根因：误认为单区域内多可用区就是完整容灾 解决：核心系统规划跨区域/跨云容灾，定期演练切换

问题2：按量付费不注意用量，月底账单超预期

根因：缺乏预算告警和用量监控 解决：设置用量阈值告警，定期Review资源使用
