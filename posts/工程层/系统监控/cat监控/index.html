CAT分布式监控

核心结论：CAT通过客户端埋点采集监控数据，通过消息树组织调用链，通过服务端聚合分析，实现从应用性能到业务指标的全链路监控，埋点规范和数据采样是落地关键。

CAT通过客户端埋点采集、消息树组织、服务端聚合，实现全链路监控和实时告警

核心概念：Transaction、Event、Heartbeat、Metric

类型说明示例
Transaction有执行时间的嵌套结构SQL执行、RPC调用、方法执行
Event无执行时间的离散事件异常、日志、用户行为
Heartbeat周期性心跳JVM状态、线程数、系统负载
Metric指标统计QPS、RT、成功率
消息树（Message Tree）结构：

Root Transaction: URL-/api/order/create
├── Transaction: Service-createOrder (耗时50ms)
│   ├── Transaction: DAO-insertOrder (耗时10ms)
│   │   └── Event: SQL (INSERT INTO order ...)
│   ├── Transaction: Cache-getUser (耗时5ms)
│   └── Transaction: RPC-callPayment (耗时30ms)
│       └── Event: Request (payId=12345)
├── Event: OrderCreated (orderId=10001)
└── Metric: order.create (count=1)

Java埋点示例

// Transaction埋点
Transaction t = Cat.newTransaction("Service", "createOrder");
try {
    Order order = orderService.create(request);
    Cat.logEvent("OrderCreated", String.valueOf(order.getId()));
    t.setStatus(Transaction.SUCCESS);
} catch (Exception e) {
    Cat.logError(e);
    t.setStatus(e);
} finally {
    t.complete();
}

// 简化方式（使用AOP或注解）
@CatTransaction(type = "Service", name = "createOrder")
public Order createOrder(OrderRequest request) {
    return orderService.create(request);
}

实际应用中的决策与权衡

场景埋点策略原因
核心接口全量Transaction精确监控RT和成功率
高频调用采样（1%）减少性能开销
异常处理必报Event不遗漏错误
业务指标Metric聚合实时大盘

常见问题与误区

问题1：埋点过多影响性能

根因：每个埋点都有创建和序列化开销 解决：核心链路全量、非核心采样、异步上报

问题2：异步线程丢失链路上下文

根因：ThreadLocal传递不到子线程 解决：使用CatContext传递上下文、包装Callable/Runnable
