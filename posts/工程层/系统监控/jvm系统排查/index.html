JVM与系统问题排查

核心结论：JVM问题排查遵循"监控发现→日志定位→工具分析→验证修复"的流程，CPU高通常与热点代码或死循环有关，内存泄漏需通过堆dump分析对象引用链，GC问题需结合日志和可视化工具判断。

JVM问题排查遵循从监控到工具分析的系统化流程，CPU、内存、GC是三大核心排查方向

CPU高排查流程

1. 定位高CPU进程
   top / htop → 找到PID

2. 定位高CPU线程
   top -Hp PID → 找到TID
   printf "%x\n" TID → 转换为十六进制

3. 查看线程堆栈
   jstack PID | grep TID_HEX -A 30
   
4. 分析热点代码
   - 某方法执行时间长 → 优化算法
   - 频繁GC → 调整GC参数或内存
   - 死循环 → 修复代码

内存泄漏排查

1. 发现内存增长
   jstat -gcutil PID 1000 → 观察老年代增长
   
2. 生成堆dump
   jmap -dump:format=b,file=heap.hprof PID
   或 -XX:+HeapDumpOnOutOfMemoryError

3. 分析dump文件
   MAT（Eclipse Memory Analyzer）
   - Dominator Tree：查看大对象
   - Path to GC Roots：查看引用链
   - Leak Suspects：自动泄漏报告

常见泄漏源：
- 静态集合缓存未清理
- ThreadLocal未remove
- 未关闭的资源（流、连接）
- 监听器未注销

GC问题排查

1. 启用GC日志
   -Xlog:gc*:file=gc.log:time:filecount=5,filesize=100m

2. 分析GC日志
   - GCeasy（在线工具）
   - gceasy.io
   - GCViewer（本地工具）

3. 关键指标
   - GC频率：多久一次
   - GC耗时：STW时间
   - 内存回收率：每次回收多少
   - 晋升速率：对象进入老年代速度

调优方向：
- 堆太小 → 频繁GC → 增大堆
- 年轻代太小 → 对象过早晋升 → 增大新生代
- 大对象多 → 分配失败 → 调整大对象阈值

常用排查命令速查

问题命令说明
CPU高top + jstack定位线程和热点代码
内存占用jmap -heap堆内存分布
内存泄漏jmap -dump + MAT分析对象引用
GC情况jstat -gcutil实时GC统计
线程死锁jstack -l查看死锁信息
类加载jcmd PID VM.classloader_stats类加载器统计
系统调用strace -p PID跟踪系统调用
网络连接netstat -anp / ss -s连接状态统计
