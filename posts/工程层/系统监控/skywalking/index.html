SkyWalking链路追踪

核心结论：SkyWalking通过探针自动采集调用链数据，通过OAP服务端分析聚合，通过UI可视化展示，支持无侵入式接入，是云原生环境下应用性能监控的标准方案。

SkyWalking通过探针自动采集、OAP分析、UI可视化，实现无侵入式全链路追踪

核心架构：探针-OAP-存储-UI

┌──────────┐      ┌─────────┐      ┌──────────┐      ┌────────┐
│  Agent   │ ───→ │   OAP   │ ───→ │ Storage  │ ←──→ │   UI   │
│ (探针)    │ gRPC │ (分析)   │      │ (ES/H2)  │      │ (可视化)│
└──────────┘      └─────────┘      └──────────┘      └────────┘

探针类型：
- Java Agent：字节码增强，无侵入
- Sidecar：Envoy/Service Mesh集成
- 语言探针：Go/Node.js/Python等

自动采集的Span类型：

Span类型说明
Entry Span请求入口（HTTP/RPC接收）
Exit Span请求出口（HTTP/RPC发送）
Local Span本地方法

Java Agent接入

# JVM启动参数
-javaagent:/path/to/skywalking-agent.jar
-DSW_AGENT_NAME=order-service
-DSW_AGENT_COLLECTOR_BACKEND_SERVICES=127.0.0.1:11800

自动拦截的组件：

  • HTTP：Tomcat/Spring MVC/Spring WebFlux
  • RPC：Dubbo/gRPC/Feign
  • DB：JDBC/MyBatis/JPA
  • MQ：RocketMQ/Kafka/RabbitMQ
  • Cache：Redis/Jedis/Lettuce

实际应用中的决策与权衡

场景采样率原因
开发环境100%详细排查
测试环境50%平衡
生产环境1-10%性能优先
核心接口100%关键业务保障

常见问题与误区

问题1：Agent导致应用启动慢

根因：字节码增强需遍历和修改类 解决：排除不必要的插件、使用缓存

问题2：链路数据存储膨胀

根因：生产环境数据量巨大 解决：设置TTL、采样率、只保留错误链路详情
