公会/主播/家族系统

核心结论：公会/主播/家族系统通过层级关系链管理组织结构和收益分配，核心挑战是层级关系的灵活查询、收益分成的精确计算，以及结算数据的幂等和可追溯。

公会系统通过层级关系管理组织，通过配置化分成比例实现收益分配

层级模型

平台
├── 公会A（分成比例30%）
│   ├── 主播A1（分成70%）
│   ├── 主播A2（分成70%）
│   └── 家族F1
│       ├── 成员M1
│       └── 成员M2
├── 公会B
│   └── 主播B1

收益分配（用户打赏100元）：
  平台：100 * 50% = 50元
  公会：100 * 30% = 30元
  主播：100 * 20% = 20元

数据模型设计

-- 组织表
CREATE TABLE organization (
    id BIGINT PRIMARY KEY,
    name VARCHAR(100),
    type ENUM('guild', 'family'),  -- 公会/家族
    parent_id BIGINT,              -- 上级组织
    divide_rate DECIMAL(5,2),      -- 平台分成比例
    status TINYINT
);

-- 成员关系表
CREATE TABLE member_relation (
    id BIGINT PRIMARY KEY,
    org_id BIGINT,
    user_id BIGINT,
    role ENUM('owner', 'admin', 'member'),
    divide_rate DECIMAL(5,2),      -- 个人分成比例
    join_time DATETIME
);

常见问题与误区

问题1：层级关系查询性能差

根因：递归查询层级效率低 解决：路径枚举（path字段）、闭包表、或限制层级深度

问题2：收益分成计算不一致

根因：浮点数精度、并发修改分成比例 解决：使用整数分（100元=10000分）、结算时锁定比例快照
