第三方系统交互设计

核心结论：与第三方系统交互的核心挑战是网络不可靠和接口差异，通过统一接口层封装异构协议，通过幂等和重试保障可靠性，通过熔断降级保障自身稳定性。

第三方交互通过统一封装、幂等重试、熔断降级三大机制保障可靠性

设计原则

原则说明实现
统一封装屏蔽第三方差异适配器模式，统一出入参
幂等设计重复调用无副作用业务订单号唯一性
超时重试网络抖动容错指数退避，限制重试次数
熔断降级第三方故障自我保护失败率阈值触发熔断
异步解耦降低耦合和延迟MQ中转

接口层架构

业务层
  ↓ 调用
统一接口层（Adapter）
  ├── 微信支付Adapter
  ├── 支付宝Adapter
  ├── 短信服务商Adapter
  └── 推送服务商Adapter

统一接口层职责：
  - 入参转换（内部标准 → 第三方格式）
  - 出参转换（第三方格式 → 内部标准）
  - 签名/验签
  - 超时/重试/熔断
  - 日志记录

幂等与重试

// 带幂等的HTTP调用
public Result callWithRetry(String url, Request request) {
    String idempotencyKey = generateKey(request);
    
    for (int i = 0; i < MAX_RETRY; i++) {
        try {
            return httpClient.call(url, request, idempotencyKey, TIMEOUT);
        } catch (TimeoutException e) {
            if (i == MAX_RETRY - 1) throw e;
            Thread.sleep((long) (Math.pow(2, i) * 100));  // 指数退避
        }
    }
    throw new RuntimeException("Should not reach here");
}

常见问题与误区

问题1：第三方接口变更导致故障

根因：直接依赖第三方数据结构 解决：适配器层做字段映射，变更只改适配器

问题2：第三方超时拖垮自身服务

根因：同步阻塞等待，线程池耗尽 解决：熔断降级（Hystrix/Sentinel）、异步调用、设置短超时
