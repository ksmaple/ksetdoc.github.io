支付业务

核心结论：支付业务的核心挑战是资金安全和一致性，通过唯一订单号实现幂等，通过状态机管理支付生命周期，通过定时对账发现并修正差异，通过风控系统拦截可疑交易。

支付系统通过状态机、幂等设计和日终对账，保障资金安全和数据一致性

支付核心流程

支付状态机：

INIT → 创建支付单
  ↓
PENDING → 用户确认支付，调起渠道
  ↓
PROCESSING → 渠道处理中
  ↓
SUCCESS / FAIL / CLOSED

状态不可逆：
  SUCCESS不可变为FAIL
  需退款走独立退款流程

幂等设计

支付请求：
  业务订单号 + 支付方式 + 金额 → 生成唯一支付流水号
  
重复请求处理：
  - 同一流水号重复请求 → 返回原结果
  - 不同流水号但同一订单 → 拒绝（防重复支付）
  
实现：
  数据库唯一索引（支付流水号）
  或 Redis SETNX 分布式锁

对账机制

日终对账流程：

1. 下载渠道对账单（微信/支付宝/银联）
2. 我方支付记录与渠道记录逐笔核对
   - 金额一致 + 状态一致 → 正常
   - 我方有，渠道无 → 渠道漏单，人工处理
   - 渠道有，我方无 → 我方漏单，补单
   - 金额不一致 → 异常，人工处理
3. 生成差异报表，人工确认
4. 差异处理（补单/退款/调账）

风控策略

策略说明
频次控制同一账户短时间多次支付
金额限制单笔/单日限额
黑名单已知风险账户/设备/IP
行为分析异常登录地、异常设备
3D验证大额支付需额外验证

常见问题与误区

问题1：支付成功但订单状态未更新

根因：渠道回调失败或处理异常 解决：主动轮询 + 对账兜底

问题2：重复支付

根因：幂等控制失效 解决：支付前检查订单状态，已支付直接返回成功
