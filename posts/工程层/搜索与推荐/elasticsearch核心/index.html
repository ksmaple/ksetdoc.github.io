Elasticsearch核心

核心结论：Elasticsearch基于Lucene实现分布式搜索，通过倒排索引加速全文检索，通过分片和副本实现水平扩展和高可用，理解索引、分片、映射和查询DSL是使用ES的基础。

Elasticsearch基于Lucene构建，通过分片分布式存储，通过倒排索引实现高性能全文搜索

核心概念：索引、文档、映射、分片

概念说明类比关系型数据库
Index索引，文档的集合Database
Document文档，一条JSON数据Row
Mapping映射，字段类型定义Schema
Type类型（7.0+已废弃）Table
Shard分片，数据的最小单元Partition
Replica副本，分片的复制Replica
索引结构：

Index: products
├── Primary Shard 0 (Node-1)
│   └── Replica 0 (Node-2)
├── Primary Shard 1 (Node-2)
│   └── Replica 1 (Node-3)
├── Primary Shard 2 (Node-3)
│   └── Replica 2 (Node-1)
└── Primary Shard 3 (Node-1)
    └── Replica 3 (Node-2)

路由：shard = hash(routing) % number_of_primary_shards
routing默认是文档_id

倒排索引与文本分析

文本分析流程（Analysis）：

原始文本: "The quick brown foxes jumped!"
    ↓
Character Filter: 去除HTML、替换字符
    ↓
Tokenizer: 分词 → [the, quick, brown, foxes, jumped]
    ↓
Token Filter: 
  - Lowercase: [the, quick, brown, foxes, jumped]
  - Stop words: [quick, brown, foxes, jumped]（去除the）
  - Stemming: [quick, brown, fox, jump]（foxes→fox, jumped→jump）
    ↓
存入倒排索引

常用分析器：

分析器特点适用场景
standard标准分词，按词切分通用英文
ik_smartIK智能分词，粗粒度中文搜索
ik_max_wordIK细粒度分词中文索引（最大化召回）
keyword不分词，原样存储精确匹配
whitespace按空格切分简单英文

查询DSL：从简单到复杂

// Match查询（全文搜索，会分词）
{
  "query": {
    "match": {
      "title": "苹果手机"
    }
  }
}

// Term查询（精确匹配，不分词）
{
  "query": {
    "term": {
      "status": "published"
    }
  }
}

// Bool组合查询
{
  "query": {
    "bool": {
      "must": [
        { "match": { "title": "手机" } },
        { "range": { "price": { "lte": 5000 } } }
      ],
      "must_not": [
        { "term": { "status": "deleted" } }
      ],
      "filter": [
        { "term": { "category": "electronics" } }
      ]
    }
  }
}

// Aggregation聚合
{
  "aggs": {
    "by_category": {
      "terms": { "field": "category" }
    },
    "avg_price": {
      "avg": { "field": "price" }
    }
  }
}

实现原理与关键机制

分片分配与路由

文档写入：
  1. 计算路由: shard = hash(_id) % num_primary_shards
  2. 请求发送到该shard的主分片所在节点
  3. 主分片写入，并行复制到副本分片
  4. 主分片等待副本确认（根据wait_for_active_shards配置）
  5. 返回成功响应

查询流程：
  1. 请求发送到任意节点（协调节点）
  2. 协调节点转发到所有相关分片（主或副）
  3. 各分片本地搜索，返回TopN结果
  4. 协调节点归并结果，返回最终TopN

近实时搜索（Near Realtime）

写入文档 → 内存缓冲区（Index Buffer）
    ↓ 每秒refresh
段（Segment）生成 → 文件系统缓存（可搜索）
    ↓ 30分钟或缓冲区满flush
磁盘持久化（FSync）

refresh: 使新文档可搜索（默认1秒）
flush: 将段写入磁盘，清空事务日志

实际应用中的决策与权衡

场景方案原因
全文搜索Match + 中文分词器召回率高
精确过滤Term + Filter可缓存，性能高
日志分析时间索引 + Terms聚合时序特性
高并发查询增加副本数分散读压力
大数据量增加主分片数分散写压力

常见问题与误区

问题1：深度分页性能差

根因：from+size越大，协调节点归并压力越大 解决：使用search_after或scroll，限制最大分页深度

问题2：Mapping爆炸（Mapping Explosion）

根因：字段过多或字段名动态变化 解决：设置index.mapping.total_fields.limit，使用嵌套对象或扁平化字段

问题3：脑裂（Split Brain）

根因：网络分区导致多个节点认为自己是Master 解决：设置discovery.zen.minimum_master_nodes为多数（ES 7.0+已内置）
