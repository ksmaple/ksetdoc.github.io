MySQL存储引擎

核心结论：InnoDB通过聚簇索引组织数据，通过MVCC实现无锁读，通过行锁和间隙锁实现并发控制，是MySQL生产环境的事实标准；MyISAM已逐步被InnoDB取代，仅在特殊读多写少场景考虑。

InnoDB通过聚簇索引、MVCC、行锁三大机制实现高并发事务安全，是MySQL生产环境的首选存储引擎

InnoDB核心特性

特性实现优势
聚簇索引主键索引即数据主键查询快，范围查询高效
MVCCundo log + 事务ID读不阻塞写，写不阻塞读
行级锁记录锁 + 间隙锁并发度高
外键支持级联操作数据完整性
崩溃恢复Redo Log + Double Write数据不丢失
缓冲池内存缓存数据和索引减少磁盘IO

MVCC：多版本并发控制

MVCC核心：每行数据有两个隐藏列

DB_TRX_ID（6字节）：最后修改该行的事务ID
DB_ROLL_PTR（7字节）：指向undo log的回滚指针

Read View（快照）：事务开始时创建的一致性视图
  - creator_trx_id：创建该视图的事务ID
  - up_limit_id：活跃事务最小ID
  - low_limit_id：即将分配的下一个事务ID
  - trx_ids：活跃事务ID列表

可见性判断：
  1. DB_TRX_ID == creator_trx_id：可见（自己修改的）
  2. DB_TRX_ID < up_limit_id：可见（事务已提交）
  3. DB_TRX_ID >= low_limit_id：不可见（事务未开始）
  4. DB_TRX_ID在trx_ids中：不可见（事务活跃未提交）
  5. 其他情况：可见

InnoDB的三种行记录格式：

格式特点适用场景
Compact变长字段长度列表 + NULL标志位通用场景
Dynamic大字段只存指针，行更短默认格式，推荐
Compressed行数据压缩存储节省空间，CPU换IO

锁机制：行锁 + 间隙锁 + 临键锁

InnoDB锁类型：

行级锁：
  - 记录锁（Record Lock）：锁定单行记录
  - 间隙锁（Gap Lock）：锁定范围，防止幻读
  - 临键锁（Next-Key Lock）：记录锁 + 间隙锁

锁模式：
  - 共享锁（S Lock）：SELECT ... LOCK IN SHARE MODE
  - 排他锁（X Lock）：SELECT ... FOR UPDATE / INSERT / UPDATE / DELETE

意向锁（表级）：
  - 意向共享锁（IS）：事务有意向获取行级S锁
  - 意向排他锁（IX）：事务有意向获取行级X锁

实际应用中的决策与权衡

场景选择原因
通用OLTPInnoDB事务、行锁、崩溃恢复
只读分析（不更新）MyISAM / 列存表锁简单、压缩率高
日志/时序数据InnoDB + 分区范围查询优化
内存表MEMORY速度快，但数据不持久

常见问题与误区

问题1：RR隔离级别下仍有幻读

根因：InnoDB的RR通过间隙锁解决幻读，但快照读（普通SELECT）不受间隙锁保护 解决：使用SELECT … FOR UPDATE（当前读）加临键锁

问题2：死锁频繁

根因：事务顺序不一致、持有锁时间过长 解决：固定加锁顺序、减少事务范围、使用死锁检测（innodb_deadlock_detect）

问题3：主键用UUID导致性能差

根因：UUID随机插入导致页分裂、聚簇索引碎片化 解决：使用自增ID或雪花算法（趋势递增）
