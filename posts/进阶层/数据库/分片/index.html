分库分表

核心结论：分库分表通过水平拆分（数据分散）和垂直拆分（职责分离）突破单库性能瓶颈，但引入了分布式事务、跨分片查询、数据迁移等复杂度，应在单库优化无法满足需求时作为最后手段引入。

分库分表通过拆分策略将数据分布到多个节点，以牺牲部分查询灵活性换取存储和性能扩展性

拆分策略

策略方式适用场景
垂直分库按业务拆分（用户库/订单库/商品库）业务边界清晰，耦合低
垂直分表大字段拆分（主表+扩展表）单表字段过多，冷热数据分离
水平分库同表数据分散到不同库单库数据量过大
水平分表同表数据分散到不同表单表数据量过大，同一实例内
垂直分库                    水平分表
user_db                     db0: order_0, order_2
├── user                    db1: order_1, order_3
└── user_profile

order_db
└── order

分片键选择

方案分片键优点缺点
哈希分片user_id % 4数据均匀分布范围查询需扫描全部分片
范围分片create_time支持范围查询热点问题（新数据集中在最新分片）
混合分片时间+哈希兼顾均匀与范围实现复杂
// ShardingSphere哈希分片配置
sharding:
  tables:
    t_order:
      actual-data-nodes: ds${0..1}.t_order_${0..3}
      table-strategy:
        standard:
          sharding-column: order_id
          sharding-algorithm-name: order-hash
  sharding-algorithms:
    order-hash:
      type: INLINE
      props:
        algorithm-expression: t_order_${order_id % 4}

分布式ID

方案优点缺点
数据库自增简单性能瓶颈，不适合分片
Snowflake高性能，趋势递增时钟回拨问题
号段模式批量获取，性能高依赖数据库
UUID无中心无序，占用空间大
// Snowflake实现（位运算）
// 64位 = 1位符号 + 41位时间戳 + 10位机器ID + 12位序列号

public long nextId() {
    long timestamp = System.currentTimeMillis();
    return ((timestamp - EPOCH) << TIMESTAMP_LEFT)
         | (datacenterId << DATACENTER_LEFT)
         | (workerId << WORKER_LEFT)
         | sequence;
}

ShardingSphere集成

spring:
  shardingsphere:
    datasource:
      names: ds0, ds1
      ds0:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        jdbc-url: jdbc:mysql://localhost:3306/db0
      ds1:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        jdbc-url: jdbc:mysql://localhost:3306/db1
    rules:
      sharding:
        tables:
          t_order:
            actual-data-nodes: ds${0..1}.t_order_${0..3}
            key-generate-strategy:
              column: order_id
              key-generator-name: snowflake

分片后的挑战

问题说明解决方案
跨分片JOIN无法直接JOIN不同分片的数据全局表广播、应用层聚合、宽表冗余
跨分片排序分页ORDER BY + LIMIT需全局排序应用层归并、限制深度分页
分布式事务跨库事务无法保证ACIDSeata AT模式、最终一致性
数据迁移扩容需重新分片双写过渡、增量同步

分页优化

// 问题：limit 1000000, 10 需全局扫描大量数据
// 优化1：限制分页深度
if (pageNum > 1000) {
    throw new BizException("不支持深分页，请使用搜索条件缩小范围");
}

// 优化2：游标分页（基于排序键）
SELECT * FROM t_order 
WHERE create_time < #{lastTime} 
ORDER BY create_time DESC 
LIMIT 10;

常见问题与误区

问题1：过早引入分库分表

根因：单表几百万数据就开始拆分，增加不必要复杂度 解决：先尝试索引优化、读写分离、归档历史数据，单表千万级以上再考虑拆分

问题2：分片键选择不当导致数据倾斜

根因：按月份分片但某月数据量激增 解决：预先评估数据分布，哈希分片+动态扩容，定期监控各分片数据量
