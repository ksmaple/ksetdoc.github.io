MySQL瓶颈与扩展

核心结论：MySQL单库的瓶颈在CPU、内存、磁盘IO和连接数四个维度，扩展路径从读写分离到分库分表逐步递进，每种方案都在一致性、复杂度和成本之间做权衡。

MySQL单库存在CPU、内存、磁盘、连接数四重瓶颈，扩展路径为读写分离→垂直拆分→水平分表→分库分表

单库瓶颈分析

瓶颈表现临时解决根本解决
CPU慢查询多、计算密集优化SQL、加缓存读写分离、分库
内存缓冲池不足、频繁换页增大内存分库分散压力
磁盘IOiowait高、磁盘满SSD、加索引分表、归档
连接数Too many connections增大max_connections连接池、分库

读写分离：一主多从架构

写请求 → 主库（Master）→ 异步复制 → 从库（Slave）
读请求 → 从库1 / 从库2 / 从库N

复制方式：
  - 异步复制：主库写完即返回，延迟小但可能丢数据
  - 半同步复制：至少一个从库确认收到binlog才返回
  - 组复制（MGR）：Paxos协议，强一致性

延迟问题：
  - 主从延迟导致读取旧数据
  - 解决方案：强制走主库（关键读）、延迟阈值告警

分库分表：水平与垂直拆分

垂直拆分：按业务模块拆分

原始库：
  user表、order表、product表、log表

垂直拆分后：
  用户库：user表
  订单库：order表
  商品库：product表
  日志库：log表

特点：业务清晰，但单表数据量仍可能过大，存在跨库Join问题

水平拆分：按数据行拆分

原始表：order（1亿条）

水平分表（单库内）：
  order_0（0~5000万）
  order_1（5000万~1亿）
  
水平分库（多库）：
  DB_0：order_0, order_1
  DB_1：order_2, order_3
  
分片键选择：
  - 用户ID取模：user_id % 4 → 同一用户数据在同一分片
  - 时间范围：按月/按年分表
  - 哈希分片：均匀分布，但范围查询需扫描全部分片

分库分表带来的问题：

问题解决方案
分布式事务最终一致性、Seata、本地消息表
跨分片Join全局表、应用层组装、宽表
全局ID雪花算法、号段模式
分页排序归并排序、限制深度分页
扩容迁移双倍扩容、一致性Hash

实际应用中的决策与权衡

数据量方案说明
< 1000万单库 + 索引优化简单高效
1000万~1亿读写分离读扩展
1亿~10亿分表（单库多表）减少单表数据
> 10亿分库分表 + 中间件ShardingSphere等

常见问题与误区

问题1：分库分表后分页性能差

根因：需聚合多个分片结果再排序 解决：限制页码深度（只允许前100页）、搜索架构迁移到ES

问题2：分片键选择不当导致热点

根因：分片键不均匀，某个分片数据量远大于其他 解决：使用更分散的分片键、二级映射、虚拟分片

问题3：扩容时数据迁移困难

根因：取模分片扩容需大量数据迁移 解决：一致性Hash、预分片（一开始就分1024片映射到少量库）
