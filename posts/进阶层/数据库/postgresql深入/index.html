PostgreSQL 架构原理与 MySQL 对比

核心结论：PostgreSQL 通过进程模型实现连接隔离、通过统一存储层管理数据、通过多版本并发控制（MVCC）实现无锁读写、通过丰富的索引类型和扩展体系覆盖从 OLTP 到 AI 向量的全场景，其设计哲学是"扩展优先、标准兼容、企业级可靠"；而 MySQL 以"简单优先、性能优先"著称，两者在架构模型、MVCC 实现、索引体系和扩展能力上存在根本性差异。

PostgreSQL 通过进程架构、统一存储层、MVCC 和扩展体系实现企业级可靠性

进程架构：每个连接独立进程，崩溃隔离优于线程模型

存储管理：表空间 + TOAST + 页面结构，大字段外置存储

MVCC 与事务：快照隔离 + 可见性规则 + Vacuum 清理

索引体系：B-Tree / GIN / GiST / BRIN 覆盖全场景

扩展生态：pgvector / PostGIS / Citus 等插件体系

实现原理与关键机制

进程模型 vs MySQL 线程模型

PostgreSQL 进程模型：

┌──────────────────────────────────────────┐
│              Postmaster (主进程)           │
│   - 监听端口 5432                          │
│   - 管理所有子进程生命周期                  │
│   - 协调崩溃恢复                           │
└──────────────────────────────────────────┘
          │
    ┌─────┼─────┬─────────┬─────────┐
    ▼     ▼     ▼         ▼         ▼
  后端    后端   后端      后端       后端
  进程    进程   进程      进程       进程
  conn1   conn2  conn3    conn4      conn5
  ───────────────────────────────────────
  每个连接 = 独立进程 (fork)

共享内存区域（所有进程共享）：
  ├── 缓冲池（Buffer Pool）     - 缓存表页和索引页
  ├── WAL 缓冲区                - 预写日志缓冲
  ├── 锁表（Lock Table）        - 进程间锁协调
  ├── 进程通信区                 - 进程间信号传递
  └── 事务状态数组               - 全局事务快照
MySQL (InnoDB) 线程模型：

┌──────────────────────────────────────────┐
│           mysqld (单进程多线程)            │
│                                          │
│   主线程 ──→ 监听端口 3306                 │
│      │                                   │
│      ├──→ 连接线程池                      │
│      │    ├── 线程 1 (conn1)             │
│      │    ├── 线程 2 (conn2)             │
│      │    └── 线程 N (connN)             │
│      │                                   │
│      ├──→ 后台线程                      │
│      │    ├── IO 线程（读/写日志）        │
│      │    ├── Purge 线程                │
│      │    └── Page Cleaner 线程         │
│      │                                   │
│      └──→ 共享缓冲池（Buffer Pool）       │
│                                          │
│   所有线程共享同一地址空间                  │
└──────────────────────────────────────────┘

核心差异：

维度PostgreSQL 进程MySQL 线程
连接开销高（fork 进程 ~1-3ms）低（线程创建 ~0.1ms）
内存占用高（每个进程 ~5-10MB 私有内存）低（线程栈 ~1MB）
崩溃隔离强（单个进程崩溃不影响其他连接）弱（线程崩溃可能导致整个实例崩溃）
并发上限受进程数限制（通常 200~1000）更高（可配置 1 万+ 线程池）
共享数据必须通过共享内存/信号量直接访问进程内全局变量
扩展性连接池必备（PgBouncer/连接池中间件）线程池可选（原生支持）
PostgreSQL 连接数过高问题：

进程数 ↑
   │
100 ┤──────────────── 正常区间
    │
200 ┤────────        建议设置 max_connections
    │                超过后需连接池
500 ┤────            每个进程 ~10MB → 5GB 私有内存
    │                共享内存争用加剧
    │
1000┤──              系统调度开销显著增加
    │                上下文切换成为瓶颈
    │
    └──────────────────────────────────→ 性能

生产建议：

  • PostgreSQL：max_connections = 200，前端配合 PgBouncer 连接池（支持 1 万+ 客户端连接）
  • MySQL：max_connections = 1000+，线程池模式可支持更高并发

MVCC 实现原理：快照隔离与可见性判断

PostgreSQL 的 MVCC 是真正的快照隔离（Snapshot Isolation），与 MySQL InnoDB 的实现方式有本质区别。

PostgreSQL MVCC 核心：每行数据携带事务元数据

┌────────────────────────────────────────────────────────────┐
│                         数据页（8KB）                       │
│  ┌──────────┬──────────┬──────────────┬──────────────────┐ │
│  │ 页头信息  │ 行指针数组│     行数据区   │     空闲空间     │ │
│  │          │          │              │                  │ │
│  │          │  ┌────┐  │  ┌──────────┐│                  │ │
│  │          │  │t1  │──┼─→│ xmin=100 ││                  │ │
│  │          │  │t2  │──┼─→│ xmax=200 ││                  │ │
│  │          │  └────┘  │  │ ctid=(0,1)│                  │ │
│  │          │          │  │ data="..."│                  │ │
│  │          │          │  └──────────┘│                  │ │
│  │          │          │  ┌──────────┐│                  │ │
│  │          │  ┌────┐  │  │ xmin=200 ││                  │ │
│  │          │  │t3  │──┼─→│ xmax=0   ││                  │ │
│  │          │  └────┘  │  │ ctid=(0,2)│                  │ │
│  │          │          │  │ data="..."│                  │ │
│  │          │          │  └──────────┘│                  │ │
│  └──────────┴──────────┴──────────────┴──────────────────┘ │
│                                                            │
│  xmin: 插入该行的事务 ID                                   │
│  xmax: 删除/更新该行的事务 ID（0 表示未删除）               │
│  ctid: 物理位置（块号, 行号）                               │
│  每行版本独立存储，更新产生新行，旧行保留                    │
└────────────────────────────────────────────────────────────┘
可见性判断规则（事务 300 的快照）：

快照内容：
  - xmin = 1, xmax = 500
  - xip_list = [200, 250]  (活跃事务列表)

对每行版本，判断事务 300 是否可见：

1. xmin = 100, xmax = 200
   - xmin=100 < 300，且不在 xip_list 中 → 已提交
   - xmax=200 在 xip_list 中 → 正在删除，但尚未提交
   - 结论：**可见**（删除未提交，仍视为有效）

2. xmin = 200, xmax = 0
   - xmin=200 在 xip_list 中 → 插入事务仍在运行
   - 结论：**不可见**

3. xmin = 400, xmax = 0
   - xmin=400 >= 300 的快照 xmax → 事务 400 在快照创建后启动
   - 结论：**不可见**

4. xmin = 150, xmax = 350
   - xmin=150 已提交 → 插入可见
   - xmax=350 在快照创建后才启动 → 删除不可见
   - 结论：**可见**
-- 查看行的系统列（xmin, xmax, ctid）
SELECT xmin, xmax, ctid, * FROM users WHERE id = 1;

-- 查看当前事务 ID
SELECT txid_current();

与 MySQL MVCC 的关键差异：

维度PostgreSQLMySQL (InnoDB)
版本存储数据页内直接存储多版本（xmin/xmax 列）Undo Log 中存储旧版本，数据页只存最新版
回滚实现无需回滚段，旧版本直接可见通过 Undo Log 回滚和构造旧版本
更新操作新行插入，旧行标记 xmax原地更新，旧版本写入 Undo Log
表膨胀存在（更新产生死元组）较轻（数据页紧凑）
清理机制Vacuum 进程异步清理死元组Purge 线程异步清理 Undo Log
索引维护索引条目指向所有版本，需清理索引只指向最新版，通过回滚指针链
快照获取导出活跃事务列表（轻量）Read View（拷贝事务列表，稍重）
序列化隔离SSI（可串行化快照隔离），检测读写冲突普通锁 + 快照，幻读需间隙锁解决

Vacuum 机制：清理死元组与防止表膨胀

PostgreSQL 表膨胀问题：

UPDATE 操作产生新版本：

事务 100: INSERT → 行 A (xmin=100, xmax=0)
事务 200: UPDATE → 行 A' (xmin=200, xmax=0)
                     行 A 变为 (xmin=100, xmax=200) ← 死元组

事务 300: UPDATE → 行 A'' (xmin=300, xmax=0)
                      行 A' 变为死元组

数据页状态（3 个版本共存）：
┌──────────────────────────────┐
│ 行 A (死)  │ 行 A' (死) │ 行 A'' (活) │
│  xmin=100  │  xmin=200  │  xmin=300   │
│  xmax=200  │  xmax=300  │  xmax=0     │
└──────────────────────────────┘

问题：
1. 表文件不断增大（即使数据量没变）
2. 索引膨胀（每个版本都有索引条目）
3. 查询需跳过死元组，效率下降

解决方案：VACUUM
-- Vacuum 类型

-- 1. 标准 VACUUM：清理死元组，释放空间给同一表复用（不收缩文件）
VACUUM users;

-- 2. VACUUM FULL：完全重写表，收缩文件到最小（需锁表）
VACUUM FULL users;

-- 3. VACUUM ANALYZE：清理 + 更新统计信息（优化器使用）
VACUUM ANALYZE users;

-- 4. 自动 Vacuum（推荐，后台自动运行）
-- postgresql.conf 配置：
autovacuum = on
autovacuum_vacuum_threshold = 50        -- 变化超过 50 行触发
autovacuum_vacuum_scale_factor = 0.2    -- 变化超过 20% 触发
autovacuum_analyze_threshold = 50
autovacuum_analyze_scale_factor = 0.1
Vacuum 执行流程：

1. 扫描表，标记对所有活跃事务不可见的死元组
2. 从索引中移除指向死元组的条目
3. 将死元组占用的空间标记为"可重用"（不收缩文件）
4. 更新空闲空间映射（FSM），供后续 INSERT/UPDATE 使用
5. 更新可见性映射（VM），加速后续 Vacuum 和索引扫描

注意：VACUUM（非 FULL）不会缩小表文件大小
      如果需要回收磁盘空间，使用 VACUUM FULL 或 pg_repack
表膨胀监控：

SELECT schemaname, relname,
       n_live_tup AS live_rows,       -- 有效行数
       n_dead_tup AS dead_rows,        -- 死元组数
       round(n_dead_tup * 100.0 / nullif(n_live_tup + n_dead_tup, 0), 2) AS dead_ratio
FROM pg_stat_user_tables
WHERE n_dead_tup > 1000
ORDER BY dead_ratio DESC;

-- dead_ratio > 20% 建议立即 Vacuum
-- dead_ratio > 50% 建议 VACUUM FULL 或 pg_repack

TOAST 机制：大字段外置存储

PostgreSQL 页面固定 8KB，单行不能跨页存储。

TOAST（The Oversized-Attribute Storage Technique）：

行数据超出 2KB（TOAST_TUPLE_THRESHOLD）时触发：

原始行：
  id=1, name="Alice", bio="很长的文本...（100KB）", avatar="二进制...（500KB）"

TOAST 处理后：
  主表行：
    id=1, name="Alice",
    bio → 指向 TOAST 表的指针（TOAST_POINTER）
    avatar → 指向 TOAST 表的指针

  TOAST 表（pg_toast.pg_toast_xxx）：
    将大字段切分为 2KB 的 chunk 存储
    chunk_id=1, seq=0, data="前2KB"
    chunk_id=1, seq=1, data="中2KB"
    chunk_id=1, seq=2, data="后2KB"

策略（由列的 STORAGE 属性控制）：
  - EXTENDED：先压缩，压缩后还大则外置（默认）
  - EXTERNAL：不压缩，直接外置
  - MAIN：尽量内联存储，必要时外置
  - PLAIN：禁止外置，必须内联（适合小字段）
-- 查看表的 TOAST 信息
SELECT relname, pg_size_pretty(pg_relation_size(oid)) AS main_size,
       pg_size_pretty(pg_relation_size(reltoastrelid)) AS toast_size
FROM pg_class
WHERE relname = 'users';

-- 定义列的 TOAST 策略
CREATE TABLE articles (
    id SERIAL PRIMARY KEY,
    title VARCHAR(200),
    content TEXT STORAGE EXTERNAL,  -- 不压缩直接外置
    summary TEXT STORAGE MAIN       -- 尽量内联
);

与 MySQL 大字段存储对比：

维度PostgreSQL TOASTMySQL InnoDB
行大小限制无限制（通过外置实现）64KB（含所有变长列）
大字段存储自动切分外置到 TOAST 表DYNAMIC 行格式：大字段页外存储
压缩支持（EXTENDED 默认压缩）支持（ROW_FORMAT=COMPRESSED）
访问方式延迟加载（查询时按需拉取）同样延迟加载
页外存储阈值2KB（可调）页大小一半（默认 4KB）

查询优化器：遗传算法 + 代价模型

PostgreSQL 查询优化流程：

SQL 语句
    │
    ├──→ 1. 词法/语法解析 → 生成解析树
    │
    ├──→ 2. 语义分析 → 绑定表/列/类型
    │
    ├──→ 3. 查询重写（Query Rewrite）
    │       ├── 视图展开
    │       ├── 规则系统（Rule System）
    │       └── 谓词下推
    │
    ├──→ 4. 生成执行计划候选
    │       ├── 单表扫描：顺序扫描 / 索引扫描 / 位图扫描
    │       ├── 连接方式：Nested Loop / Hash Join / Merge Join
    │       └── 连接顺序：动态规划（≤12表）/ 遗传算法（>12表）
    │
    ├──→ 5. 代价估算
    │       ├── CPU 代价：处理行数 × 处理单行代价
    │       ├── IO 代价：随机读/顺序读 × 块数
    │       └── 网络代价（并行查询）
    │
    └──→ 6. 选择最小代价计划 → 执行
-- EXPLAIN ANALYZE：实际执行并显示真实耗时

FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE u.created_at > '2024-01-01'
GROUP BY u.name
HAVING COUNT(o.id) > 5;

关键字段解读：

字段含义优化目标
cost=XX..YY启动代价..总代价（相对单位）越小越好
rows=XX估算返回行数接近实际行数（统计信息准确）
width=XX估算每行字节数-
Buffers: shared hit从共享缓冲池命中越高越好
Buffers: shared read从磁盘读取越低越好
Buffers: temp read/write临时文件 IO尽量避免

三种表扫描方式：

1. 顺序扫描（Seq Scan）
   全表逐页读取，适合大比例数据查询（>5% 行）
   cost = 顺序读页数 × seq_page_cost (默认 1.0)

2. 索引扫描（Index Scan）
   遍历索引树 → 获取行指针 → 回表取数据
   cost = 索引遍历 + 随机回表 × random_page_cost (默认 4.0)
   适合高选择性查询、排序、LIMIT

3. 位图索引扫描（Bitmap Index Scan）
   遍历索引 → 构建内存位图（Bitmap）→ 按页顺序回表
   兼顾索引过滤和顺序 IO，适合中等选择性查询
   Bitmap Heap Scan → 按物理顺序读取数据页

三种连接方式：

连接方式原理适用场景cost 特征
Nested Loop外层表逐行，内层表索引查小表驱动大表，内层有索引O(N × logM)
Hash Join小表建哈希表，大表探测大表连接，无合适索引O(N + M)
Merge Join两表排序后顺序合并已排序数据、范围查询O(N logN + M logM)

SELECT attname, n_distinct, null_frac, correlation

同步级别：
  - off：异步，主库提交即返回（最快，可能丢数据）
  - local：本地 WAL 落盘即返回
  - remote_write：备库收到 WAL（未落盘）即返回
  - remote_flush：备库 WAL 落盘即返回
  - remote_apply：备库应用 WAL 即返回（最强一致性）

常见问题与误区

问题1：Vacuum 不及时导致表膨胀

现象：表文件持续增长，磁盘空间不足，查询变慢 根因：大量 UPDATE/DELETE 产生死元组，autovacuum 未触发或执行缓慢 解决：

-- 手动执行 Vacuum
VACUUM ANALYZE bloated_table;

-- 严重膨胀时使用 pg_repack（不锁表重写）
pg_repack -d mydb -t bloated_table

-- 调整 autovacuum 参数（高频更新表）
ALTER TABLE high_update_table SET (
    autovacuum_vacuum_scale_factor = 0.1,
    autovacuum_analyze_scale_factor = 0.05
);

-- 监控膨胀程度
SELECT relname, pg_size_pretty(pg_relation_size(oid)) AS size,
       n_live_tup, n_dead_tup,
       round(n_dead_tup * 100.0 / nullif(n_live_tup, 0), 1) AS dead_pct
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC;

问题2：连接数过多导致 OOM

现象：服务器内存耗尽，进程被 OOM Killer 终止 根因：PostgreSQL 每个连接 ~10MB 内存，连接数设置过高 解决：

-- 1. 降低 max_connections
SHOW max_connections;  -- 检查当前值
ALTER SYSTEM SET max_connections = 200;
SELECT pg_reload_conf();

-- 2. 使用 PgBouncer 连接池（必备）
-- pgbouncer.ini
default_pool_size = 20
max_client_conn = 10000
pool_mode = transaction  -- 事务级连接复用

-- 3. 监控活跃连接
SELECT state, count(*) FROM pg_stat_activity
GROUP BY state;

问题3：索引选择不当导致查询慢

现象：有索引但查询不走索引，或走了错误的索引 根因：统计信息不准、成本估算偏差、查询条件不符合索引定义 解决：

-- 1. 更新统计信息
ANALYZE my_table;

-- 2. 检查执行计划
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM my_table WHERE ...;

-- 3. 强制使用索引（调试用，不建议生产使用）
SET enable_seqscan = off;  -- 关闭顺序扫描，观察计划变化

-- 4. 创建更合适的索引
-- 条件：WHERE lower(name) = 'alice'
-- 错误：CREATE INDEX idx ON my_table(name);
-- 正确：CREATE INDEX idx ON my_table(lower(name));

问题4：JSONB 查询性能差

现象：JSONB 字段加了 GIN 索引但查询仍然慢 根因：GIN 索引适合 @> 包含查询，但 -》 路径查询可能不走索引 解决：

-- 1. 确保使用 @> 运算符（走 GIN 索引）
-- 慢（可能不走索引）
SELECT * FROM products WHERE data -> 'category' = 'electronics';

-- 快（走 GIN 索引）
SELECT * FROM products WHERE data @> '{"category": "electronics"}';

-- 2. 对特定路径创建表达式索引
CREATE INDEX idx_products_category ON products ((data ->> 'category'));

问题5：逻辑复制延迟或冲突

现象：备库数据滞后，或复制中断 根因：大事务、长事务、DDL 操作、主备结构不一致 解决：

-- 监控复制延迟
SELECT client_addr, state,
       pg_size_pretty(pg_wal_lsn_diff(sent_lsn, flush_lsn)) AS replication_lag
FROM pg_stat_replication;

-- 避免大事务（分批提交）
-- 避免在复制表上执行 DDL（逻辑复制不支持 DDL 自动同步）

-- 冲突处理（备库）
SHOW pg_stat_subscription;
