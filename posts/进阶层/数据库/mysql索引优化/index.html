MySQL索引与优化

核心结论：MySQL索引通过B+树实现高效查找，索引设计的核心原则是减少回表、避免全表扫描，EXPLAIN是排查慢SQL的首要工具，理解索引下推和覆盖索引是优化的关键。

MySQL通过B+树索引加速查询，索引设计决定查询性能上限，EXPLAIN是排查SQL性能的首要工具

B+树索引结构与查询原理

聚簇索引（主键索引）：

        [10, 20, 30]
       /     |     \
  [5,7,9] [15,18] [25,28,29]
     ↓        ↓         ↓
   数据行   数据行    数据行
   
特点：叶子节点存储完整数据行，按主键排序

二级索引（非主键索引）：

        ["Alice", "Bob", "Charlie"]
       /           |            \
  [...]         [...]          [...]
     ↓             ↓              ↓
   主键值10     主键值20       主键值30
   
特点：叶子节点存储索引列值 + 主键值
查询时需回表：先查二级索引获取主键 → 再查聚簇索引获取完整数据

索引类型：

索引类型特点适用场景
主键索引聚簇索引，唯一非空表必须有主键
唯一索引值唯一，允许NULL业务唯一约束
普通索引无唯一约束加速查询
组合索引多列联合索引多条件查询
全文索引文本分词索引文本搜索
空间索引地理数据索引GIS应用

EXPLAIN分析：排查慢SQL的首要工具

EXPLAIN SELECT * FROM user WHERE name = 'Tom' AND age > 18;

+----+-------------+-------+------------+------+---------------+------+---------+------+------+----------+-------------+
| id | select_type | table | partitions | type | possible_keys | key  | key_len | ref  | rows | filtered | Extra       |
+----+-------------+-------+------------+------+---------------+------+---------+------+------+----------+-------------+
|  1 | SIMPLE      | user  | NULL       | ALL  | NULL          | NULL | NULL    | NULL | 1000 |    10.00 | Using where |
+----+-------------+-------+------------+------+---------------+------+---------+------+------+----------+-------------+

type = ALL：全表扫描！需要优化

EXPLAIN关键字段解读：

字段含义优化目标
type访问类型争取达到range及以上
possible_keys可能用到的索引-
key实际使用的索引确保使用了正确索引
rows扫描行数越小越好
Extra额外信息避免Using filesort、Using temporary

type性能排序（好 → 差）：

system > const > eq_ref > ref > range > index > ALL

索引设计原则与最左前缀

-- 组合索引 (a, b, c)
CREATE INDEX idx_abc ON table_name(a, b, c);

-- 生效的查询（遵循最左前缀）
WHERE a = 1                          -- ✓ 使用索引a
WHERE a = 1 AND b = 2                -- ✓ 使用索引a,b
WHERE a = 1 AND b = 2 AND c = 3      -- ✓ 使用索引a,b,c
WHERE a = 1 AND c = 3                -- ✓ 使用索引a（c不走索引）

-- 不生效的查询
WHERE b = 2                          -- ✗ 缺少a
WHERE b = 2 AND c = 3                -- ✗ 缺少a
WHERE a > 1 AND b = 2                -- ✓ a走索引，b不走（范围后停止）

索引设计原则：

原则说明
最左前缀组合索引按查询条件顺序设计
选择性高选择性列放前面（区分度高的列）
覆盖索引查询列都在索引中，避免回表
避免冗余已有(a,b)索引，无需单独(a)索引
控制数量索引过多影响写入性能
避免NULL索引列尽量NOT NULL

实际应用中的决策与权衡

场景方案原因
频繁查询的列建立索引加速查询
频繁写入的表控制索引数量减少维护开销
排序/分组字段建立索引避免filesort
大字段前缀索引减少索引大小
范围查询范围列放组合索引最后最左前缀利用最大化

常见问题与误区

问题1：索引了但查询不走索引

根因：索引列使用了函数、隐式类型转换、like ‘%xxx’、OR条件 解决：避免索引列函数操作、确保类型一致、用UNION替代OR

问题2：索引过多导致写入慢

根因：每次INSERT/UPDATE/DELETE都需维护所有索引 解决：删除冗余索引、合并相似索引、读写分离

问题3：慢SQL无法定位

根因：缺乏监控和日志 解决：开启慢查询日志（long_query_time）、使用性能_schema
