Redis发布订阅

核心结论：Redis Pub/Sub通过频道实现简单的消息广播，适合实时性要求高的轻量级通知场景，但不支持持久化、确认和消费组，无法替代专业的消息队列。

Redis Pub/Sub实现简单的消息广播，轻量高效但缺乏可靠性和扩展性

基本模型：频道（Channel）和模式（Pattern）

发布者                    Redis服务器                    订阅者
   |                         |                            |
   |  PUBLISH news "hello"   |                            |
   | ----------------------> |                            |
   |                         | ---- PUSH "hello" --------> | Subscriber-1 (SUBSCRIBE news)
   |                         | ---- PUSH "hello" --------> | Subscriber-2 (SUBSCRIBE news)
   |                         |                            |

模式订阅：
  Subscriber-3: PSUBSCRIBE news.*
  可接收 news.sport, news.tech 等频道消息

命令：

命令说明
SUBSCRIBE channel订阅频道
UNSUBSCRIBE channel退订频道
PSUBSCRIBE pattern按模式订阅
PUNSUBSCRIBE pattern按模式退订
PUBLISH channel message发布消息

Java中使用Redis Pub/Sub

// Jedis订阅
Jedis jedis = new Jedis("localhost", 6379);
jedis.subscribe(new JedisPubSub() {
    @Override
    public void onMessage(String channel, String message) {
        System.out.println("收到消息: " + message + " 来自频道: " + channel);
    }
    
    @Override
    public void onSubscribe(String channel, int subscribedChannels) {
        System.out.println("订阅频道: " + channel);
    }
}, "news", "chat");

// 发布消息
Jedis publisher = new Jedis("localhost", 6379);
publisher.publish("news", "重大新闻！");

实际应用中的决策与权衡

维度Redis Pub/Sub专业MQ（RocketMQ/Kafka）
持久化无（消息即发即弃）有（磁盘存储）
消费确认无有（ACK机制）
消费组无（广播给所有订阅者）有（负载均衡消费）
消息堆积内存限制，易丢消息磁盘存储，可堆积
性能极高（纯内存）高（磁盘顺序写）
适用场景实时通知、简单广播可靠消息传递、削峰

常见问题与误区

问题1：消息丢失

根因：消费者离线期间发布的消息无法接收 解决：改用Redis Stream（支持持久化和消费者组）或专业MQ

问题2：消费者阻塞发布者

根因：消费者处理慢，导致消息积压在Redis内存 解决：消费者异步处理，或改用Stream
