RocketMQ架构与使用

核心结论：RocketMQ通过NameServer无状态集群管理Broker路由，通过Broker主从架构保障高可用，通过顺序消息和事务消息满足复杂业务场景，是金融级高可靠消息队列的代表。

RocketMQ通过NameServer管理路由、Broker存储消息、Producer/Consumer协同工作，支持顺序消息和事务消息

核心架构：NameServer + Broker + Producer + Consumer

NameServer集群（无状态）
    ├── 维护Broker路由信息
    └── Producer/Consumer定时拉取路由

Broker集群
    ├── Master（主节点，读写）
    │   └── Slave（从节点，同步复制）
    └── 存储消息，提供读写服务

Producer（生产者）
    └── 发送消息到指定Topic的Broker

Consumer（消费者）
    └── 订阅Topic，从Broker拉取消息

核心组件职责：

组件职责高可用
NameServer服务注册与路由发现多实例部署，无状态
Broker消息存储与转发主从复制，多Master多Slave
Producer消息发送客户端故障转移
Consumer消息消费集群消费自动重平衡

消息模型：Topic + MessageQueue

Topic: 消息主题，逻辑分类

Topic: OrderTopic
├── MessageQueue-0（Broker-A）
├── MessageQueue-1（Broker-A）
├── MessageQueue-2（Broker-B）
└── MessageQueue-3（Broker-B）

Producer发送消息时：
  - 轮询选择MessageQueue（默认）
  - 或按Key哈希选择（保证同一Key进入同一队列）

Consumer消费时：
  - 集群模式：多个Consumer均分MessageQueue
  - 广播模式：每个Consumer消费全部消息

发送方式：

方式特点适用场景
同步发送等待Broker确认可靠性要求高
异步发送发送后立即返回，回调确认吞吐量要求高
单向发送不等待确认日志采集等不关注结果
// 同步发送
SendResult result = producer.send(msg);

// 异步发送
producer.send(msg, new SendCallback() {
    @Override
    public void onSuccess(SendResult sendResult) {
        System.out.println("发送成功");
    }
    @Override
    public void onException(Throwable e) {
        System.out.println("发送失败: " + e.getMessage());
    }
});

顺序消息与事务消息

顺序消息：

// 全局顺序：所有消息按发送顺序消费（性能差，单队列）
// 分区顺序：同一业务Key的消息顺序消费（推荐）

String orderId = "ORDER_10001";
SendResult result = producer.send(msg, new MessageQueueSelector() {
    @Override
    public MessageQueue select(List<MessageQueue> mqs, Message msg, Object arg) {
        Long id = (Long) arg;
        long index = id % mqs.size();
        return mqs.get((int) index);
    }
}, orderId);  // 同一orderId的消息进入同一队列

事务消息：

事务消息流程（保证本地事务与消息发送一致）：

1. Producer发送半消息（对消费者不可见）
2. Producer执行本地事务（如写DB）
3. Producer根据本地事务结果提交或回滚半消息
4. Broker定期回查本地事务状态（防Producer宕机）

半消息：已发送到Broker，但Consumer不可见，等待事务确认
TransactionListener transactionListener = new TransactionListener() {
    @Override
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        try {
            // 执行本地事务
            orderService.createOrder((Order) arg);
            return LocalTransactionState.COMMIT_MESSAGE;  // 提交半消息
        } catch (Exception e) {
            return LocalTransactionState.ROLLBACK_MESSAGE; // 回滚半消息
        }
    }
    
    @Override
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        // 回查本地事务状态
        boolean exists = orderService.isOrderExist(msg.getKeys());
        return exists ? LocalTransactionState.COMMIT_MESSAGE 
                      : LocalTransactionState.ROLLBACK_MESSAGE;
    }
};

实际应用中的决策与权衡

场景方案原因
金融交易同步发送 + 事务消息可靠性优先
日志采集异步发送 + 批量发送吞吐量优先
订单状态变更顺序消息状态变更有序
削峰填谷集群消费 + 批量消费提升消费能力
通知推送广播消费每个节点都处理

常见问题与误区

问题1：消息重复消费

根因：网络抖动导致消费确认未到达Broker，消息被重新投递 解决：消费者幂等设计（唯一Key去重）

问题2：消息积压

根因：消费能力 < 生产速度 解决：扩容Consumer、批量消费、跳过非关键消息

问题3：顺序消息单点瓶颈

根因：同一队列只能单线程消费 解决：细化分区Key，增加队列数
