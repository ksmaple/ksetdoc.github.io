线程池原理与调优

核心结论：线程池通过复用线程减少创建销毁开销，通过阻塞队列缓冲任务，通过拒绝策略处理超负荷，核心参数（核心线程数、最大线程数、队列容量）的配置直接决定了系统的吞吐和稳定性。

线程池通过复用线程和队列缓冲实现资源控制，核心参数配置不当是导致生产故障的主要原因

ThreadPoolExecutor核心参数

public ThreadPoolExecutor(
    int corePoolSize,           // 核心线程数
    int maximumPoolSize,        // 最大线程数
    long keepAliveTime,         // 空闲线程存活时间
    TimeUnit unit,              // 时间单位
    BlockingQueue<Runnable> workQueue,  // 任务队列
    ThreadFactory threadFactory,         // 线程工厂
    RejectedExecutionHandler handler     // 拒绝策略
)

参数作用与任务提交流程：

提交任务 → 当前线程数 < corePoolSize？
    → 是：创建新线程执行任务
    → 否：队列未满？
        → 是：任务入队列
        → 否：当前线程数 < maximumPoolSize？
            → 是：创建新线程（临时线程）
            → 否：执行拒绝策略

线程空闲超过keepAliveTime：
    - 当前线程数 > corePoolSize：销毁多余线程
    - 当前线程数 <= corePoolSize：保留（除非allowCoreThreadTimeOut=true）

JDK提供的快捷创建方法：

方法特点风险
newFixedThreadPool(n)固定线程数，无界队列队列无限增长，OOM
newCachedThreadPool()线程数无上限，SynchronousQueue线程数爆炸
newSingleThreadExecutor()单线程，无界队列同FixedThreadPool
newScheduledThreadPool(n)支持定时任务同FixedThreadPool

阿里巴巴规范：禁止使用Executors快捷方法，应直接使用ThreadPoolExecutor构造函数，明确指定有界队列。

拒绝策略

策略行为适用场景
AbortPolicy（默认）抛出RejectedExecutionException快速失败
CallerRunsPolicy由提交任务的线程自己执行降低提交速度，自我保护
DiscardPolicy静默丢弃任务不重要任务
DiscardOldestPolicy丢弃最老任务，尝试提交新任务新任务优先
// 生产级线程池配置
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    4,                              // 核心线程数
    8,                              // 最大线程数
    60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(1000), // 有界队列！
    new ThreadFactoryBuilder().setNameFormat("biz-pool-%d").build(),
    new ThreadPoolExecutor.CallerRunsPolicy()  // 自我保护
);

// 优雅关闭
executor.shutdown();  // 不再接受新任务，等待已有任务完成
try {
    if (!executor.awaitTermination(60, TimeUnit.SECONDS)) {
        executor.shutdownNow();  // 强制中断
    }
} catch (InterruptedException e) {
    executor.shutdownNow();
}

实现原理与关键机制

线程池状态管理

线程池用高3位表示状态，低29位表示线程数：

RUNNING    → 接受新任务，处理队列任务
SHUTDOWN   → 不接受新任务，处理队列任务（调用shutdown）
STOP       → 不接受新任务，不处理队列，中断正在执行的任务（调用shutdownNow）
TIDYING    → 所有任务终止，线程数为0，准备执行terminated()
TERMINATED → terminated()执行完毕

状态转换：
RUNNING → SHUTDOWN（shutdown()）
RUNNING → STOP（shutdownNow()）
SHUTDOWN → TIDYING（线程池和队列为空）
STOP → TIDYING（线程池为空）
TIDYING → TERMINATED（terminated()完成）

动态线程池：在线调整参数

// 动态修改核心参数（无需重启）
executor.setCorePoolSize(10);
executor.setMaximumPoolSize(20);
executor.setKeepAliveTime(120, TimeUnit.SECONDS);

// 配合配置中心（Nacos/Apollo）实现动态调优
@NacosConfigListener(dataId = "thread-pool.yml")
public void onConfigChange(String config) {
    ThreadPoolConfig cfg = parse(config);
    executor.setCorePoolSize(cfg.getCoreSize());
    executor.setMaximumPoolSize(cfg.getMaxSize());
}

实际应用中的决策与权衡

场景核心线程数队列原因
CPU密集型CPU核数+1较小（100左右）线程切换开销
IO密集型CPU核数*2或更高较大（1000+）线程阻塞等待IO
混合型根据任务比例调整适中压测确定

线程池监控指标：

指标获取方式告警阈值
活跃线程数getActiveCount()> max * 80%
队列大小getQueue().size()> queue容量 * 80%
完成任务数getCompletedTaskCount()业务监控
拒绝任务数自定义计数> 0

常见问题与误区

问题1：线程池提交任务后无响应

根因：队列无界，任务堆积，内存耗尽 解决：使用有界队列 + CallerRunsPolicy + 监控告警

问题2：ThreadLocal在线程池中导致数据错乱

根因：线程复用，ThreadLocal未清理 解决：任务执行完手动remove()，或使用TransmittableThreadLocal

问题3：shutdown()后任务丢失

根因：shutdown()只等待已提交任务，未提交的任务被拒绝 解决：先标记服务关闭状态，等待存量任务完成，再调用shutdown()
