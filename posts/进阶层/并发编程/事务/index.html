事务机制

核心结论：事务通过ACID特性保证数据一致性，隔离级别在一致性和并发性能之间权衡，Spring通过声明式事务简化编程，分布式事务通过2PC、TCC、Saga等模式解决跨服务一致性。

事务通过ACID保证数据一致性，隔离级别控制并发副作用，分布式事务通过补偿机制实现最终一致性

ACID与隔离级别：一致性与并发的权衡

ACID特性：

特性说明实现机制
Atomicity（原子性）要么全成功，要么全回滚Undo Log
Consistency（一致性）数据状态合法约束+触发器+应用逻辑
Isolation（隔离性）事务间互不干扰MVCC + 锁
Durability（持久性）提交后数据不丢失Redo Log + 磁盘写入

隔离级别与并发问题：

隔离级别脏读不可重复读幻读实现
READ UNCOMMITTED✗允许✗允许✗允许无锁
READ COMMITTED✓禁止✗允许✗允许MVCC
REPEATABLE READ✓禁止✓禁止✗允许（MySQL Innodb禁止）MVCC + 间隙锁
SERIALIZABLE✓禁止✓禁止✓禁止所有操作加锁
并发问题示例：

脏读（Dirty Read）：
  事务A读取 → 事务B修改但未提交 → 事务A读取到未提交数据 → 事务B回滚

不可重复读（Non-repeatable Read）：
  事务A读取 → 事务B修改并提交 → 事务A再次读取，数据变化

幻读（Phantom Read）：
  事务A查询（返回3条）→ 事务B插入新数据并提交 → 事务A再次查询，返回4条

Spring声明式事务：@Transactional

@Transactional(
    propagation = Propagation.REQUIRED,   // 传播行为
    isolation = Isolation.READ_COMMITTED, // 隔离级别
    timeout = 30,                          // 超时时间（秒）
    readOnly = false,                      // 是否只读
    rollbackFor = Exception.class          // 哪些异常触发回滚
)
public void transfer(Long fromId, Long toId, BigDecimal amount) {
    accountDao.decrease(fromId, amount);
    accountDao.increase(toId, amount);
}

传播行为：

传播行为说明场景
REQUIRED（默认）当前无事务则新建，有则加入默认场景
REQUIRES_NEW挂起当前事务，新建独立事务独立操作（如日志记录）
NESTED在当前事务中创建savepoint嵌套事务部分回滚
SUPPORTS有事务则加入，无则以非事务执行查询方法
NOT_SUPPORTED挂起当前事务，以非事务执行不需要事务的操作
MANDATORY必须有事务，否则抛异常强制事务上下文
NEVER必须无事务，否则抛异常禁止事务

实现原理与关键机制

Spring事务实现：AOP代理

@Transactional的方法调用：

外部调用 transfer() → 代理对象拦截
    ↓
TransactionInterceptor.invoke()
    ↓
1. 获取@Transactional配置
2. 获取/创建事务（根据传播行为）
3. 执行业务方法
    ↓ 异常？
    → 是：回滚事务（rollback）
    → 否：提交事务（commit）

事务同步管理器（TransactionSynchronizationManager）：
  - 绑定Connection到当前线程（ThreadLocal）
  - 保证同一线程内的DAO操作使用同一Connection

分布式事务：CAP与BASE理论

CAP定理：一致性、可用性、分区容错性，三者不可兼得

分布式事务解决方案：

1. 2PC（两阶段提交）
   阶段1：协调者询问所有参与者是否可提交
   阶段2：全部同意则提交，任一反对则回滚
   问题：同步阻塞、单点故障、数据不一致风险

2. TCC（Try-Confirm-Cancel）
   Try：预留资源
   Confirm：确认执行业务
   Cancel：释放预留资源
   优点：最终一致性、性能较好
   缺点：业务侵入大，需实现三个接口

3. Saga模式
   将长事务拆分为多个本地事务
   每个本地事务有对应的补偿操作
   失败时按相反顺序执行补偿
   适合：业务流程长、需异步执行

4. 本地消息表（最终一致性）
   业务表 + 消息表同一事务写入
   定时任务扫描消息表，发送到MQ
   消费者处理，失败重试

实际应用中的决策与权衡

场景方案原因
单服务单库@Transactional简单可靠
单服务多库分布式事务框架（Seata AT）对业务侵入小
跨服务调用Saga / TCC / 本地消息表最终一致性
高并发场景最终一致性替代强一致性性能优先
读多写少读写分离 + @Transactional(readOnly=true)提升读性能

常见问题与误区

问题1：@Transactional同类调用不生效

根因：this.method()不走代理，事务拦截器不生效 解决：注入自身代理，或拆分到另一个Bean

问题2：异常被catch导致不回滚

根因：Spring只在异常抛出到代理层时才回滚 解决：catch后重新抛出，或手动调用TransactionAspectSupport回滚

问题3：事务中调用外部HTTP服务

根因：长事务持有数据库连接，HTTP超时导致连接池耗尽 解决：HTTP调用放在事务外，或使用异步补偿机制
