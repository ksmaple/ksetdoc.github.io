Redis核心数据结构

核心结论：Redis通过底层六种数据结构（SDS、双向链表、压缩列表、哈希表、跳表、整数数组）实现对外五种数据类型，每种类型针对特定场景优化，理解底层实现是选择正确数据结构和预估内存的关键。

Redis对外提供五种数据类型，底层通过六种内部数据结构实现，每种类型针对特定场景优化

String：最基础的二进制安全字符串

底层实现：SDS（Simple Dynamic String）

struct sdshdr {
    int len;      // 已使用长度
    int free;     // 未使用长度
    char buf[];   // 字符数组
}

特点：
- 二进制安全（可存储图片、序列化对象）
- O(1)获取长度
- 预分配空间，减少扩容
- 最大512MB

String的典型应用：

场景命令说明
缓存对象SET / GET存储JSON/Protobuf序列化数据
计数器INCR / DECR原子加减，适合限流统计
分布式锁SETNX + EXPIRE原子获取锁
会话存储SETEX带过期时间的缓存
共享配置GET / SET全局配置中心
# 计数器应用
INCR page:view:1001        # 文章1001的浏览量+1
INCRBY page:view:1001 10   # 批量增加
GET page:view:1001

# 分布式锁
SET lock:order:123 NX EX 30  # 原子设置锁，30秒过期

Hash：适合存储对象的键值对集合

底层实现：
- 数据少：ziplist（压缩列表）
- 数据多：hashtable（哈希表）

ziplist条件（默认）：
  field-value数量 < 512，且每个值长度 < 64字节

Hash vs String存储对象：

# String方式（整个对象序列化）
SET user:1001 '{"name":"Tom","age":25,"city":"Beijing"}'
# 修改年龄：需要取出整个对象，修改，写回

# Hash方式（字段独立存储）
HSET user:1001 name Tom
HSET user:1001 age 25
HSET user:1001 city Beijing
# 修改年龄：HSET user:1001 age 26（只修改一个字段）
维度String（JSON）Hash
内存占用大小（ziplist时）
修改字段全量读写单字段读写
获取字段解析JSONHGET直接获取
适用场景对象整体读写字段频繁修改

List：双向链表，适合队列场景

底层实现：
- 数据少：ziplist
- 数据多：quicklist（ziplist + 双向链表）

quicklist：将多个ziplist用双向链表连接
  兼顾ziplist的内存效率和链表的快速插入删除

List的典型应用：

场景命令说明
消息队列LPUSH / BRPOP生产者LPUSH，消费者BRPOP阻塞等待
时间线LPUSH + LRANGE最新N条数据
最近访问LPUSH + LTRIM保持固定长度
# 最新消息时间线
LPUSH timeline:user:1001 "msg:10001"
LPUSH timeline:user:1001 "msg:10002"
LTRIM timeline:user:1001 0 99  # 只保留100条
LRANGE timeline:user:1001 0 9  # 获取最近10条

Set：无序唯一集合

底层实现：
- 数据为整数且数量少：intset（整数数组）
- 其他情况：hashtable（value为NULL）

Set的典型应用：

场景命令说明
标签系统SADD / SREM用户标签
共同好友SINTER交集运算
抽奖SRANDMEMBER / SPOP随机取成员
黑白名单SISMEMBERO(1)判断是否存在

ZSet（Sorted Set）：有序集合

底层实现：
- 数据少：ziplist
- 数据多：skiplist + hashtable
  skiplist：按score排序，支持范围查询
  hashtable：存储member到score的映射，O(1)查score

ZSet的典型应用：

场景命令说明
排行榜ZADD / ZREVRANGE按分数排序
延迟队列ZADD(score=时间戳) / ZRANGEBYSCORE到时间再处理
滑动窗口限流ZADD + ZREMRANGEBYSCORE移除过期成员，统计数量
# 游戏排行榜
ZADD leaderboard 1500 "player:Tom"
ZADD leaderboard 2300 "player:Jerry"
ZADD leaderboard 1800 "player:Spike"
ZREVRANGE leaderboard 0 9 WITHSCORES  # 前10名

# 延迟队列
ZADD delay_queue 1714400000 "task:1001"  # 2024-04-30执行
ZRANGEBYSCORE delay_queue 0 1714400000   # 获取到期的任务

常见问题与误区

问题1：大Key导致性能问题

根因：单个Key数据量过大（如Hash有百万field），阻塞Redis单线程 排查：redis-cli –bigkeys 或 MEMORY USAGE key 解决：拆分大Key、使用UNLINK替代DEL（异步删除）

问题2：Key设计不当导致热点

根因：高频访问集中在少数Key上 解决：Key前缀+随机后缀分散（如user:1001:session:a, user:1001:session:b）

问题3：ZSet score精度问题

根因：score是double类型，精确比较可能出错 解决：时间戳放大（score=时间戳*1000+序号）、使用整数score
