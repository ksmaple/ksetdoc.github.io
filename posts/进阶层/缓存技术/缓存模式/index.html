缓存设计模式

核心结论：缓存面临的三大经典问题——穿透、击穿、雪崩——本质分别是查询不存在数据、热点Key过期、大量Key同时过期，解决方案分别从布隆过滤器、互斥锁、随机过期时间三个方向入手。

缓存穿透、击穿、雪崩是生产环境三大经典问题，分别从过滤、加锁、分散三个维度解决

缓存穿透：查询不存在的数据

场景：恶意请求大量不存在的Key
      请求Key → 缓存未命中 → DB查询（也不存在）→ 返回空
      重复上述过程，DB被大量无效查询压垮

解决方案：

1. 布隆过滤器（推荐）
   预热时将所有有效Key加入布隆过滤器
   查询时先查布隆过滤器：
     - 判断不存在 → 直接返回，不走缓存和DB
     - 判断可能存在 → 查缓存 → 查DB

2. 缓存空值
   即使DB不存在，也在缓存中存空值（短过期时间，如30秒）
   SET key "" EX 30
   
   风险：大量不存在的Key会占用缓存内存

缓存击穿：热点Key过期瞬间

场景：热点Key刚好过期，大量请求同时打到DB

      Thread-1: Key过期 → 查DB → 重建缓存（耗时1秒）
      Thread-2: Key过期 → 查DB → 重建缓存
      Thread-3: Key过期 → 查DB → 重建缓存
      ... 大量线程同时查DB

解决方案：

1. 互斥锁（分布式锁）
   只允许一个线程重建缓存：
   
   public String getData(String key) {
       String value = redis.get(key);
       if (value == null) {
           // 获取分布式锁
           if (redis.setnx("lock:" + key, "1", 60)) {
               try {
                   value = db.query(key);
                   redis.setex(key, 3600, value);
               } finally {
                   redis.del("lock:" + key);
               }
           } else {
               // 其他线程等待后重试
               Thread.sleep(100);
               return getData(key);
           }
       }
       return value;
   }

2. 逻辑过期（永不过期）
   不设置TTL，在Value中存储逻辑过期时间
   发现过期后异步重建，旧数据继续返回
   
   数据结构：{ "data": "xxx", "expireAt": 1714400000 }

缓存雪崩：大量Key同时过期

场景：缓存重启、批量导入、固定过期时间导致大量Key同时失效

      10:00:00  key1过期, key2过期, key3过期 ... key10000过期
      大量请求同时打到DB

解决方案：

1. 随机过期时间
   基础过期时间 + 随机偏移：
   EXPIRE key (3600 + random(0, 300))
   
2. 多级缓存
   L1：本地缓存（Caffeine）→ L2：Redis → DB
   本地缓存兜底，防止全部穿透到DB

3. 高可用架构
   Redis集群 + 哨兵，避免单点故障

缓存与数据库一致性

方案写操作读操作一致性复杂度
Cache Aside先更新DB，再删缓存先读缓存，miss则读DB写缓存最终一致低
Read/Write Through读写都走缓存，缓存同步更新DB同左强一致中
Write Behind只写缓存，异步批量写DB读缓存弱一致高
Cache Aside（最常用）：

读：
  读缓存 → hit → 返回
        → miss → 读DB → 写缓存 → 返回

写：
  写DB → 删缓存（不是更新缓存）
  
  为什么删不是更新？
  - 并发写可能导致缓存脏数据
  - 删缓存简单，下次读自然重建

延迟双删（进一步优化）：
  1. 删缓存
  2. 更新DB
  3. 睡眠500ms（等待主从同步）
  4. 再次删缓存

常见问题与误区

问题1：先删缓存还是先更新DB？

根因：并发时序导致数据不一致 分析：

  • 先删缓存，再更新DB：更新DB前，其他线程读DB旧值写入缓存
  • 先更新DB，再删缓存：删缓存前，其他线程读缓存旧值（短暂不一致，可接受） 结论：先更新DB再删缓存，配合延迟双删

问题2：缓存更新失败导致不一致

根因：DB更新成功，但缓存删除失败 解决：消息队列异步重试删缓存、binlog监听异步删缓存（Canal）

问题3：本地缓存与Redis的一致性问题

根因：多级缓存各层独立管理过期 解决：本地缓存设置更短TTL、使用消息广播刷新本地缓存
