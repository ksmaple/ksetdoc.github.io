Redis持久化与高可用

核心结论：Redis通过RDB快照和AOF日志两种持久化方式保障数据安全，RDB适合全量备份和快速恢复，AOF适合实时持久化，生产环境通常同时使用；高可用通过主从复制、哨兵和集群三种架构逐级递进。

Redis通过RDB快照+AOF日志实现持久化，通过主从复制+哨兵+集群实现高可用扩展

持久化：RDB与AOF的权衡

维度RDBAOF
原理定时快照，保存内存数据记录每条写命令
文件大小紧凑（二进制压缩）较大（可重写压缩）
恢复速度快（直接加载内存）慢（重放命令）
数据安全可能丢失最后一次快照最多丢失1秒数据（每秒刷盘）
性能影响快照时fork子进程，短暂阻塞持续fsync开销
适用场景全量备份、灾难恢复实时持久化、误操作恢复
RDB触发条件（默认）：
- 900秒内至少1个key变化
- 300秒内至少10个key变化
- 60秒内至少10000个key变化

AOF刷盘策略：
- always：每条命令fsync（最安全，最慢）
- everysec：每秒fsync（默认，平衡）
- no：由OS决定（最快，最不安全）

混合持久化（Redis 4.0+）：
- AOF文件包含RDB前缀 + 增量AOF命令
- 重写时先生成RDB，再追加后续命令
- 兼顾恢复速度和数据安全

主从复制：读写分离的基础

主从复制流程：

1. 从节点发送 SYNC 命令
2. 主节点执行 BGSAVE，生成RDB文件
3. 主节点发送RDB文件给从节点
4. 从节点加载RDB，清空旧数据
5. 主节点将复制期间的写命令缓存到replication buffer
6. 发送buffer中的命令给从节点
7. 后续持续同步：主节点写命令异步发送给从节点

复制方式：
- 全量复制：初次同步或主从断开过久
- 部分复制（PSYNC 2.0+）：基于复制偏移量，只同步差异

哨兵（Sentinel）：自动故障转移

Sentinel架构：

       ┌─────────┐
       │ Client  │
       └────┬────┘
            │ 连接Sentinel获取主节点地址
            ↓
    ┌───────┴───────┐
    │  Sentinel-1   │
    │  Sentinel-2   │  （至少3个，奇数个）
    │  Sentinel-3   │
    └───────┬───────┘
            │ 监控 + 自动故障转移
            ↓
    ┌───────┴───────┐
    │    Master     │  ← 主节点（写）
    └───────┬───────┘
            │ 复制
    ┌───────┴───────┐
    │   Slave-1     │  ← 从节点（读）
    │   Slave-2     │
    └───────────────┘

Sentinel职责：
1. 监控：周期性检查主从节点健康
2. 通知：故障时通知管理员或其他应用
3. 自动故障转移：主节点宕机，选举从节点为新主
4. 配置提供者：客户端向Sentinel查询当前主节点地址

集群（Cluster）：水平扩展

Redis Cluster架构：

采用哈希槽（Hash Slot）分片，共16384个槽

Master-A (0-5460)    Master-B (5461-10922)    Master-C (10923-16383)
    ↓                        ↓                         ↓
 Slave-A1                 Slave-B1                  Slave-C1
 Slave-A2                 Slave-B2                  Slave-C2

Key路由：CRC16(key) % 16384 → 确定槽 → 确定节点

Moved重定向：
  客户端请求Key到错误节点 → 返回MOVED slot ip:port
  客户端更新槽映射，重新请求

实际应用中的决策与权衡

场景架构容量可用性
缓存、可重建单实例 + RDB内存限制低
读多写少主从 + 哨兵内存限制高
大数据量集群线性扩展高
跨机房主从跨机房内存限制异地灾备

常见问题与误区

问题1：主从复制延迟导致数据不一致

根因：网络延迟、从节点处理慢 解决：监控复制偏移量、强制读主库（关键读）、延迟阈值告警

问题2：AOF重写导致内存暴涨

根因：重写期间新旧AOF同时存在 解决：预留足够内存、控制重写触发时机

问题3：Cluster模式下多Key操作受限

根因：MGET/MSET等操作的Key需在同一个槽 解决：使用Hash Tag（{user}:1001:name和{user}:1001:age映射到同一槽）
