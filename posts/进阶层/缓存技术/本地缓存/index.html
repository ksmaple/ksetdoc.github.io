本地缓存方案

核心结论：本地缓存通过避免网络IO和序列化开销提供纳秒级访问速度，Caffeine凭借W-TinyLFU算法和异步加载机制成为Java本地缓存的首选，与Redis配合使用形成多级缓存架构。

本地缓存通过避免网络开销提供极致性能，Caffeine基于W-TinyLFU算法在命中率上领先，与Redis配合形成多级缓存

Caffeine：高性能本地缓存

// Caffeine基本用法
LoadingCache<String, User> userCache = Caffeine.newBuilder()
    .maximumSize(10000)                          // 最大条目数
    .expireAfterWrite(10, TimeUnit.MINUTES)      // 写入后10分钟过期
    .refreshAfterWrite(1, TimeUnit.MINUTES)      // 1分钟后异步刷新
    .recordStats()                                // 开启统计
    .build(key -> loadUserFromDB(key));          // 自动加载

User user = userCache.get("user:1001");  // 存在则返回，不存在则自动加载

Caffeine核心特性：

特性说明
W-TinyLFU淘汰策略比LRU更高的命中率，适应访问频率变化
异步加载缓存未命中时异步加载，不阻塞主线程
自动刷新后台线程异步刷新过期缓存
弱引用支持允许Key/Value被GC回收
事件监听移除、过期事件回调

缓存淘汰策略对比：

策略原理特点
LRU最近最少使用实现简单，但对突发流量敏感
LFU最少使用频率历史频率高但不再访问的数据难以淘汰
W-TinyLFU频率+时间窗口兼顾历史频率和近期访问，自适应

Guava Cache：Caffeine的前身

// Guava Cache用法（与Caffeine类似）
LoadingCache<String, User> cache = CacheBuilder.newBuilder()
    .maximumSize(1000)
    .expireAfterWrite(10, TimeUnit.MINUTES)
    .build(new CacheLoader<String, User>() {
        @Override
        public User load(String key) {
            return loadUserFromDB(key);
        }
    });
维度Guava CacheCaffeine
维护状态停止新特性开发活跃维护
性能好更好（6倍吞吐）
淘汰算法LRUW-TinyLFU
异步支持有限完善
推荐度遗留项目新项目首选

多级缓存架构

请求 → L1本地缓存（Caffeine）→ hit → 返回（~纳秒级）
              ↓ miss
        L2分布式缓存（Redis）→ hit → 写L1 → 返回（~微秒级）
              ↓ miss
        DB → 写L2 → 写L1 → 返回（~毫秒级）

一致性保障：
  - 写操作：先写DB，删Redis，广播通知各节点删本地缓存
  - 本地缓存TTL设置短于Redis（如Redis 10分钟，本地2分钟）

实际应用中的决策与权衡

场景方案原因
单节点应用Caffeine简单高效
微服务多节点Caffeine + Redis本地+分布式结合
需要持久化Ehcache（磁盘缓存）重启后数据不丢失
大对象缓存软引用/弱引用内存不足时自动回收
Spring生态@Cacheable + Caffeine/Redis注解驱动，解耦

常见问题与误区

问题1：本地缓存多节点不一致

根因：每个节点独立管理缓存，更新后其他节点未感知 解决：Redis Pub/Sub广播刷新、本地缓存设置短TTL、接受最终一致性

问题2：本地缓存内存溢出

根因：缓存对象过大或条目过多 解决：设置maximumSize/maximumWeight、使用弱引用、监控缓存大小

问题3：缓存加载方法阻塞主线程

根因：CacheLoader.load()同步执行耗时操作 解决：使用AsyncCache异步加载、设置expireAfterWrite而非refresh
