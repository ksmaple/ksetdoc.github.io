Netty网络框架

核心结论：Netty基于NIO多路复用和事件驱动模型，通过EventLoop处理IO事件，通过ChannelPipeline串联业务处理器，通过ByteBuf实现高效内存管理，是构建高性能网络服务的首选框架。

Netty通过EventLoop线程模型处理IO，通过ChannelPipeline串联处理器，通过ByteBuf管理内存，实现高并发低延迟网络通信

核心组件：EventLoop、Channel、Pipeline、ByteBuf

组件职责类比
EventLoopIO事件循环处理NIO中的Selector + 线程
Channel网络连接抽象SocketChannel
ChannelPipeline处理器责任链Filter链/拦截器链
ChannelHandler具体处理器Servlet/Controller
ByteBuf字节缓冲区NIO ByteBuffer的增强版
Bootstrap启动辅助类Spring ApplicationContext
Netty服务端线程模型：

Boss Group（通常1个线程）
  └── 监听端口，accept新连接
      ↓
      将新Channel注册到Worker Group的某个EventLoop

Worker Group（N个EventLoop线程，通常CPU核数*2）
  └── EventLoop-1: 管理 Channel-1, Channel-5, Channel-9...
  └── EventLoop-2: 管理 Channel-2, Channel-6, Channel-10...
  └── EventLoop-N: 管理 Channel-N...
  
  每个EventLoop绑定一个Selector，处理分配给它的所有Channel的IO事件
// Netty服务端启动代码
public class NettyServer {
    public static void main(String[] args) {
        EventLoopGroup bossGroup = new NioEventLoopGroup(1);   // 接收连接
        EventLoopGroup workerGroup = new NioEventLoopGroup();   // 处理IO
        
        try {
            ServerBootstrap bootstrap = new ServerBootstrap();
            bootstrap.group(bossGroup, workerGroup)
                .channel(NioServerSocketChannel.class)
                .childHandler(new ChannelInitializer<SocketChannel>() {
                    @Override
                    protected void initChannel(SocketChannel ch) {
                        ch.pipeline()
                            .addLast(new LoggingHandler(LogLevel.INFO))
                            .addLast(new StringDecoder())    // 解码：ByteBuf → String
                            .addLast(new StringEncoder())    // 编码：String → ByteBuf
                            .addLast(new BusinessHandler()); // 业务处理
                    }
                });
            
            ChannelFuture future = bootstrap.bind(8080).sync();
            future.channel().closeFuture().sync();
        } finally {
            bossGroup.shutdownGracefully();
            workerGroup.shutdownGracefully();
        }
    }
}

ChannelPipeline：处理器责任链

ChannelPipeline中的数据流向：

入站（Inbound：读取数据）：
  Socket → HeadContext → LoggingHandler → StringDecoder → BusinessHandler → TailContext
                                                           ↑
                                                      业务处理

出站（Outbound：写入数据）：
  BusinessHandler → StringEncoder → LoggingHandler → HeadContext → Socket
       ↑
  ctx.writeAndFlush(response)

注意：Inbound从头到尾，Outbound从尾到头
public class BusinessHandler extends ChannelInboundHandlerAdapter {
    
    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) {
        String request = (String) msg;
        System.out.println("收到消息：" + request);
        
        // 处理业务逻辑
        String response = process(request);
        
        // 写入响应（触发Outbound处理器链）
        ctx.writeAndFlush(response);
    }
    
    @Override
    public void exceptionCaught(ChannelHandlerContext ctx, Throwable cause) {
        cause.printStackTrace();
        ctx.close();
    }
}

实现原理与关键机制

ByteBuf：比Java NIO ByteBuffer更高效的缓冲区

特性ByteBufJava ByteBuffer
读写索引分离readerIndex / writerIndexposition翻转
容量自动扩展动态扩容固定容量
引用计数retain()/release()无
池化PooledByteBufAllocator无
复合缓冲区CompositeByteBuf无
// ByteBuf使用示例
ByteBuf buffer = Unpooled.buffer(1024);

try {
    // 写入数据
    buffer.writeBytes("Hello Netty".getBytes());
    
    // 读取数据
    while (buffer.isReadable()) {
        byte b = buffer.readByte();
        System.out.print((char) b);
    }
} finally {
    buffer.release();  // 引用计数减1，为0时释放
}

// 池化ByteBuf（推荐生产环境使用）
ByteBuf pooledBuf = PooledByteBufAllocator.DEFAULT.buffer(1024);
pooledBuf.writeBytes("Pooled".getBytes());
// ... 使用 ...
pooledBuf.release();

内存分配策略：

分配器特点适用场景
UnpooledByteBufAllocator每次创建新缓冲区简单应用
PooledByteBufAllocator基于jemalloc的内存池高并发生产环境

EventLoop的任务调度

EventLoop eventLoop = channel.eventLoop();

// 延迟执行任务
eventLoop.schedule(() -> {
    System.out.println("5秒后执行");
}, 5, TimeUnit.SECONDS);

// 周期性执行任务
eventLoop.scheduleAtFixedRate(() -> {
    System.out.println("每10秒执行一次");
}, 0, 10, TimeUnit.SECONDS);

// 提交普通任务（在EventLoop线程执行）
eventLoop.execute(() -> {
    channel.writeAndFlush("异步消息");
});

关键规则：

  • 同一个Channel的所有IO操作由同一个EventLoop线程执行（线程安全）
  • 不要在ChannelHandler中执行耗时操作，会阻塞EventLoop
  • 耗时操作应提交到业务线程池

粘包/拆包解决方案

TCP粘包问题：

发送方连续发送： [包1] [包2] [包3]
接收方可能收到：[包1+包2前半] [包2后半+包3]

Netty解决方案：

1. FixedLengthFrameDecoder：固定长度解码
   ch.pipeline().addLast(new FixedLengthFrameDecoder(1024));

2. LineBasedFrameDecoder：换行符分隔
   ch.pipeline().addLast(new LineBasedFrameDecoder(8192));

3. DelimiterBasedFrameDecoder：自定义分隔符
   ch.pipeline().addLast(new DelimiterBasedFrameDecoder(8192, Delimiters.lineDelimiter()));

4. LengthFieldBasedFrameDecoder：长度字段（最常用）
   // 格式：| 长度(4B) | 内容(NB) |
   ch.pipeline().addLast(new LengthFieldBasedFrameDecoder(65535, 0, 4, 0, 4));

常见问题与误区

问题1：ByteBuf内存泄漏

根因：未调用release()，引用计数未归零 排查：添加-Dio.netty.leakDetectionLevel=advanced启动参数 解决：使用ReferenceCountUtil.release(msg)，或在SimpleChannelInboundHandler中自动释放

问题2：ChannelHandler中执行阻塞操作

根因：EventLoop线程被阻塞，该EventLoop管理的所有Channel都无法处理IO 解决：使用EventExecutorGroup将耗时操作提交到业务线程池

问题3：ChannelInactive后仍尝试写数据

根因：连接已关闭，Channel不可写 解决：写数据前检查channel.isActive()和channel.isWritable()
