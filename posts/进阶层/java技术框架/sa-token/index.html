Sa-Token权限认证框架

核心结论：Sa-Token通过简洁的API设计和开箱即用的功能，将登录认证、权限校验、分布式Session等复杂逻辑封装为单句调用，适合快速开发场景，但在超大规模分布式场景下需评估其分布式Session实现与业务需求的匹配度。

Sa-Token通过API极简设计降低认证开发成本，通过多模式登录和分布式Session支持复杂业务场景

核心特性对比

特性Sa-TokenSpring SecurityShiro
上手难度极低，单句调用高，需深度配置中，需理解Realm
代码侵入无，方法级注解中，Filter链配置中，Subject编程
分布式Session内置集成Redis需自行整合需自行整合

文档友好度中文详尽英文为主中文较少

核心API

// 登录（自动处理Session、Token生成）
StpUtil.login(10001);

// 权限校验（无权限自动抛异常）
StpUtil.checkRole("super-admin");
StpUtil.checkPermission("user:add");

// 获取当前登录用户
long userId = StpUtil.getLoginIdAsLong();

// 注销
StpUtil.logout();

多账号体系

// 用户端
StpUtil.login(10001);                    // 默认体系
StpUtil.getLoginIdDefaultAsLong();

// 管理端（独立命名空间）
StpAdminUtil.login(10001);
StpAdminUtil.getLoginIdAsLong();
配置项说明
token-nameCookie/Header中的Token名称
timeoutToken有效期（秒）
activity-timeout临时有效期（续签用）
is-concurrent是否允许同一账号多端登录
is-share多端登录是否共享Token

分布式Session集成

# application.yml - Redis集成
sa-token:
  token-name: satoken
  timeout: 2592000
  activity-timeout: -1
  is-concurrent: true
  is-share: false

spring:
  redis:
    host: localhost
    port: 6370
    database: 1
// Sa-Token自动识别Redis依赖并切换分布式存储
// 无需额外配置SessionManager

权限模型

// 登录时绑定角色和权限
StpUtil.login(10001);
StpUtil.getSession().set("roleList", Arrays.asList("admin", "super-admin"));
StpUtil.getSession().set("permissionList", Arrays.asList("user:add", "user:delete"));

// 方法级注解校验
@SaCheckRole("admin")
@SaCheckPermission("user:add")
@RequestMapping("/user/add")
public SaResult add(User user) {
    return SaResult.ok();
}

路由拦截

@Configuration
public class SaTokenConfigure implements WebMvcConfigurer {
    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(new SaInterceptor(handle -> StpUtil.checkLogin()))
            .addPathPatterns("/**")
            .excludePathPatterns("/user/login", "/user/register");
    }
}

常见问题与误区

问题1：分布式环境下Token续签不一致

根因：多实例节点时间不同步或Redis时钟漂移 解决：使用NTP同步服务器时间，续签逻辑以Redis TTL为准

问题2：权限变更后已登录用户权限未刷新

根因：权限缓存于Session中，修改数据库不会自动刷新 解决：权限变更后调用踢人下线或强制重新登录
