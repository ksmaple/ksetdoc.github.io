API网关

核心结论：API网关作为微服务入口，统一处理路由转发、协议转换、安全认证、限流熔断等横切关注点，将后端服务的复杂性对客户端屏蔽，是微服务架构的关键基础设施。

API网关通过统一入口聚合微服务，通过过滤器链处理横切关注点，将客户端与后端服务解耦

网关的核心职责：路由 + 横切关注点

客户端请求 → API网关 → 路由匹配 → 过滤器链 → 后端服务

Spring Cloud Gateway：响应式网关实现

# application.yml
spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://user-service  # 负载均衡到user-service
          predicates:
            - Path=/api/users/**  # 路径匹配
          filters:
            - StripPrefix=1       # 去掉/api前缀
            - AddRequestHeader=X-Request-Source,Gateway
            - RequestRateLimiter=
                redis-rate-limiter.replenishRate: 100
                redis-rate-limiter.burstCapacity: 200
                key-resolver: "#{@userKeyResolver}"
        
        - id: order-service
          uri: lb://order-service
          predicates:
            - Path=/api/orders/**
            - Method=GET,POST
          filters:
            - CircuitBreaker=name:orderCB,fallbackUri:forward:/fallback
// 自定义全局过滤器：认证
@Component
public class AuthGlobalFilter implements GlobalFilter, Ordered {
    
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String token = exchange.getRequest().getHeaders().getFirst("Authorization");
        
        if (!validateToken(token)) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }
        
        // 将用户信息传递给下游服务
        ServerHttpRequest request = exchange.getRequest().mutate()
            .header("X-User-Id", getUserIdFromToken(token))
            .build();
        
        return chain.filter(exchange.mutate().request(request).build());
    }
    
    @Override
    public int getOrder() { return -100; }  // 优先级高，先执行
}

实现原理与关键机制

网关的过滤器链执行顺序

请求处理顺序：

1. Netty接收HTTP请求
2. RoutePredicateHandlerMapping匹配路由
3. FilteringWebHandler执行过滤器链
   
   过滤器顺序：
   ┌─────────────────────────────────────────┐
   │  NettyWriteResponseFilter               │ ← 最高优先级（-2）
   │  RouteToRequestUrlFilter                │ ← 路由转换（10000）
   │  LoadBalancerClientFilter               │ ← 负载均衡（10100）
   │  ... 自定义GlobalFilter ...             │
   │  ForwardRoutingFilter                   │ ← 转发路由（Integer.MAX_VALUE）
   └─────────────────────────────────────────┘

响应返回顺序：与请求顺序相反

限流算法：令牌桶 vs 漏桶

算法原理特点适用场景
令牌桶以固定速率往桶里放令牌，请求需获取令牌才能通过允许一定突发流量大多数限流场景
漏桶请求进入桶中，以固定速率流出严格限制流出速率需要平滑流量的场景
滑动窗口统计固定时间窗口内的请求数简单直观简单限流
// Redis令牌桶限流配置
@Bean
public KeyResolver userKeyResolver() {
    return exchange -> Mono.just(
        exchange.getRequest().getHeaders().getFirst("X-User-Id")
    );
}

// 使用Redis实现分布式限流
@Component
public class RedisRateLimiter {
    public boolean allowRequest(String key, int maxRequests, int windowSeconds) {
        String redisKey = "rate_limit:" + key;
        Long current = redisTemplate.opsForValue().increment(redisKey);
        if (current == 1) {
            redisTemplate.expire(redisKey, windowSeconds, TimeUnit.SECONDS);
        }
        return current <= maxRequests;
    }
}

熔断机制：断路器模式

断路器三种状态：

CLOSED（关闭）  →  正常放行请求
    │
    ↓ 失败率超过阈值
OPEN（打开）    →  快速失败，不调用后端
    │
    ↓ 等待超时时间
HALF-OPEN（半开）→  放行少量请求探测
    │
    ↓ 探测成功          ↓ 探测失败
CLOSED              OPEN
// Spring Cloud Circuit Breaker
@Bean
public Customizer<ReactiveResilience4JCircuitBreakerFactory> defaultCustomizer() {
    return factory -> factory.configureDefault(id -> new Resilience4JConfigBuilder(id)
        .circuitBreakerConfig(CircuitBreakerConfig.custom()
            .failureRateThreshold(50)      // 失败率50%触发熔断
            .waitDurationInOpenState(Duration.ofMillis(1000))  // 熔断持续时间
            .permittedNumberOfCallsInHalfOpenState(10)  // 半开状态允许请求数
            .slidingWindowSize(100)        // 统计窗口
            .build())
        .timeLimiterConfig(TimeLimiterConfig.custom()
            .timeoutDuration(Duration.ofSeconds(2))  // 超时时间
            .build())
        .build());
}

常见问题与误区

问题1：网关成为单点瓶颈

根因：所有流量经过网关，网关性能不足或部署单点 解决：网关集群部署、无状态设计、L7负载均衡、水平扩展

问题2：网关响应慢但后端服务正常

根因：过滤器链中存在耗时操作（如同步调用外部服务） 解决：过滤器中避免阻塞操作，使用异步/响应式编程

问题3：网关路由配置不生效

根因：路由优先级、Predicate顺序、配置刷新问题 排查：检查/actuator/gateway/routes端点，确认实际生效的路由
