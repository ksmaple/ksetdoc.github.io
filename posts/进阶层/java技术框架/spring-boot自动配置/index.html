Spring Boot自动配置

核心结论：Spring Boot通过条件注解判断类路径和环境，自动装配Bean，通过Starter封装常用依赖和默认配置，将Spring应用的初始化代码从数百行缩减到几乎为零。

Spring Boot通过@Conditional条件判断 + SPI机制 + 外部化配置，实现开箱即用的自动装配

自动配置的本质：条件化Bean注册

Spring传统方式（大量配置）：

@Configuration
public class DataSourceConfig {
    @Bean
    public DataSource dataSource() {
        BasicDataSource ds = new BasicDataSource();
        ds.setUrl("jdbc:mysql://localhost:3306/db");
        ds.setUsername("root");
        ds.setPassword("123456");
        return ds;
    }
    
    @Bean
    public JdbcTemplate jdbcTemplate(DataSource ds) {
        return new JdbcTemplate(ds);
    }
}

Spring Boot方式（零配置）：

application.properties:
  spring.datasource.url=jdbc:mysql://localhost:3306/db
  spring.datasource.username=root
  spring.datasource.password=123456

依赖引入spring-boot-starter-jdbc后，DataSource和JdbcTemplate自动配置

核心注解：

注解条件示例
@ConditionalOnClass类路径存在指定类@ConditionalOnClass(DataSource.class)
@ConditionalOnMissingBean容器中不存在指定Bean@ConditionalOnMissingBean(DataSource.class)
@ConditionalOnProperty配置属性匹配@ConditionalOnProperty(prefix=“app”, name=“enabled”, havingValue=“true”)
@ConditionalOnWebApplication是Web应用自动配置Servlet相关组件
@AutoConfigureAfter在指定配置之后控制配置加载顺序
@Configuration
@ConditionalOnClass(DataSource.class)  // 类路径有DataSource才生效
@ConditionalOnMissingBean(DataSource.class)  // 用户未自定义DataSource
@EnableConfigurationProperties(DataSourceProperties.class)  // 绑定配置属性
public class DataSourceAutoConfiguration {
    
    @Bean
    @ConditionalOnMissingBean  // 用户可覆盖
    public DataSource dataSource(DataSourceProperties properties) {
        return DataSourceBuilder.create()
            .url(properties.getUrl())
            .username(properties.getUsername())
            .password(properties.getPassword())
            .build();
    }
}

SPI机制：META-INF/spring.factories

Spring Boot通过SPI（Service Provider Interface）发现自动配置类：

resources/META-INF/spring.factories（Spring Boot 2.7+改为META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports）

org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
com.example.autoconfigure.MyAutoConfiguration,\
com.example.autoconfigure.YourAutoConfiguration

启动时扫描所有jar包中的spring.factories，加载所有AutoConfiguration类

外部化配置：优先级层级

配置优先级（高 → 低）：

1. 命令行参数（--server.port=8081）
2. Java系统属性（System.getProperties()）
3. 环境变量（SERVER_PORT=8081）
4. application-{profile}.properties
5. application.properties（类路径根）
6. @PropertySource注解
7. SpringApplication.setDefaultProperties

配置绑定：
  @ConfigurationProperties(prefix = "app.mail")
  public class MailProperties {
      private String host;
      private int port = 25;
      private String username;
      // getter/setter
  }
  
  对应配置：
    app.mail.host=smtp.example.com
    app.mail.port=587
    app.mail.username=admin

实现原理与关键机制

Starter的组成结构

spring-boot-starter-web：
  1. 依赖spring-boot-starter（核心Starter）
  2. 依赖spring-boot-starter-json（Jackson）
  3. 依赖spring-boot-starter-tomcat（嵌入式容器）
  4. 依赖spring-web（Spring Web MVC）
  5. 依赖spring-webmvc（DispatcherServlet等）
  
  但不包含自动配置类本身！
  自动配置在spring-boot-autoconfigure包中

自定义Starter结构：
  my-spring-boot-starter/
  ├── pom.xml（引入自动配置模块的依赖）
  └── 通常无代码，纯依赖聚合
  
  my-spring-boot-autoconfigure/
  ├── src/main/java/
  │   └── com/example/autoconfigure/
  │       ├── MyAutoConfiguration.java（@AutoConfiguration）
  │       ├── MyProperties.java（@ConfigurationProperties）
  │       └── MyService.java
  └── src/main/resources/
      └── META-INF/
          └── spring/
              └── org.springframework.boot.autoconfigure.AutoConfiguration.imports

自动配置的排除与覆盖

// 方式1：排除特定自动配置
@SpringBootApplication(exclude = {DataSourceAutoConfiguration.class})

// 方式2：通过配置排除
spring.autoconfigure.exclude=com.example.DataSourceAutoConfiguration

// 方式3：自定义Bean覆盖（@ConditionalOnMissingBean）
@Bean
public DataSource dataSource() {
    // 自定义DataSource，自动配置不会生效
    return new HikariDataSource();
}

实际应用中的决策与权衡

场景推荐方案原因
封装内部公共组件自定义Starter复用配置、减少重复
环境差异化配置application-{profile}.yml开发/测试/生产隔离
敏感信息环境变量或配置中心不提交到版本控制
配置校验@Validated + @ConfigurationProperties启动时校验，早失败
动态配置Spring Cloud Config / Nacos运行时刷新

常见问题与误区

问题1：自动配置未生效

根因：Starter未引入、@Conditional条件不满足、被其他配置排除 排查：启动时添加–debug参数查看自动配置报告（Positive/Negative matches）

问题2：配置属性不生效

根因：前缀错误、拼写错误、配置被高优先级覆盖 排查：使用/actuator/configprops端点查看实际绑定值

问题3：自定义Starter循环依赖

根因：AutoConfiguration.imports中配置顺序不当 解决：使用@AutoConfigureBefore/@AutoConfigureAfter控制加载顺序
