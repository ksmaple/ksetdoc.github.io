Spring核心机制

核心结论：Spring通过IoC容器管理对象生命周期，通过AOP在不修改源码的情况下增强功能，二者共同构成了Spring框架的基石，理解Bean的创建过程和代理机制是排查Spring问题的关键。

Spring通过IoC容器解耦对象依赖，通过AOP解耦横切关注点，二者共同实现非侵入式编程

IoC控制反转：依赖注入替代直接创建

传统编程：对象主动创建依赖

public class UserService {
    private UserDao userDao = new UserDao();  // 强耦合
}

IoC编程：容器注入依赖

@Service
public class UserService {
    @Autowired
    private UserDao userDao;  // 容器运行时注入
}

IoC容器的核心职责：

职责说明对应接口
实例化Bean根据配置创建对象BeanFactory
依赖注入装配Bean之间的依赖关系AutowireCapableBeanFactory
生命周期管理初始化和销毁回调InitializingBean / DisposableBean
作用域控制Singleton / Prototype / Request / SessionScope
BeanFactory层级：

BeanFactory（根接口）
    ↓
HierarchicalBeanFactory（支持层级）
    ↓
AutowireCapableBeanFactory（支持自动装配）
    ↓
ConfigurableBeanFactory（支持配置）
    ↓
DefaultListableBeanFactory（完整实现）
    ↓
ApplicationContext（应用上下文，扩展了事件、资源、国际化）

Bean生命周期：从定义到销毁的完整过程

1. 实例化（Instantiation）
   ↓ 反射或CGLIB创建对象
2. 属性赋值（Population）
   ↓ @Autowired、@Value注入
3. 初始化（Initialization）
   ↓ BeanNameAware.setBeanName()
   ↓ BeanFactoryAware.setBeanFactory()
   ↓ ApplicationContextAware.setApplicationContext()
   ↓ @PostConstruct方法
   ↓ InitializingBean.afterPropertiesSet()
   ↓ 自定义init-method
4. 使用（In Use）
5. 销毁（Destruction）
   ↓ @PreDestroy方法
   ↓ DisposableBean.destroy()
   ↓ 自定义destroy-method
@Component
public class MyBean implements InitializingBean, DisposableBean, 
        ApplicationContextAware, BeanNameAware {
    
    @PostConstruct
    public void postConstruct() {
        System.out.println("1. @PostConstruct");
    }
    
    @Override
    public void afterPropertiesSet() {
        System.out.println("2. InitializingBean.afterPropertiesSet()");
    }
    
    public void initMethod() {
        System.out.println("3. 自定义init-method");
    }
    
    @PreDestroy
    public void preDestroy() {
        System.out.println("1. @PreDestroy");
    }
    
    @Override
    public void destroy() {
        System.out.println("2. DisposableBean.destroy()");
    }
}

AOP面向切面编程：横向抽离通用逻辑

业务代码：
  [校验] → [事务开始] → [业务逻辑] → [日志] → [事务提交]

AOP分离后：
  核心业务：  [业务逻辑]
  横切关注点：[校验] [事务] [日志]
  
  运行时编织：
    Proxy → [Before校验] → [业务逻辑] → [AfterReturning日志] → [After事务]

AOP核心概念：

概念说明示例
Aspect切面，横切关注点的模块化@Aspect注解的类
Join Point连接点，程序执行点方法执行、异常抛出
Pointcut切入点，匹配连接点的表达式execution(* com.example...(..))
Advice通知，在切点执行的动作@Before、@After、@Around
Weaving织入，将切面应用到目标编译期/类加载期/运行期
@Aspect
@Component
public class LoggingAspect {
    
    @Pointcut("execution(* com.example.service.*.*(..))")
    public void serviceMethods() {}
    
    @Around("serviceMethods()")
    public Object logAround(ProceedingJoinPoint joinPoint) throws Throwable {
        long start = System.currentTimeMillis();
        System.out.println("方法开始: " + joinPoint.getSignature());
        
        try {
            Object result = joinPoint.proceed();  // 执行目标方法
            System.out.println("方法成功，耗时: " + (System.currentTimeMillis() - start) + "ms");
            return result;
        } catch (Exception e) {
            System.out.println("方法异常: " + e.getMessage());
            throw e;
        }
    }
}

实现原理与关键机制

Spring AOP代理：JDK动态代理 vs CGLIB

维度JDK动态代理CGLIB
原理实现InvocationHandler接口继承目标类，重写方法
要求目标类必须实现接口目标类不能是final
代理类实现相同接口的新类目标类的子类
性能反射调用稍慢生成字节码稍快
Spring默认目标有接口时用JDK目标无接口时用CGLIB
// JDK动态代理
UserService proxy = (UserService) Proxy.newProxyInstance(
    classLoader,
    new Class[]{UserService.class},
    (proxy1, method, args) -> {
        System.out.println("Before");
        Object result = method.invoke(target, args);
        System.out.println("After");
        return result;
    }
);

// CGLIB代理
Enhancer enhancer = new Enhancer();
enhancer.setSuperclass(UserServiceImpl.class);
enhancer.setCallback((MethodInterceptor) (obj, method, args, proxy) -> {
    System.out.println("Before");
    Object result = proxy.invokeSuper(obj, args);
    System.out.println("After");
    return result;
});
UserServiceImpl proxy = (UserServiceImpl) enhancer.create();

循环依赖：三级缓存的解决机制

Spring通过三级缓存解决单例Bean的循环依赖：

一级缓存（singletonObjects）：完整初始化的Bean
二级缓存（earlySingletonObjects）：早期暴露的Bean（已实例化但未填充属性）
三级缓存（singletonFactories）：ObjectFactory，用于生成早期引用

场景：A依赖B，B依赖A

1. 创建A → 实例化A → 将A的ObjectFactory放入三级缓存
2. A需要注入B → 去容器找B
3. 创建B → 实例化B → B需要注入A
4. 从三级缓存获取A的ObjectFactory → 获取A的早期引用（可能是代理对象）
5. B完成初始化
6. A注入B完成 → A完成初始化

限制：

  • 仅支持单例作用域的setter注入
  • 构造器注入的循环依赖无法解决（需@Lazy延迟注入）
  • Prototype作用域不支持循环依赖

实际应用中的决策与权衡

场景推荐方案原因风险
复杂依赖管理@Autowired构造器注入不可变、测试友好循环依赖需@Lazy
配置属性注入@Value / @ConfigurationProperties类型安全配置变更需重启
方法级事务@Transactional + AOP声明式、简洁同类内部调用不生效（需注入自身代理）
日志/监控@Around切面无侵入过多切面影响性能
条件化装配@Conditional / @Profile环境适配条件复杂时难以调试

常见问题与误区

问题1：@Transactional同类内部调用不生效

根因：AOP代理只在方法被外部调用时生效，this.method()调用的是原对象而非代理 解决：注入自身代理，或拆分到另一个Bean中

问题2：@Async方法内部异常未捕获

根因：异步方法在另一个线程执行，异常被包装在Future中 解决：返回CompletableFuture并处理异常，或使用AsyncUncaughtExceptionHandler

问题3：@Autowired字段注入 vs 构造器注入

根因：字段注入依赖反射，不利于单元测试和不可变设计 结论：优先构造器注入（Spring 4.3+单构造器自动注入），字段注入仅用于遗留代码
