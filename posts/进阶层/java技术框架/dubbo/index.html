Dubbo RPC框架

核心结论：Dubbo通过注册中心实现服务自动发现，通过代理模式屏蔽远程调用细节，通过集群策略处理容错和负载均衡，是Java生态中成熟的RPC解决方案。

Dubbo通过注册中心解耦服务提供者与消费者，通过代理和协议层屏蔽远程调用差异，通过集群策略实现容错和负载均衡

核心架构：Provider-Consumer-Registry-Monitor

┌─────────────┐         注册/发现          ┌─────────────┐
│   Provider   │ ───────────────────────→ │   Registry   │
│  (服务提供者) │ ←─────────────────────── │  (注册中心)   │
└──────┬──────┘      服务地址列表           └──────┬──────┘
       │                                         │
       │           RPC调用（Netty/HTTP）          │
       │ ←─────────────────────────────────────→ │
       │                                         │
┌──────┴──────┐                                  │
│   Consumer   │                                  │
│  (服务消费者) │                                  │
└─────────────┘                                  │
                                                 │
┌─────────────┐                                  │
│   Monitor    │ ←───────────────────────────────┘
│  (监控中心)   │        调用统计上报
└─────────────┘

核心角色：

角色职责可选实现
Provider暴露服务，接收RPC调用应用服务实例
Consumer订阅服务，发起RPC调用应用服务实例
Registry服务注册与发现Nacos、Zookeeper、Consul
Monitor收集调用统计Dubbo Admin、Prometheus
Container服务运行容器Spring Boot

服务暴露与引用：代理模式屏蔽远程调用

// 服务提供者暴露服务
@Service  // Dubbo的@Service，非Spring的
public class UserServiceImpl implements UserService {
    @Override
    public User getUser(Long id) {
        return new User(id, "Tom");
    }
}

// 配置（application.yml）
dubbo:
  application:
    name: user-service-provider
  registry:
    address: nacos://localhost:8848
  protocol:
    name: dubbo
    port: 20880
  scan:
    base-packages: com.example.service
// 服务消费者引用服务
@Service
public class OrderService {
    
    @Reference  // Dubbo的@Reference
    private UserService userService;
    
    public Order createOrder(Long userId) {
        User user = userService.getUser(userId);  // 像本地调用一样使用
        return new Order(user);
    }
}

代理生成过程：

Consumer端：
  @Reference UserService userService
    ↓
  Dubbo生成代理类（Javassist/ JDK动态代理）
    ↓
  调用userService.getUser(id)
    ↓
  代理拦截 → 构建Invocation对象
    ↓
  负载均衡选择Provider → 网络发送请求
    ↓
  等待响应 → 反序列化结果 → 返回

对Consumer而言，完全感知不到远程调用

集群容错策略：失败时的应对方式

策略说明适用场景
Failover失败自动切换，重试其他节点（默认）读操作、幂等操作
Failfast快速失败，只发一次非幂等写操作
Failsafe失败安全，忽略异常日志记录、监控上报
Failback失败自动恢复，定时重发消息通知
Forking并行调用多个节点，取最快返回实时性要求高
Broadcast广播调用所有节点集群通知
// 配置集群策略
@Reference(cluster = "failfast", retries = 0)
private UserService userService;

// 配置负载均衡
@Reference(loadbalance = "roundrobin")
private UserService userService;

负载均衡算法：

算法说明
Random随机（默认），可带权重
RoundRobin轮询
LeastActive最小活跃数（调用慢的自动降权）
ConsistentHash一致性Hash（同参数总是路由到同一节点）
ShortestResponse最短响应时间

实现原理与关键机制

Dubbo协议帧格式

Dubbo协议（基于TCP）：

0       1       2       3       4       5       6       7
+-------+-------+-------+-------+-------+-------+-------+-------+
| 魔数(16bit)   | 标志(8bit)    | 状态(8bit)   | 消息ID(64bit)  |
+-------+-------+-------+-------+-------+-------+-------+-------+
| 数据长度(32bit)                                               |
+-------+-------+-------+-------+-------+-------+-------+-------+
| 序列化后的数据（请求/响应）                                    |
+-------+-------+-------+-------+-------+-------+-------+-------+

标志位包含：请求/响应、双向/单向、事件标识、序列化方式

支持的序列化方式：

序列化优点缺点
Hessian2（默认）紧凑、跨语言性能一般
Fastjson2高性能、兼容好安全性历史问题
Kryo极快、紧凑需注册类、兼容性差
Protobuf跨语言、Schema化需定义.proto文件
JDK原生兼容性好慢、体积大

服务注册与发现流程

服务注册（Provider）：
  1. Provider启动，解析@Service注解
  2. 创建URL：dubbo://192.168.1.10:20880/com.example.UserService
  3. 向Registry注册URL
  4. 启动NettyServer监听端口

服务发现（Consumer）：
  1. Consumer启动，解析@Reference注解
  2. 向Registry订阅服务
  3. 收到服务地址列表
  4. 建立与Provider的长连接
  5. 生成代理类，代理类中持有Invoker链

服务调用：
  1. Consumer调用代理方法
  2. ClusterInvoker选择具体Provider（负载均衡）
  3. ProtocolInvoker序列化请求，通过网络发送
  4. Provider收到请求，反序列化，反射调用实现类
  5. Provider序列化响应，返回给Consumer
  6. Consumer反序列化响应，返回给调用方

实际应用中的决策与权衡

场景配置建议原因
内网微服务Triple协议（HTTP/2）兼容gRPC，云原生友好
跨语言调用Triple + Protobuf与Go/Python互通
低延迟要求Dubbo协议 + Kryo二进制协议效率高
高可用读服务Failover + 重试3次自动切换容错
非幂等写服务Failfast + 不重试避免重复写入
大数据传输Hessian2兼容性好

常见问题与误区

问题1：Dubbo调用超时

根因：网络延迟、Provider处理慢、Consumer设置的超时时间太短 解决：调整timeout参数、Provider端优化、增加重试

问题2：注册中心不一致导致调用失败

根因：Provider和Consumer连接的注册中心不同，或注册中心数据不同步 排查：检查dubbo.registry.address配置一致性

问题3：Dubbo 2.7+与3.0+的兼容性

根因：Dubbo 3.0引入应用级服务发现，与2.7的接口级不兼容 解决：统一版本，或开启接口级/应用级双注册
