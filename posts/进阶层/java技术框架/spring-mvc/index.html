Spring MVC请求处理

核心结论：Spring MVC通过DispatcherServlet统一接收请求，借助HandlerMapping定位处理器、HandlerAdapter执行处理器、ViewResolver解析视图，形成高度可扩展的请求处理管道。

Spring MVC通过DispatcherServlet作为前端控制器，将请求分发到具体处理器，拦截器链在关键节点提供扩展点

核心组件与请求处理流程

HTTP请求 → DispatcherServlet → HandlerMapping → HandlerAdapter → Controller → ViewResolver → 响应

详细流程：

1. 请求到达 DispatcherServlet（所有请求的入口）
2. HandlerMapping 根据URL找到对应的 Handler（Controller方法）
3. HandlerExecutionChain 组装 Handler + Interceptor链
4. HandlerAdapter 适配并执行 Handler
5. Controller 处理业务逻辑，返回 ModelAndView 或 @ResponseBody
6. ViewResolver 解析视图名称到具体View
7. View 渲染输出（HTML/JSON/XML）
8. 响应返回客户端

核心组件职责：

组件职责典型实现
DispatcherServlet前端控制器，统一接收和分发请求FrameworkServlet
HandlerMappingURL到处理器的映射RequestMappingHandlerMapping
HandlerAdapter适配不同类型的HandlerRequestMappingHandlerAdapter
HandlerInterceptor请求前后拦截处理HandlerInterceptor接口
ViewResolver视图名称到View的解析InternalResourceViewResolver
View最终渲染输出ThymeleafView / MappingJackson2JsonView

DispatcherServlet的请求分发机制

// DispatcherServlet.doDispatch() 核心逻辑
protected void doDispatch(HttpServletRequest request, HttpServletResponse response) {
    HandlerExecutionChain mappedHandler = null;
    
    try {
        // 1. 检查是否是文件上传请求
        processedRequest = checkMultipart(request);
        
        // 2. 获取HandlerExecutionChain（Handler + Interceptors）
        mappedHandler = getHandler(processedRequest);
        if (mappedHandler == null) {
            noHandlerFound(processedRequest, response);
            return;
        }
        
        // 3. 获取HandlerAdapter
        HandlerAdapter ha = getHandlerAdapter(mappedHandler.getHandler());
        
        // 4. 执行拦截器 preHandle
        if (!mappedHandler.applyPreHandle(processedRequest, response)) {
            return;  // 拦截器返回false，中断处理
        }
        
        // 5. 执行Handler（Controller方法）
        mv = ha.handle(processedRequest, response, mappedHandler.getHandler());
        
        // 6. 执行拦截器 postHandle
        mappedHandler.applyPostHandle(processedRequest, response, mv);
        
        // 7. 处理分发结果（视图渲染或异常处理）
        processDispatchResult(processedRequest, response, mappedHandler, mv, exception);
        
    } catch (Exception ex) {
        // 8. 触发拦截器 afterCompletion（包括异常场景）
        triggerAfterCompletion(processedRequest, response, mappedHandler, ex);
    }
}

拦截器链：在请求处理的关键节点插入逻辑

public interface HandlerInterceptor {
    // Controller执行前
    default boolean preHandle(HttpServletRequest request, HttpServletResponse response, 
                              Object handler) throws Exception {
        return true;  // 返回false则中断请求
    }
    
    // Controller执行后，视图渲染前
    default void postHandle(HttpServletRequest request, HttpServletResponse response,
                            Object handler, ModelAndView modelAndView) throws Exception {
    }
    
    // 请求完成后（包括异常场景）
    default void afterCompletion(HttpServletRequest request, HttpServletResponse response,
                                 Object handler, Exception ex) throws Exception {
    }
}
@Component
public class AuthInterceptor implements HandlerInterceptor {
    
    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response,
                             Object handler) throws Exception {
        String token = request.getHeader("Authorization");
        if (!validateToken(token)) {
            response.setStatus(HttpServletResponse.SC_UNAUTHORIZED);
            return false;  // 中断请求
        }
        return true;
    }
    
    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response,
                                Object handler, Exception ex) throws Exception {
        // 清理ThreadLocal等资源
        UserContext.clear();
    }
}

拦截器 vs 过滤器（Filter）：

维度Filter（Servlet规范）Interceptor（Spring MVC）
归属Servlet容器Spring框架
执行时机DispatcherServlet之前DispatcherServlet内部
可获取信息原始Request/ResponseHandler信息、ModelAndView
使用场景通用处理（编码、日志）业务相关（权限、审计）
配置方式@WebFilter或FilterRegistrationBean实现HandlerInterceptor + 注册

实现原理与关键机制

@RequestMapping的匹配原理

// RequestMappingHandlerMapping 构建映射关系

@Controller
@RequestMapping("/users")
public class UserController {
    
    @GetMapping("/{id}")
    @ResponseBody
    public User getUser(@PathVariable Long id) {
        return userService.findById(id);
    }
    
    @PostMapping
    @ResponseBody
    public User createUser(@RequestBody User user) {
        return userService.save(user);
    }
}

映射存储结构：
  URL: /users/{id}, Method: GET → UserController.getUser
  URL: /users, Method: POST → UserController.createUser

匹配过程：
  1. 根据URL前缀筛选候选映射
  2. 精确匹配 > 路径变量匹配 > 通配符匹配
  3. HTTP方法、Consumes、Produces进一步筛选
  4. 最佳匹配（Best Matching Pattern）

参数解析：@RequestBody和@ResponseBody的处理

@RequestBody 请求体解析：

HTTP请求 → HttpMessageConverter（如MappingJackson2HttpMessageConverter）
         → 将JSON/XML反序列化为Java对象
         
@RequestBody User user
  Content-Type: application/json
  Body: {"name":"Tom","age":25}
  ↓
  ObjectMapper.readValue(body, User.class)

@ResponseBody 响应体转换：

Controller返回值 → HttpMessageConverter
                 → 将Java对象序列化为JSON/XML
                 
@ResponseBody User user
  Accept: application/json
  ↓
  ObjectMapper.writeValueAsString(user)
  ↓
  {"name":"Tom","age":25}

常用的HttpMessageConverter：

Converter支持的MediaType用途
MappingJackson2HttpMessageConverterapplication/jsonJSON序列化/反序列化
StringHttpMessageConvertertext/*字符串处理
ByteArrayHttpMessageConverterapplication/octet-stream字节数组
FormHttpMessageConverterapplication/x-www-form-urlencoded表单数据

场景推荐方案原因

常见问题与误区

问题1：拦截器preHandle返回false后afterCompletion不执行

根因：afterCompletion只在请求正常完成或异常时触发，preHandle中断不触发 解决：在preHandle中自行清理资源

问题2：@RequestBody参数校验失败返回400但不明确

根因：默认校验异常信息不够友好 解决：全局异常处理器捕获MethodArgumentNotValidException，返回结构化错误信息

问题3：异步请求（Callable/DeferredResult）拦截器行为异常

根因：异步请求在不同线程执行，拦截器上下文可能丢失 解决：使用AsyncHandlerInterceptor接口，正确处理异步开始和完成回调
