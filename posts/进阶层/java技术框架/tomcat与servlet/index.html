Tomcat与Servlet

核心结论：Tomcat通过Connector接收HTTP请求，通过Container（Engine-Host-Context-Wrapper）层级管理Servlet生命周期，线程池模型决定了并发处理能力，理解Acceptor-Poller-Worker线程分工是调优关键。

Tomcat通过Connector处理网络连接，通过Container层级管理Servlet，线程池和I/O模型决定并发性能

Tomcat架构：Connector + Container双核心

Server（Tomcat实例）
└── Service
    ├── Connector（HTTP/AJP连接器）
    │   ├── ProtocolHandler（协议处理器）
    │   │   ├── Endpoint（网络端点：NIO/NIO2/APR）
    │   │   │   ├── Acceptor（接收连接）
    │   │   │   ├── Poller（轮询就绪事件）
    │   │   │   └── Executor（工作线程池）
    │   │   └── Processor（处理HTTP协议）
    │   └── Adapter（适配到Container）
    │
    └── Engine（Servlet引擎）
        └── Host（虚拟主机）
            └── Context（Web应用）
                └── Wrapper（Servlet包装器）
                    └── Servlet实例

核心组件职责：

组件职责
ServerTomcat顶层容器，管理Service生命周期
Service包含一个Engine和多个Connector
Connector接收客户端连接，解析HTTP协议
Engine处理所有请求的顶层Servlet引擎
Host虚拟主机，对应一个域名
ContextWeb应用，对应一个WAR包或应用目录
WrapperServlet的包装器，管理Servlet生命周期

Servlet生命周期：初始化→服务→销毁

public interface Servlet {
    void init(ServletConfig config) throws ServletException;  // 初始化
    void service(ServletRequest req, ServletResponse res) throws ServletException, IOException;  // 处理请求
    void destroy();  // 销毁
    ServletConfig getServletConfig();
    String getServletInfo();
}
Servlet生命周期：

1. 加载和实例化（Web应用启动或首次请求）
   ↓
2. init(ServletConfig) 初始化
   ↓ 只执行一次
3. service(ServletRequest, ServletResponse) 处理请求
   ↓ 每次请求执行
4. destroy() 销毁
   ↓ Web应用关闭时执行一次

注意：Servlet默认单例多线程，需保证线程安全

Servlet的线程安全问题：

@WebServlet("/counter")
public class CounterServlet extends HttpServlet {
    // 错误：实例变量共享，线程不安全
    private int count = 0;
    
    @Override
    protected void doGet(HttpServletRequest req, HttpServletResponse resp) {
        count++;  // 并发下可能丢失更新
        resp.getWriter().write("Count: " + count);
    }
}

// 正确：使用局部变量或线程安全类
public class SafeCounterServlet extends HttpServlet {
    private AtomicInteger count = new AtomicInteger(0);
    
    @Override
    protected void doGet(HttpServletRequest req, HttpServletResponse resp) {
        int current = count.incrementAndGet();
        resp.getWriter().write("Count: " + current);
    }
}

实现原理与关键机制

Tomcat线程模型：Acceptor-Poller-Worker

NIO模式（默认）：

Acceptor线程（1个）
  └── 循环accept新连接
      └── 将SocketChannel注册到Poller

Poller线程（通常1个，可配置）
  └── Selector.select() 轮询就绪事件
      └── 事件就绪 → 提交到Executor线程池

Worker线程（Executor线程池，默认200）
  └── 执行Processor → Adapter → Container → Servlet.service()

Tomcat连接器配置：

<Connector port="8080" protocol="org.apache.coyote.http11.Http11NioProtocol"
           maxThreads="200"
           minSpareThreads="10"
           maxConnections="10000"
           acceptCount="100"
           connectionTimeout="20000"
           redirectPort="8443" />
参数说明
maxThreads最大工作线程数
minSpareThreads最小空闲线程数
maxConnections最大连接数（NIO下默认10000）
acceptCount等待队列长度（连接满后新连接的排队数）
connectionTimeout连接超时时间

请求处理流程：

客户端请求
  ↓
Acceptor接收连接 → SocketChannel
  ↓
Poller检测到可读事件
  ↓
从线程池获取Worker线程
  ↓
Http11Processor解析HTTP请求
  ↓
CoyoteAdapter适配为Request/Response
  ↓
StandardEngineValve → HostValve → ContextValve → WrapperValve
  ↓
FilterChain（过滤器链）
  ↓
Servlet.service()
  ↓
响应返回（反向经过上述层级）

Servlet 3.0+ 异步处理

@WebServlet(urlPatterns = "/async", asyncSupported = true)
public class AsyncServlet extends HttpServlet {
    
    @Override
    protected void doGet(HttpServletRequest req, HttpServletResponse resp) {
        AsyncContext asyncContext = req.startAsync();  // 开启异步
        
        asyncContext.start(() -> {
            try {
                // 耗时操作（如调用外部服务）
                String result = callExternalService();
                
                HttpServletResponse response = (HttpServletResponse) asyncContext.getResponse();
                response.getWriter().write(result);
                asyncContext.complete();  // 完成异步请求
            } catch (Exception e) {
                asyncContext.complete();
            }
        });
        
        // 主线程立即返回，不等待耗时操作完成
    }
}

异步Servlet的优势：

  • 释放工作线程，处理更多并发请求
  • 适合IO密集型场景（外部API调用、数据库查询）

实际应用中的决策与权衡

场景配置建议原因
CPU密集型应用maxThreads ≈ CPU核数线程过多上下文切换开销大
IO密集型应用maxThreads 较大（200-500）线程阻塞等待IO时可服务其他请求
高并发短连接acceptCount增大防止连接被拒绝
长连接WebSocketNIO2或APR更好的异步支持
嵌入式部署Spring Boot内嵌Tomcat简化部署
静态资源前置NginxTomcat处理静态资源效率低

常见问题与误区

问题1：Tomcat报Connection refused

根因：连接数达到maxConnections + acceptCount上限 解决：增大maxConnections或acceptCount，或优化应用处理速度

问题2：线程全部阻塞导致服务无响应

根因：慢SQL或外部服务超时，线程池耗尽 排查：jstack查看线程状态，定位BLOCKED/WAITING线程 解决：设置超时时间、熔断降级、异步处理

问题3：Servlet不是线程安全的

根因：Servlet默认单例，多线程共享实例变量 解决：避免实例变量（使用局部变量），或使用线程安全类
